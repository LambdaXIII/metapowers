# Code Review Agent System Prompt — Evaluation Report

**Evaluator:** Agent-as-first-reader simulation
**Date:** 2026-06-17
**File:** S01-b.md

---

## Executive Summary

The prompt is functional but contains several structural gaps and ambiguity points that would likely cause an LLM-based agent to skip valid issues, misclassify problems, or stall. The most critical problems are: (1) no definition of **what constitutes a "line"** in the context of a diff, (2) the severity taxonomy has no operational criteria, and (3) there is no fallback protocol for ambiguous or unparseable diffs. A new agent reading this cold would produce inconsistent reviews.

---

## 1. Structural Overview

| Section | Present? | Notes |
|---|---|---|
| Role definition | ✅ | "Code reviewer" — clear |
| Workflow steps | ✅ | Linear: diff → file → classify → comment |
| Prohibitions | ✅ | No edits, no refactors, no style bikeshedding |
| Prioritization | ✅ | Correctness > Security > Performance > Readability |
| Output format | ✅ | File:line, severity, description, fix suggestion |

**Immediate structural gap:** The prompt is a flat list. There are no conditional branches, no edge-case handlers, and no worked examples. An agent has no way to resolve ambiguity internally.

## 2. Confusion Points (Detailed)

### 2.1 What is a "line" / "line number"?

The prompt requires `File:Line` in every comment. But in a **diff (PR review) context**, three distinct things can be called a "line":

1. The **diff hunk line number** (e.g., `@@ -12,7 +12,8 @@` — the position in the unified diff).
2. The **old-file line number** (the line in the original file).
3. The **new-file line number** (the line in the changed file).

If the agent is writing **inline comments on a PR** (e.g., GitHub), the platform usually expects the new-file line number. The prompt does not specify which. Result: the agent picks arbitrarily, producing comments that attach to wrong locations, or it stalls trying to decide.

**Severity: High.** Causes real output errors.

### 2.2 Severity Levels Are Not Operationalised

The prompt says "按严重度分类问题" (classify by severity) but never defines the levels. Words like "blocker", "critical", "major", "minor", "suggestion" are used in the industry with varying meanings. Without concrete criteria:

- Does a correctness bug always get "blocker" or "critical"?
- Is a missing null check a "security" or "correctness" issue?
- Can multiple issues share the same severity in one file?

The model must **invent** a taxonomy. Different runs of the same review will produce inconsistent severity assignments.

**Severity: High.**

### 2.3 What Counts as "Style" (Prohibited)?

The prompt bans "风格偏好置评" (style preference comments). This is dangerously vague near the border of "readability" (which is an allowed dimension):

- "This variable name is unclear" — readability or style?
- "This function is too long, extract helpers" — readability or style?
- "Use f-strings instead of concatenation" — readability or style?

Without a concrete boundary, a cautious agent may **skip legitimate readability issues**, while an aggressive agent may **still make style comments**, both in good faith.

**Severity: Medium.**

### 2.4 The "Write Inline Comments" Step Is Underdefined

The workflow: "读取 diff → 逐文件检查 → 按严重度分类问题 → 写行内评论"

Step 3 says "classify by severity" but step 4 says "write inline comments". The missing step is: **when do you stop classifying and start writing?** Does the agent:

- (a) Read all files, build a complete list, then write all comments?
- (b) Read one file, write those comments, move on?

(a) requires keeping a potentially large state across the whole PR. (b) may produce comments on later files that contradict earlier ones. The prompt does not specify.

**Severity: Medium. Causes workflow fragmentation.**

### 2.5 No Treatment of the Diff Format Itself

A PR diff can contain:
- Additions only (a new file)
- Deletions only
- Renames without content changes
- Binary file diffs
- Merge commits with conflict markers

The prompt assumes all changes are "逐文件检查" (inspect file by file). It says nothing about:
- How to handle deleted files (no lines to comment on, but deletions may have rationale worth questioning).
- How to handle binary files (no parseable code).
- How to handle massive diffs (1000+ files) where exhaustive review is impossible.

An agent encountering these will either silently skip them or hallucinate an analysis of binary content.

**Severity: Medium (high for real-world use).**

### 2.6 No Confidence / Uncertainty Protocol

Nowhere does the prompt tell the agent what to do when it cannot determine:
- Whether a block of code is safe.
- Whether a change fixes or introduces a bug.
- Whether a dependency upgrade is breaking.

An agent with no "say uncertain" instruction will **confabulate confident-but-wrong comments** rather than flagging uncertainty.

**Severity: High.** Safety issue for production use.

### 2.7 The Priority Order Lacks a Tiebreaker

"正确性 > 安全性 > 性能 > 可读性" tells the agent which dimension is most important. But a single issue can affect multiple dimensions (e.g., an SQL injection is both a correctness bug and a security vulnerability). Does the agent:

- Report it only under the highest-priority dimension (correctness)?
- Report it under all applicable dimensions (duplicate comments)?
- Pick one?

The prompt doesn't say. Duplication wastes reviewer time; under-reporting misses context.

**Severity: Medium.**

## 3. Structural Issues

### 3.1 No "Before You Begin" Orientation

The prompt launches directly into "you are a code reviewer." Missing:
- Context about who the user is (author of the PR? reviewer? maintainer?).
- What the agent should do if the PR description contradicts the diff.
- Whether to review commit messages, PR description, or both.

### 3.2 No Tone / Voice Guidance

"Code reviewer" can mean anything from "friendly peer" to "strict gatekeeper." The prompt doesn't set tone. The agent may produce terse, critical comments that discourage contributors, or overly soft comments that fail to flag real issues.

### 3.3 No Self-Correction Loop

If the agent reads a later file and realises its earlier comment was wrong, the prompt provides no mechanism to:
- Recall the earlier comment.
- Revise or retract it.
- Flag that it found a false positive.

### 3.4 No Summary/Overview Output

The prompt only describes per-line comments. Real PR reviews benefit from a **summary comment** at the top (e.g., "Overall this PR looks good with 3 issues to address"). There's no instruction to produce one.

### 3.5 No Input Validation Step

The prompt never says "verify that the input is a valid diff before starting." If an agent receives an empty response, an error page, or a patch file that doesn't apply, it may still attempt to find issues in whatever text it received.

## 4. Summary of Recommendations

| Issue | Severity | Fix |
|---|---|---|
| Ambiguous "line" reference | High | Specify "new-file line number in the diff hunk" |
| Undefined severity levels | High | Define 3–4 levels with concrete criteria (e.g., Blocker = incorrect behaviour, Critical = security/data loss, etc.) |
| No uncertainty protocol | High | Add "If you cannot determine, flag it as uncertain rather than guessing" |
| Style vs readability boundary | Medium | Give examples of prohibited vs allowed comments on readability |
| Missing diff-edge-case handling | Medium | Add rules for deletions, binary files, and large diffs |
| Dimension tiebreaker | Medium | State: "If an issue spans multiple dimensions, report it under the highest-severity dimension once, and note the secondary dimensions in the description." |
| No summary comment | Low | Add "Write a brief summary comment after all per-line comments." |
| No tone guidance | Low | Add "Be constructive, specific, and respectful." |
| No self-correction / contradictory comment rule | Medium | Add "If you discover a contradiction with an earlier comment, note it." |
| No input validation | Low | Add "If the input is not a valid diff, say so and stop." |

---

## 5. Verdict

**Usable with significant risk.** The prompt would produce plausible-looking reviews on clean, small PRs with only additive changes. On any PR with edge cases (deletions, binary files, large scope, ambiguous issues), the agent would silently degrade in quality: misplacing comments, misassigning severity, or confidently reporting hallucinations. For a production code-review assistant, at least the High-severity gaps must be addressed before deployment.
