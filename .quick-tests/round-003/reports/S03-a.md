## 2. 停顿 / 困惑 / 误解位置

### 2.1 design-journal.md:3 — 时间线矛盾

**原文**: `This protocol was designed and refined in session 2026-05-16.`

**困惑**: design-journal 声称协议在单一 session (2026-05-16) 设计完成，但 SKILL.md 行397-399 记录了两次后续重大变更：

| 日期 | 变更内容 |
|------|---------|
| 2026-05-17 | `web_search` 恢复可用，搜索引擎主入口从 `uvx ddgs` 切换为 `web_search` |
| 2026-05-29 | 移除 camo-run 依赖，Camofox 退役 |

design-journal 的 "single session" 陈述与实际变更历史矛盾。**回访用户如果只读 journal 会产生"这个协议是一次性设计好的"错误印象**。

### 2.2 design-journal.md:7 — 过时信息

**原文**: `- **Search entry unified** to \`uvx ddgs\` (only available search tool in this environment)`

**困惑**: 这条 "Key Design Decision" 在 journal 中是核心记录，但 SKILL.md 行397 指出 `uvx ddgs` 已于 2026-05-17 降级为备用（fallback），主入口变为 `web_search` 工具。

**风险**: 回访用户读 journal 后可能优先使用 `uvx ddgs`（已被墙/降级），而不是主入口 `web_search`。

### 2.4 SKILL.md:84 — 配置语法不完整

**困惑**: 文档提到可以分别设置 `web.search_backend` 和 `web.extract_backend`，但**未给出具体命令示例**。前面只有 `hermes config set web.backend parallel` 的示例。用户不知道如何分别设置：

### 2.5 SKILL.md:10 vs 硬约束冲突

**原文**: `完整的联网搜索协议：从"需要信息"到"拿到内容"的决策树与工具链`

**问题**: 该协议**假设所有工具完全可用**。但在本次测试场景中（S03-a 的约束），`web_search`、`web_extract`、`browser_navigate` 全部被禁止。协议没有清晰的"在不联网/受限环境下如何操作"的路径。

**SKILL.md 行258-265**（4.0a 提取链可用性检查）提到了当 terminal 工具和 web_extract 都不可用时回退到 search-synthesis 模式，但：
- 该回退模式仅针对"提取"阶段，不覆盖"搜索发现"阶段
- 没有说明当搜索工具也被禁用时怎么办
- search-synthesis 模式仅是一个段落描述，没有操作细节

### 2.6 SKILL.md:197 — 内部引用歧义

**原文**: `搜索 issues 优先用 \`gh search issues\`，而非 \`gh issue list\`。`

**困惑**: 这个指引本身清晰，但注意第 173 行 `gh search issues` 使用的是 `--repo owner/repo` 限定符，而第 182 行 `gh issue list` 使用 `-R owner/repo` 限定符。语法不统一（`--repo` vs `-R`）。虽然两者功能等价，但新手可能困惑是否互换了语法概念。

### 2.7 SKILL.md:397 — 版本历史位置偏僻

**原文**: `**2026-05-17**：web_search 工具已恢复可用...`

**困惑**: 这是极其重要的变更信息（决定了整个协议的主入口），但被放在第 8 节"历史与迁移"中，紧跟在"本协议由 web-content-fetching skill 升级而来"的上下文之后。

**回访用户从 design-journal.md 进入后，如果只读到 SKILL.md 的决策树（第1节）就停止，会错过这个关键变更信息。**

## 4. 异常路径推演

### 异常路径 A: 从 design-journal 跳过 SKILL.md 决策树，直接跳到第 4 节（提取）

**推演**:
1. 用户读 design-journal.md → 看到"Three-layer extraction" → 认为自己只需要提取（已有 URL）
2. 打开 SKILL.md 直接跳到第 4 节（内容提取）
3. 跳过第 2 节（搜索发现）→ **错过** `web.backend` 配置检查（行59-86）
4. 跳过第 3 节（GitHub API）→ **误用** 网页提取链去获取 GitHub 内容
5. 直接在 4.1 执行 `uvx --from crawl4ai crwl <url>` → 如果后端配置为 `ddgs`，Crawl4AI 可能因为后续提取失败而白跑 140MB 的 Chromium 下载
6. **结果**: 效率低下 + 可能失败，因为没有先检查 `web_extract` 是否可用（4.0节）和配置后端

**根本原因**: SKILL.md 的结构允许任意位置进入，但没有"前置条件检查"清单。第 2 节的配置检查是全局依赖，不阅读该节会导致整个协议失效。

### 异常路径 B: 从"浏览器是兜底方案"误解为"直接走浏览器即可"

**推演**:
1. 用户读 design-journal.md 行9: "Three-layer extraction **with browser as universal fallback**"
2. 读 SKILL.md 行35: "浏览器是兜底方案：效率低于 Crawl4AI/Jina，但最通用"
3. **误解**: 认为 browser 既然"最通用"，直接走 browser 更稳妥
4. 跳过 Layer 1 (Crawl4AI) 和 Layer 2 (Jina Reader)，直接使用 `browser_navigate(url)`
5. SKILL.md 行375 明确禁止这种行为："不可跳过 Layer 1/2"
6. **结果**: 违反协议规则，效率降低（browser 比 Crawl4AI 重得多）

**原因**: "兜底"（safety net）和"最通用"之间的语义张力——如果某个东西最通用，自然会被误解为最佳默认选择。协议仅在约束部分（6.3）才明确禁止跳过 Layer 1/2，而概念部分（设计日志 + 总览）容易产生误导。

## 5. 意外发现

### 正面发现

1. **references/web-backends-china.md 质量高** — 包含实测数据（响应时间、具体 endpoint、MCP 协议细节），不是笼统描述。对于国内用户有实际操作价值。

2. **SKILL.md:230 JSON 处理原则清晰** — 明确禁止 `python3 -c` 管道解析，给出 `jq` 速查表。这是一个常见坑，文档主动覆盖了。

3. **SKILL.md:132-139 中文搜索污染问题** — 较少技能文档会覆盖此类 SEO 污染话题，显示了真实场景经验。

4. **SKILL.md:84 支持分别配置 search/extract backend** — 这是一个灵活的架构设计（虽然缺少命令示例是个小瑕疵）。

### 负面发现

1. **design-journal.md 缺少"最后更新"标记** — 无法让回访用户判断内容是否过时。截至 2026-06-17，其核心陈述（Search entry unified to `uvx ddgs`）已过时 31 天。

2. **SKILL.md 的决策树（第1节 ASCII 图）无法被视障工具解析** — 纯 ASCII 艺术无法被屏幕阅读器理解。考虑到这是"首次阅读就应该掌握"的核心决策路径，无障碍性缺失。

3. **硬约束场景下协议整体失效** — 当 `web_search` 被禁用时（如本测试 S03-a），整个第 2 节（搜索发现）、第 3 节（GitHub API）、第 4 节（内容提取）均无法操作。协议没有设计"只读模式"或"离线回放"路径。
