# Johnny Decimal

> 用有限十进制编号（10×10 = 100 个槽位）为信息分类。每个区域和类别有固定 ID，跨工具保持一致。适合需要强硬分类边界、内容类型稳定且有限的场景。

**创建者**: Johnny（2013）  
**核心分类维度**: 固定编号空间（最多 10 个区域 × 10 个类别 = 100 个槽位）

## 核心机制

- **区域（Area）**：顶层分类，00-99，每 10 个一组。如 00-09 Home, 10-19 Learning
- **类别（Category）**：区域内子分类。如 10 Learning / 11 Courses, 12 Books
- **项目（Item）**：类别内的具体内容，按需编号
- **ID 格式**：`区域.类别`（如 21.04 = Work 区域的第 5 个类别）

**关键约束**：最多 100 个槽位。不常用或过时内容不移除 ID（留空），但永不重用已释放的 ID。

## 结构示例

```
📁 00-09 Home          # 个人：健康、法律、财务、家庭
📁 10-19 Learning      # 学习与教学
📁 20-29 Work          # 工作与职业
📁 30-39 Projects      # 业余项目与爱好
📁 40-49 Content       # 互联网/发布内容
📁 50-59 Knowledge     # 知识库/书籍笔记
📁 70-79 Tools         # 工具与设置
📁 90-99 Archive       # 归档
📄 Master MOC.md       # 根索引——"知识的 Google 地图"
```

## 适用场景

- **内容类型稳定且可预测**：你知道自己会长期保持这些类别
- **跨工具一致性**：同一编号在文件系统、笔记、书签中保持一致
- **需要强硬分类边界**：不想在"这个该放哪"上花时间——编号是最终答案
- **团队协作**：编号是共享语言，减少"这个文件在哪"的沟通成本

## 不适用场景

- **内容类型快速变化**：新主题出现频繁，100 个槽位不够灵活
- **探索性知识积累**：你不知道 6 个月后会需要哪些分类
- **需要重新分类**：重新编号意味着全局更新引用，成本极高

## 改编要点

- Johnny Decimal 的**有限空间设计**是与其他系统的关键差异——如果你不需要这种约束，就不要用
- 留空 ID 是特性不是 bug——它记录了"这里曾经有一个分类"的历史
- 可以只借"编号前缀排序"的思路（如 `01-Projects`、`02-Areas`），而不强制 10×10 空间
- 与 MOC 配合使用：Master MOC 指向编号区域，弥补编号系统缺乏语义的缺点
