## 示例1：市场分析领域

**任务类型**：专家分析型
**设计思路**：从信息到洞察的转化路径

```markdown
## 核心方法

### 从信息到洞察的转化路径

**第一层：信息分层与验证**
- **原始数据识别**：区分一手数据（调研、财报）vs 二手数据（报道、分析）
- **信源评估**：高（官方财报、权威调研）、中（行业媒体）、低（社交媒体）
- **交叉验证**：同一事实是否有多个独立来源确认？

**第二层：系统化分析框架运用**
- **宏观扫描**：运用PEST框架识别四大力量的交互作用
  * 政治因素如何影响行业准入门槛？（监管政策、贸易壁垒）
  * 经济周期与技术变革的关联是什么？（投资热度、产能周期）
  * 社会趋势正在重塑哪些需求？（消费习惯、价值观变化）
  * 技术突破可能带来哪些范式转移？（替代威胁、效率革命）

- **竞争结构判断**：运用波特五力评估行业利润空间和竞争强度
  * 新进入者的真实威胁是什么？（资金门槛、技术壁垒、渠道控制）
  * 供应商议价能力的临界点在哪里？（集中度、替代材料）
  * 买方议价能力在增强还是减弱？（信息透明度、转换成本）
  * 替代品威胁被低估了吗？（跨行业竞争、技术替代）
  * 现有竞争者的利润空间还有多大？（产能利用率、差异化程度）

**第三层：模式识别与洞察提炼**
- **横向对比**：跨地域、跨周期、跨行业的规律发现
- **纵向推演**：从历史数据预测未来趋势（注意：区分周期性和趋势性）
- **异常检测**：识别与常规不符的信号（可能是机会或风险）

**第四层：SO WHAT转化**
- **业务含义**：每个发现对业务决策意味着什么？
- **行动指向**：分析必须导向可执行的建议
- **优先级排序**：基于影响力和可行性排序

### 质量标准

- **可验证性**：所有关键结论必须有数据支撑
- **客观性**：同时呈现正反两面证据
- **时效性**：优先使用近2年数据
- **可操作性**：每个建议必须具体可执行
```

## 示例2：代码审查领域

**任务类型**：任务执行型
**设计思路**：分层审查法

```markdown
## 核心方法

### 分层审查法

**第一层：形式检查（自动化优先）**
- **语法检查**：是否有语法错误？（如：括号匹配、缩进）
- **命名规范**：变量/函数命名是否符合团队规范？
- **格式一致性**：代码风格是否统一？（如：空格、换行）

**第二层：逻辑检查**
- **边界条件**：是否处理了空输入、最大值、异常值？
- **循环终止**：循环是否有明确的退出条件？是否会死循环？
- **资源释放**：文件、连接、内存是否正确释放？
- **并发安全**：多线程环境下是否有竞态条件？

**第三层：设计检查**
- **职责单一**：每个函数/类是否只做一件事？
- **复杂度**：圈复杂度是否过高？（建议<10）
- **重复代码**：是否存在可以提取的公共逻辑？
- **抽象层次**：接口设计是否合理？实现细节是否隐藏？

**第四层：价值评估**
- **需求对齐**：代码是否真正解决了需求？
- **测试覆盖**：关键路径是否有测试覆盖？
- **文档完整**：关键函数是否有注释？复杂逻辑是否解释清楚？
- **性能考量**：是否有明显的性能瓶颈？（如：O(n²)算法）

### 审查输出格式

```markdown
## 审查报告

### 问题汇总
| 等级 | 类别 | 位置 | 问题描述 | 建议修复 |
|------|------|------|---------|---------|
| 🔴 | 逻辑 | L45 | 缺少空值检查 | 添加 if (x == null) 处理 |
| 🟡 | 设计 | L120-150 | 函数过长（80行） | 拆分为3个函数 |

### 亮点
- [代码亮点1]
- [代码亮点2]
```

### 质量标准

- **可执行性**：每个问题都有明确的修复建议
- **建设性**：不仅指出问题，也肯定优点
- **优先级**：🔴必须修复，🟡建议修复，🟢可选优化
```

## 示例3：文档撰写领域

**任务类型**：任务执行型
**设计思路**：受众导向的写作流程

```markdown
## 核心方法

### 受众导向写作流程

**步骤1：受众分析**
- **读者是谁**？（技术背景、业务背景、决策权）
- **他们关心什么**？（技术细节、业务价值、实施成本）
- **他们有多少时间**？（决定文档长度和深度）
- **他们要用文档做什么**？（决策参考、技术实施、知识传承）

**步骤2：信息架构**
- **核心信息**：一句话概括文档的核心观点
- **支撑论点**：3-5个支撑核心观点的关键论据
- **证据链**：每个论据的数据/案例支撑
- **结构选择**：
  - 决策型读者 → 金字塔结构（结论先行）
  - 学习型读者 → 渐进式结构（由浅入深）
  - 执行型读者 → 清单式结构（步骤明确）

**步骤3：内容撰写**
- **标题设计**：准确、简洁、有吸引力
- **段落结构**：每段一个观点，段首句概括
- **可视化**：复杂数据用图表，流程用图示
- **术语管理**：首次出现解释，避免过度使用缩写

**步骤4：说服力优化**
- **利益点前置**：读者能得到什么价值？
- **异议预设**：可能的反对意见是什么？如何回应？
- **行动号召**：读者下一步应该做什么？

**步骤5：质量检查**
- **完整性**：是否回答了读者的所有关键问题？
- **准确性**：数据、引用是否正确？
- **可读性**：长句是否拆分？专业术语是否解释？
- **一致性**：术语使用是否统一？格式是否规范？

### 质量标准

- **读者测试**：目标读者能否快速找到想要的信息？
- **5秒测试**：读者5秒内能否理解文档核心观点？
- **行动导向**：读者看完后知道下一步做什么吗？
```

## 示例4：客户服务领域

**任务类型**：角色驱动型
**设计思路**：场景化交互协议

```markdown
## 交互协议

### 场景识别与响应策略

**场景A：咨询型**
- **识别信号**：用户询问"如何...""什么是..."
- **响应策略**：
  1. 确认理解："您是想了解...对吗？"
  2. 提供信息：清晰、简洁的回答
  3. 确认满足："这解答了您的问题吗？"
- **升级条件**：问题超出知识范围 → 转接专家

**场景B：问题型**
- **识别信号**：用户表达"无法...""出错了..."
- **响应策略**：
  1. 共情确认："我理解这很困扰..."
  2. 信息收集："为了帮您解决，我需要了解..."
  3. 方案提供："您可以尝试..."
  4. 跟进确认："问题解决了么？"
- **升级条件**：技术问题无法解决 → 创建工单

**场景C：抱怨型**
- **识别信号**：负面情绪词汇、感叹号、大写
- **响应策略**：
  1. 情绪安抚："非常抱歉给您带来不好的体验..."
  2. 责任承担：不推诿，明确"我们会..."
  3. 解决方案：具体的补救措施
  4. 预防承诺："我们会改进...避免再次发生"
- **升级条件**：情绪激烈、要求赔偿 → 升级主管

**场景D：需求型**
- **识别信号**：用户说"我希望...""能不能增加..."
- **响应策略**：
  1. 记录需求：详细记录用户建议
  2. 反馈机制：告知"您的建议已记录，我们会评估"
  3. 替代方案：如暂不支持，提供替代方法
- **升级条件**：重要客户的关键需求 → 反馈产品团队

### 通用原则

- **首次响应**：30秒内响应，让用户知道"我在"
- **等待管理**：如需查询，告知"我需要X分钟查询"
- **结束确认**："还有其他我可以帮您的吗？"
- **记忆保持**：跨对话记住用户历史和偏好

### 边界约束

- **安全边界**：不索要密码、不提供账户操作
- **能力边界**："这个问题我需要转接技术专家"
- **伦理边界**：拒绝歧视性、攻击性语言
```

## 核心原则总结

无论哪个领域，方法设计都要：

1. **理解领域最佳实践**：这个领域通常如何解决这类问题？
2. **转化为可执行形式**：将方法论转化为Agent可遵循的步骤/判断标准
3. **明确质量标准**：什么算"做得好"？
4. **考虑边界情况**：异常情况如何处理？

**关键区分**：
- 不要仅列工具名（如"做PEST分析"）
- 要描述如何运用（如"运用PEST框架识别四大力量的交互作用，关注..."）
