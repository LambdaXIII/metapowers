# metapowers — Agent 规则

本仓库是**帮助 Agent 开发 Skill 的工程仓库**。以下规范以技能开放标准为基线，在此基础上追加项目级约束（标注为「🔧 metapowers 追加」）。

---

## 1. 目录结构

```
metapowers/
├── dev/                          # 开发工作空间
│   └── <skill-name>/             # 每个技能一个子项目（类似库/包）
│       ├── content/              # 技能的直接内容（= 发布包，仅含以下标准条目）
│       │   ├── SKILL.md
│       │   ├── references/       # 可选 · 按需加载的参考文档
│       │   ├── templates/        # 可选 · 提示词/文档模板
│       │   ├── examples/         # 可选 · 参考示例
│       │   └── scripts/          # 可选 · 可执行脚本/资源
│       ├── SKILL-DESIGN.md        # 设计锚定文档（开发侧，必须）
│       ├── CHANGELOG.md          # 变更记录（开发侧，必须）
│       ├── test-space/           # 测试用例（gitignored）
│       └── research/             # 开发参考资料
├── skills/                       # 发布目录 —— 干净技能包
│       ├── SKILL.md
│       ├── references/
│       ├── templates/
│       ├── examples/
│       └── scripts/
└── AGENTS.md                     # 本文件
```

- **`dev/`**：唯一的开发操作空间。所有技能的新建、修改、调试均在此进行。
- **`skills/`**：只读的发布产出。仅在用户明确要求发布时，从 `content/` 干净复制。
- **SKILL-DESIGN 和 CHANGELOG**：开发侧文档，位于 `dev/<name>/` 下——不随技能分发，不进入 `content/`。
- **test-space/**：目录占位，gitignored。具体测试机制待后续规范。

### 技能内容结构（基线标准 + metapowers 追加）
以下为技能 `content/` 目录的完整规范。标注「基线」的条目来自技能开放标准，「🔧 metapowers 追加」为本项目的额外硬性约束。

> **🔴 硬性约束：`content/` 目录仅包含下表所列条目。** 不得出现任何其他文件或目录（如 `RESEARCH.md`、自定义子目录等）。违反此约束的技能不是合法技能包。

| 文件/目录 | 性质 | 来源 | 说明 |
|-----------|------|------|------|
| `SKILL.md` | 必须 | 基线 | 技能主文件。YAML frontmatter（`name`、`description` 为基线必须字段；`metadata` 为 🔧 metapowers 追加必须字段，详见 4.1 节）+ Markdown 正文（触发规则、执行流程、目录树） |
| `references/` | 可选 | 基线 | 按需加载的参考文档。Agent 仅在需要时读取，不占主上下文。文件为 Markdown 格式 |
| `templates/` | 可选 | 基线 | 提示词或文档模板。Agent 按需读取，不占主上下文。文件为 Markdown 或纯文本格式 |
| `examples/` | 可选 | 基线 | 参考示例。帮助 agent 理解技能在不同场景下的推荐做法。每个示例文件应以摘要头开始 |
| `scripts/` | 可选 | 基线 | 可执行脚本或资源文件。供技能指令引用的代码、脚本、配置文件等 |
| `LICENSE` | 条件性必须 | 🔧 metapowers 追加 | 技能许可证原文。仅当使用自定义或非公开引用的许可证时必须包含，详见 §4.4。标准开源协议（MIT、Apache-2.0、GPL 等）无需此文件，直接在 frontmatter 中标识即可 |

> **目录树从技能根开始**：SKILL.md 中展示技能目录树时，根目录必须是技能自身（如 `my-skill/`），不得包含上层分类目录（如 `skills/my-skill/` 或 `dev/my-skill/content/`）。上层路径是磁盘位置细节，不属于技能身份的一部分。

## 2. 核心开发规则（🔴 硬性）

以下规则在 metapowers 仓库中工作时**不可绕过**。违反任一条即为工作方式错误。

### 2.1 源码优先 —— 路径即身份

> 在 metapowers 仓库内工作时（CWD 为本仓库），涉及技能名称时的**默认目标永远是源码**。
> 只有用户**明确说「调用/使用/运行」某技能**时，目标才是运行时已安装版本。

**具体约束**：

- 技能内容（SKILL.md、references/ 等）的读写以 `dev/<name>/content/` 为根。开发侧文档（SKILL-DESIGN.md、CHANGELOG.md）位于 `dev/<name>/` 下，不在 content/ 内。
- 读取正在开发的技能源码时，始终使用文件系统路径（`dev/<name>/content/`）。不要通过技能加载机制来读取源码——技能加载机制获取的是运行时已安装版本，而非仓库中的开发版本。
- 拿不准意图时，默认选择是源码而非运行时。只有用户明确要求「帮我用 XX 技能做 YY」时才走运行时。


### 2.2 SKILL-DESIGN 锚定 —— 涉及即读

> 一旦话题涉及某个开发中的技能，应**尽早**读取该技能的 `dev/<name>/SKILL-DESIGN.md`，避免思考方向偏离已确立的设计思想。

**具体约束**：

- 这是前置步骤。涉及即读，不等编码阶段，避免方向偏移。
- 如果 SKILL-DESIGN 不存在（新建技能），先与用户自由讨论技能的作用和目标、确定命名，再初始化目录结构（见 §3.3）。不得跳过设计讨论直接编写内容。
- 修改过程中，持续以 SKILL-DESIGN 中的设计原则为判断依据——新代码是否符合已锚定的方向？如不符合，应先讨论是否调整方向（更新 SKILL-DESIGN），而非在代码中制造分歧。

### 2.3 文档自包含 —— 不给未来留谜题

> 你编写的每一份文档（SKILL-DESIGN、CHANGELOG、SKILL.md、references/ 等），其目标读者是**未来重新打开它的人**——可能是未来的你，也可能是其他开发者。他们**未参与**当前对话，**不知道**对话中的上下文。

**具体约束**：

- **禁止**使用「如前所述」「基于以上讨论」「按照之前的决定」等会话引用——文档不是对话的延伸。
- **必须**在文档开头说明背景和目的：这份文档是什么、为什么存在、解决什么问题。
- **决策必须附带理由**：不仅写「选择了方案 A」，还要写「因为方案 B 有 X 问题，方案 C 有 Y 限制，因此选择 A」。
- **术语首次出现时必须定义**：不要假定读者知道项目内部术语或缩写。
- 写完后自问：**「一个没有参与这场对话的人，能仅凭这份文档理解全部必要信息吗？」**

### 2.4 CHANGELOG 克制 —— 版本号有纪律

> CHANGELOG 记录**逻辑变更**，不是操作日志。版本号不在开发过程中逐次递增——功能完成后再统一确定。

**版本号判定标准**：

| 级别 | 触发条件 | 示例 |
|------|----------|------|
| **Patch** (x.y.z → x.y.z+1) | bug 修复、错别字、格式调整、表述澄清 | 修正 SKILL.md 中一处错误的条件判断 |
| **Minor** (x.y.z → x.y+1.0) | 新功能、新 reference/template、实质性改进 | 新增一个 reference 文件、重构某段流程 |
| **Major** (x.y.z → x+1.0.0) | 不兼容变更、完全重写、设计哲学变化 | 从分类式改为路由式、移除核心机制 |

**具体约束**：

- 在 feature 分支上开发期间保持版本号不变，功能完成后再更新。不要在每编辑一个文件后递增版本号。
- CHANGELOG 条目以**逻辑变更为单位**，不是以文件编辑为单位。一个 PR 对应一组 CHANGELOG 条目（或一条）。
- 格式遵循 [Keep a Changelog](https://keepachangelog.com/)。
- 禁止无实质内容的 CHANGELOG 条目（如「更新文档」「修改了一些内容」）。

### 2.5 SKILL-DESIGN 保护 —— 只在设计意图变更时修改

> SKILL-DESIGN.md 是技能的**设计锚定文档**——记录设计方向、原则理由和关键决策。它不是实现笔记，不随每次代码修改同步更新。

**SKILL-DESIGN 应当包含**：

- **设计意图**：这个技能解决什么问题？为什么需要它？
- **关键架构决策**：重要的取舍及其理由（如「为什么拆成 references/ 和 templates/」、「为什么用 X 而非 Y」）
- **措辞风格约定**：技能中一致的用语选择及其原因
- **维护注意事项**：修改时需要注意的坑、耦合点、不变量

**SKILL-DESIGN 不得包含**：

- 版本号或版本历史——版本信息在 frontmatter `metadata.version` 和 `CHANGELOG.md` 中
- 文件目录树——SKILL.md 正文已包含
- SKILL.md 中已有的触发规则、执行流程——SKILL-DESIGN 解释「为什么」，SKILL.md 定义「怎么做」，各司其职
- 局部实现细节——实现细节的变更不应触发 SKILL-DESIGN 更新

**更新时机**：

> 每次修改技能内容后，自问：「**这个改动是否改变了设计意图？是否引入了新的设计决策？**」
> - YES（改变既有意图）→ 更新 SKILL-DESIGN，记录新的设计决策和理由
> - NEW（新增功能或机制）→ 在 SKILL-DESIGN 中追加相应的设计决策
> - NO（纯实现、措辞、bugfix）→ 不动 SKILL-DESIGN

### 2.6 原子修改 —— 备份、重写、对照、清理

> 当计划重写整个文件或进行大幅度修改时，不得直接清空或删除原文件。必须通过备份保留原始内容作为参考。

这是针对 agent 常见失败模式的安全网：写到一半忘记原始信息、原内容已被覆盖无法恢复。备份保证原始内容始终可查阅。

**操作步骤**：

1. **备份原文件**：将原文件重命名为 `<文件名>.bak.<扩展名>`（如 `SKILL.bak.md`）
2. **写入新文件**：以原文件名创建新文件，完成全部内容编写。此时原文件名下为新内容，`.bak` 文件保留原始内容可供随时查阅
3. **对照检查**：对照 `.bak` 文件，确认关键内容已覆盖、没有遗漏重要段落或逻辑分支。发现遗漏 → 补充新文件 → 重新检查 → 确认后继续
4. **删除备份**：确认无误后删除 `.bak` 文件

> 局部修改（增删几行、替换段落）使用 `edit` 工具正常操作，不需要走备份流程。


### 2.7 设计内容分离 —— 开发者信息不入技能

> 技能的**设计原则和内部约定**（如「本技能统一使用术语 A 而非 B」「这里故意不用 X 模式因为 Y」）属于开发侧知识，必须写入 `SKILL-DESIGN.md`，**不得泄漏到技能内容中**（SKILL.md、references/、templates/）。

技能内容（`content/`）的读者是**运行时的 agent**——它需要的是可执行的指令和参考，而非设计者的内心独白。设计原则混入技能内容会造成两种破坏：运行时 agent 被无关信息干扰；设计意图的变更无法独立追踪（混在内容中难以定位）。

**具体约束**：

- 编写任何 `content/` 内文件之前，先自问：**「这段信息是给运行时 agent 用的，还是给未来维护这个技能的开发者用的？」**
- 给开发者的 → `SKILL-DESIGN.md`（措辞风格约定、取舍理由、维护注意事项、设计不变量）
- 给运行时 agent 的 → `content/` 内对应文件（触发规则、操作步骤、参考信息、模板）
- 已在 `SKILL-DESIGN.md` 中记录的设计原则，不得在技能内容中复述

### 2.8 技能内容零歧义 —— 首次阅读即全懂

> 技能中的所有内容（SKILL.md、references/、templates/）在被阅读时**一定是第一次阅读**。必须保证重读时能够复现设计时的深刻理解，绝不能包含歧义、模糊表述、或依赖既有知识的隐含假设。

**具体约束**：

- 每一个术语、每一个步骤、每一个条件分支，都必须**精确且排他**——读者不会有「这里指的是 A 还是 B」的疑问
- 禁止「适当的」「合理的」「必要的」等主观模糊词——给出具体标准或明确范围
- 禁止隐含假设读者的既有知识——如果某个概念不是常识，必须就地定义或引用定义位置
- 写完任何一段内容后自问：**「一个对这项技能一无所知的 agent，仅凭这段文字，能准确无误地执行吗？」**

## 3. 开发工作流

### 3.1 分支策略

- **`main` 分支不接受直接提交**。所有对 main 的修改必须通过合并引入。
- 开发在 **feature 分支**上进行，从 main 检出。分支命名建议：`feat/<描述>`、`fix/<描述>`。
- 合并到 main 时使用 **`--no-ff`**（保留分支历史，不压缩为单 commit）。
- 一个功能分支对应一个逻辑变更。避免在一次提交中包含多个不相关的修改。
- `dev/` 目录是开发工作空间，`skills/` 是发布产出目录。

### 3.2 发布流程

发布由用户发起，暂无自动触发机制。流程如下：

```bash
# 发布技能 <name>：先删后拷，确保无残留
rm -rf skills/<name>/
cp -r dev/<name>/content/ skills/<name>/
```

**为什么「先删后拷」而非直接覆盖**：直接 `cp -r` 覆盖不会删除目标目录中已从源目录移除的文件，导致发布包残留过时内容。先 `rm -rf` 再 `cp -r` 确保发布包与 content/ 完全一致。


### 3.3 新建技能

新建技能时，建议按以下顺序推进——这是引导性流程，非硬性工作流：

1. 与用户自由讨论技能的作用和目标，达成共识后再继续
2. 讨论并确定技能命名
3. 初始化目录结构：
   ```bash
   mkdir -p dev/<name>/content/{references,templates,examples,scripts}
   mkdir -p dev/<name>/{test-space,research}
   touch dev/<name>/SKILL-DESIGN.md
   touch dev/<name>/CHANGELOG.md
   ```
4. 在项目 README 中新建对应条目

## 4. 技能规范

### 4.1 metadata 字段（🔧 metapowers 追加 · 必须）

SKILL.md 的 frontmatter 中必须包含 `metadata` 字段，包含以下子字段。**所有子字段的值在 YAML 中必须使用双引号包围**（如 `version: "1.0.0"`），以防止 YAML 的自动类型推断将版本号解析为浮点数、将日期格式字符串解析为日期对象等非预期类型转换。

```yaml
---
name: my-skill
description: 一句话描述技能的用途和触发条件
metadata:
  version: "1.0.0"
  last_updated: "2026-06-27"
  author: "<git-username>"
  license: "MIT"
---
```

| 子字段 | 说明 | 值类型 |
|--------|------|--------|
| `version` | 语义化版本号 | 字符串，YAML 中必须加双引号，如 `"1.0.0"` |
| `last_updated` | 最后更新日期 | 字符串，ISO 8601 日期格式 `"YYYY-MM-DD"`，YAML 中必须加双引号 |
| `author` | 作者标识 | 字符串 |
| `license` | 许可证标识 | 字符串，标准开源协议使用 SPDX 标识符（如 `"MIT"`、`"Apache-2.0"`）；自定义协议使用 `"Custom"` 或其他名称。详见 §4.4 |

> **为什么必须加引号**：YAML 解析器会对无引号的值做类型推断。`version: 1.0.0` 会被当作浮点数（值变为 `"1.0"`），`last_updated: 2026-06-27` 会被当作日期对象而非字符串。加双引号可强制 YAML 将其作为字符串处理，保证 Agent 读取 frontmatter 时类型一致、语义正确。

### 4.2 SKILL-DESIGN.md（🔧 metapowers 追加 · 必须）

> 详见规则 **2.5 SKILL-DESIGN 保护**。

SKILL-DESIGN.md 位于 `dev/<name>/SKILL-DESIGN.md`（开发侧，不随技能分发）。它是技能的**设计锚定文档**——记录为什么这样设计，而非怎样实现。应包含设计意图、关键架构决策（含理由）、措辞风格约定、维护注意事项。不得包含版本号、目录树、或 SKILL.md 中已有的触发规则/执行流程。

### 4.3 CHANGELOG.md（🔧 metapowers 追加 · 必须）

> 详见规则 **2.4 CHANGELOG 克制**。

CHANGELOG.md 位于 `dev/<name>/CHANGELOG.md`（开发侧，不随技能分发）。格式遵循 Keep a Changelog。版本号不在开发过程中逐次递增，功能完成后再统一确定。条目以逻辑变更为单位。

### 4.4 许可证规则（🔧 metapowers 追加 · 条件性必须）

> 技能可以选择不使用许可证（保留所有权利），也可选择任意许可证。本规则仅约定当选择非公开引用许可证时 LICENSE 文件的处理方式。

**规则**：如果技能采用**非标准、自定义或非公开引用**的许可证（即不属于 MIT、Apache-2.0、GPL-2.0、GPL-3.0、BSD-2-Clause、BSD-3-Clause、LGPL、MPL-2.0、Unlicense、CC0-1.0、ISC 等常见开源许可证的），则必须：

1. 在 `content/` 目录下创建 `LICENSE` 文件，包含许可证原文
2. 在 frontmatter `metadata.license` 中引用该许可证的名称或描述

对于上述标准开源许可证，只需在 `metadata.license` 中填写 SPDX 标识符，无需创建 `LICENSE` 文件。

对于未选择任何许可证的技能，可以省略 `license` 字段，也表示「保留所有权利」。

### 4.5 设计原则

#### 技能无硬性依赖

技能的设计和运行不得依赖其他技能、项目中的特定文件或运行环境中的特定信息。成套技能之间只能互相推荐（在 description 或「与其他技能的关系」中提及），不能设计成硬性依赖。

| 允许 | 禁止 |
|------|------|
| 在 description 中推荐相关技能 | SKILL.md 中写「必须先运行 X 技能」 |
| 在「与其他技能的关系」中说明协作关系 | references/ 引用另一个技能的文件 |
| 独立运行，不假设环境状态 | 假设项目根目录存在特定文件结构 |
| 根据 `available_skills` 列表动态选择 | 在流程中硬编码依赖某个特定技能名 |

#### 目录树从技能根开始

SKILL.md 中展示技能目录树时，根目录必须是技能自身（如 `my-skill/`），不得包含上层分类目录（如 `dev/my-skill/content/` 或 `skills/my-skill/`）。上层路径是磁盘位置细节，不属于技能身份的一部分——技能每次移动目录都得改树，纯属维护噪音。

## 5. 通用性门槛

本仓库只收通用 Skill。一个 Skill 只有在「做完全不同项目的人也能用上」时才适合放进来。

| 不适合 | 替代方案 |
|--------|----------|
| 依赖特定第三方工具或服务 | 做成独立插件 |
| 只服务某个团队或工作流 | 做成独立插件 |
| 垂直领域（游戏开发、金融分析等） | 做成独立插件 |
| 「这样可能更好」但没有实际问题驱动 | 等遇到真实问题再来 |

---

## 6. 行为约束

- **真实需求驱动**：不解决理论推测的问题。如果用户没有讲清具体场景，先问清楚。
- **展示改动**：提交前必须让用户看过完整 diff 并得到明确同意。
- **不编造**：问题描述、功能说明、测试结果必须真实。
- **先计划后动手**：任何非平凡操作之前，必须先制定计划（目标、现状、步骤、验证方式）。
- **文档先行**：在实现之前先写出设计文档。代码实现可以修改，但没有书面设计的实现无法被审查。
