# DESIGN

> 本文档记录 cx-agent-memory 技能的设计意图和核心决策，作为维护参考。

---

## 设计初衷

让 Agent 能够留下**"给未来自己的笔记"**——这是本技能的核心价值。

**关键认知**：记忆库是**Agent 服务于自己**的工具，不是给用户看的文档系统。用户是引导者和建议者，Agent 自己判断是否困扰、如何组织、何时维护。

---

## 核心原则

### 原则1：最小约束，最大自主

**唯一强制约定**：
```
index.md 是记忆库的唯一入口
```

**其他全部开放**：
- 命名：Agent 自己决定
- 结构：Agent 自己组织
- 维护：Agent 自己判断时机
- 约定：Agent 自己制定和调整

**设计理由**：约束越少，Agent 越能发展出**适合自己工作方式**的记忆库。

### 原则2：演化优于预设

记忆库是**长出来的**，不是**设计出来的**。

- 不预设"正确的"命名规范或目录结构
- 维护时**先观察现状，再制定约定**，而非套用模板
- 约定**随使用而调整**，可演进、可废弃
- 接受"当前足够好"，不强求"未来也完美"

### 原则3：入口至上

**最关键的验证**：
> 如果重新加载 index.md，Agent 能否在**10秒内**理解记忆库结构？

- index.md 是**唯一入口**，必须始终有效
- 结构拆分、配置分离都可以，但必须在 index 中**明确标记**
- 维护后的**第一验证**是"入口是否仍然有效"

---

## 核心思想

**约束 > 流程**：不规定具体的记录格式或执行步骤，而是定义核心约束，让 Agent 根据具体场景自行判断。

---

## 关键决策

| 决策 | 理由 |
|------|------|
| 不规定文件组织方式 | 不同项目需求不同，Agent 根据项目特点自行判断更灵活 |
| index.md 作为核心索引 | 提供快速定位能力，查询比记录更频繁 |
| 触发时机灵活化 | 重要决策、用户要求、会话结束前都可触发，Agent 自行判断 |
| 整理作为可选项 | 不强制整理，只在需要时触发，由 Agent 判断是否必要 |
| 维护流程：诊断-决策-执行-验证 | 强调 Agent 自主判断和前瞻性思考，而非执行固定步骤 |
| 约定文档版本化 | 约定随记忆库演化，保留历史可追溯 |

---

## 迭代历史

- **v1.0**：初始版本，包含完整的设计规划和示例文件
- **v2.0**：重构版本，精简 SKILL.md，删除冗余内容，保留核心约定
- **v2.1**：深化维护流程，从"步骤清单"改为"诊断-决策-执行-验证"框架，强调 Agent 自主和演化思维
- **v2.2**：根据 prompt-engineering 方法论审查，新增异常处理、决策框架、触发正例/负例
- **v2.3**：修复 index.md 强制协议表述，区分"读取"与"编辑"场景
- **v2.4**：全面梳理，修复概念不一致（启动协议定义），补全 AGENTS.md 集成，优化触发描述

---

## 待观察

- Agent 对"最小约束"理念的理解是否一致
- 维护流程的实际使用：Agent 是否真的能自主诊断和决策
- Index 入口验证是否能有效防止"维护后反而更难用"的情况
- 约定文档版本化是否真有必要，还是会被忽略
- v2.4 的触发优化是否解决了"编辑前忘记加载技能"的问题
- AGENTS.md 集成的新设计是否足够清晰，能否实际帮助用户建立启动协议
