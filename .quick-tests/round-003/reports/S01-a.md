## 2. 停顿 / 困惑 / 误解 记录

### 困惑点 #1 — 示例查询词 vs 用户真实需求
- **位置**: `SKILL.md:43-45`
- **描述**: 文档给的示例是 `web_search(query="python async framework", limit=5)`，但用户的需求是"Python asyncio 的 TaskGroup 功能"。这是个精确需求，用户面临选择：
  - 照抄示例中的 `"python async framework"`（文不对题，返回结果可能不包含 TaskGroup）
  - 用自己的精确词 `"python asyncio taskgroup"`（偏离示例，但符合需求）
- **文档缺失**: 没有说明"示例查询仅供参考，请根据你的实际需求修改 query 参数"。新手用户可能误以为必须照抄示例的格式。
- **严重度**: ⚠️ **中** — 经验用户能自行判断，纯新手可能困惑

### 困惑点 #2 — 后端配置检查需要先行知识
- **位置**: `SKILL.md:74-77`
- **描述**: 文档指示执行 `grep -A3 "^web:" ~/.hermes/config.yaml` 来检查当前后端。但模拟用户是"从未用过 web_search 工具"的 Python 开发者，这意味着：
  1. 他不知道 `~/.hermes/config.yaml` 文件是否存在/在哪
  2. 他不知道 `web.backend` 是什么意思
  3. 他不知道什么后端适合自己
- **文档缺失**: 没有为首次使用的用户提供"什么是后端？为什么需要检查？"的入门说明。文档假定读者已经有 Hermes 配置的先验知识。
- **严重度**: ⚠️ **中-高** — 裸上下文的新手用户在此处可能卡住

### 困惑点 #3 — 推荐后端但未解释"keyless MCP"
- **位置**: `SKILL.md:79-80`
- **描述**: 文档推荐切换到 `parallel` 后端，并标注"免费、keyless、国内可通"。但"keyless MCP"这个术语对新手没有解释——什么是 MCP？什么是 keyless？为什么 parallel 不需要 API key 而其他后端需要？
- **文档缺失**: 没有对"后端"机制的简明解释，术语墙。
- **严重度**: ⚠️ **中** — 有经验的开发者可以推测，但精确含义模糊

### 困惑点 #4 — 查询策略表缺少"标准库文档搜索"场景
- **位置**: `SKILL.md:113-119`
- **描述**: 策略表有四行（搜产品/项目、搜概念/知识、站内搜索、搜英文资料）。用户的需求是"Python 标准库文档"——既不是产品/项目，也不完全是概念/知识。最接近的是"搜概念/知识"但示例 `"fastapi websocket real-time chat"` 是第三方库而非标准库。
- **衍生问题**: 用户可能想用 `site:docs.python.org` 站内搜索策略，但那是第3行，而用户的场景是从0开始的搜索，不是已经知道目标站点后的站内搜索。
- **严重度**: ⚠️ **低-中** — 可以凑合用，但不完美匹配

### 困惑点 #5 — "后备搜索"和"提取"的边界模糊
- **位置**: `SKILL.md:88-98`（2.1a 备用搜索发现：浏览器搜索）
- **描述**: 第2.1a节说当 `web_search` 不可用时，可以用 `browser_navigate` 到搜索引擎首页进行"搜索发现"。但硬约束禁止使用 `browser_navigate`，所以这条路径被阻塞。
- **文档问题**: 第2.1a节独立于决策树主流程之外，容易被新手跳过。它的位置在 2.1 和 2.2 之间，编号为 "2.1a"，这种编号方式不像标准编号（2.1, 2.2），可能被误解为附录或附加说明。
- **严重度**: ⚠️ **低** — 编号方式不标准

### 困惑点 #6 — "web_extract" 与三层 fallback 的关系不清晰
- **位置**: `SKILL.md:237-249`（4.0 内置工具：web_extract）
- **描述**: 文档在第4.0节介绍了 `web_extract` 内置工具，然后在4.0a节给出"提取链可用性检查"决策树，再到4.1-4.3的三层 fallback 链。但实际上：
  - `web_extract` 是 Layer 0（最优先）
  - 4.0a 是是否使用终端链的决策检查
  - 4.1-4.3 是终端链的具体实现
- **层级关系不直观**: 第一次阅读容易把 4.0→4.0a→4.1→4.2→4.3 理解为"5步顺序流程"，而不是"Layer 0 → 决策分支 → Layer 1/2/3"
- **严重度**: ⚠️ **中** — 结构可优化

## 4. 异常路径推演

### 异常路径 A: 跳过 2.1.1 后端配置，直接调用 web_search

**触发**: 新手用户看到 2.1 节的示例 `web_search(query="...")`，觉得"看起来很简单"，跳过 2.1.1 的"后端配置（关键）"章节。

**推演**:
1. 用户跳过 Line 59-87（后端配置），以为 `web_search` 像 Python 内置函数一样开箱即用
2. 在 Line 43 处尝试调用 `web_search(query="python asyncio TaskGroup", limit=5)`
3. **[模拟阻塞]** — 实际上当前环境禁止调用 web_search，但即使不禁止，如果后端未正确配置（例如默认是 `ddgs` 且在国内网络环境），调用会超时失败
4. 用户回到文档翻到 2.4 失败处理表，发现需要检查后端配置 → 被迫回到跳过的 2.1.1
5. 如果用户没耐心，可能认为"这工具不好用"而放弃

**发现**: 2.1.1 节的"关键"标注是对的，但它放在示例代码之后而不是之前。用户可能先看到示例代码就尝试了，出错了才回来看。建议把后端检查前置到 2.1 之前。

### 异常路径 B: 误解"browser"为常规浏览器搜索

**触发**: 用户看到 1.0 决策树最底层写着 `Layer 3: browser ← 兜底（最通用）`，但不确定"browser"是指"用浏览器查看"还是"用浏览器搜索"。

**推演**:
1. 用户对 `web_search` 的结果不满意（或 web_search 不可用）
2. 看到 2.1a "备用搜索发现：浏览器搜索"(Line 88-98)
3. 用户理解：可以用 `browser_navigate` 直接访问 Python 官方文档网站 `https://docs.python.org` 然后手动查找
4. 但 `browser_navigate` 本身是"打开指定 URL"，不是搜索工具
5. **[术语模糊]**: "浏览器搜索" 在这里明确指"在浏览器中打开搜索引擎进行搜索"，而不是"打开目标网站手动翻阅"。文档在第 2.1a 节说了"浏览器导航到搜索引擎首页（如 Bing）进行搜索发现"，但第 4.3 节又用 browser 来做"内容提取"的兜底。
6. 同一个工具（`browser_navigate`）在两个不同场景下用途不同，文档没有明确说明如何区分"用浏览器做搜索发现"和"用浏览器做内容提取"。

**发现**: `browser_navigate` 的双重用途（搜索发现 vs 内容提取）是合理的，但缺少交叉引用来帮用户区分场景。

## 5. 意外的正面/负面发现

### ✅ 正面发现

1. **Line 121-129 失败处理表非常实用** — 对常见的 `web_search` 失败场景按"现象 → 处理"给出了清晰的操作步骤，新手可以照做。

2. **Line 86 和 references/web-backends-china.md 引用完整** — 从主文档到子文件的链接是活链，子文件提供了有价值的补充信息。

3. **Line 31-35 核心原则清晰** — 用列表形式总结关键约束（搜索引擎主入口、备用搜索引擎、curl 限用场景、浏览器是兜底），比散落在正文中更容易记忆。

4. **Line 239-249 对 `web_extract` 与后端配置关系的说明** — 明确指出如果 backend 是 `ddgs` 则只能搜索不能提取，避免了用户困惑。

5. **Section 6.1 JSON 处理原则 (Line 340-363)** — 对 `jq` 标准化的强调，以及"禁止 python3 -c 管道解析 JSON"的明确规则，是好的工程实践。

### ❌ 负面发现

1. **决策树缺少备用分支** — 核心决策树(Line 16-28)只展示了主路径，`web_search` 失败时的浏览器搜索 fallback 没有在图里体现。

2. **编号 2.1a 非常规** — 在 2.1 和 2.2 之间插入"2.1a"不符合标准编号层级，新手可能跳过或不理解它与 2.1 的关系。

3. **"search-synthesis 模式"缺少入口** — Line 262 突然引入这个概念，没有定义、没有链接到其他文档。对新手用户来说这是一个"陌生术语突然出现"的时刻。

4. **design-journal.md 对新手无用** — 该文件记录了协议设计时的决策思路（如"`web_search` tool deprecated"——但这和当前版本的 SKILL.md 中 `web_search` 是主入口的说法矛盾）。如果新手无意中读到，可能对 skill 当前状态产生困惑。

5. **`--json` 字段名易混淆** — Line 222-228 指出的 `gh repo view --json`（GraphQL 驼峰）vs `gh search repos --json`（另一套驼峰）vs `gh api` REST（蛇形）的问题，文档自己意识到了这个坑但没有给出完整的字段名速查表。
