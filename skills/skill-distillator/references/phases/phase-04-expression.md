# 第四阶段：表达

> 将编排后的内容转化为人类可读的方法论文档。

---

## 目标

按文档大纲生成最终输出文件，完成从"指令型"到"说明型"的视角转换。

## 为什么需要这些知识文档

编排阶段确定了"写什么、按什么结构写"，表达阶段解决"怎么写得好"。需要：
- **3C 原则**：确保表达清晰、简洁、一致
- **风格指南**：统一语态、时态、人称
- **指令→说明转换**：将 Agent 指令转化为人类可读的说明

## 如何加载知识文档

按顺序加载：
1. [writing/three-c-principles.md](../writing/three-c-principles.md) — 确保表达质量
2. [writing/style-guide.md](../writing/style-guide.md) — 统一写作风格
3. [writing/instructional-to-explanatory.md](../writing/instructional-to-explanatory.md) — 完成视角转换

加载方式：读取文件内容，提取写作规范，应用到文档生成。

## 核心原则：智能转换

**转化 ≠ 简化，转化 = 视角转换 + 必要丰富**

| 原始内容质量 | 处理方式 |
|-------------|----------|
| 描述清晰、适合人类阅读 | 直接摘取或微调 |
| 描述完整但偏指令型 | 视角转换（指令→说明），保持核心文字 |
| 描述过于简略 | 基于 DESIGN.md 和上下文丰富 |
| 关键概念未定义 | 补充定义和背景 |

**不可简化的地方**：
- 功能描述
- 约束条件
- 触发规则
- 核心定义

## 类型自适应策略

根据理解阶段识别的技能类型，调整转换策略：

| 技能类型 | 转换策略 |
|----------|----------|
| 纯指令型 | 授权推断设计理由，所有推断标注 `[推断: 依据]`；检查清单按"定义+检测方法"模式转换；决策规则表按"场景-行为-理由"格式转换 |
| 混合型 | 标准流程；对清晰内容直接摘取（参见"清晰则摘取"判断标准）；对指令型内容转换 |

## 推断边界定义

推断授权不等于无限推断。明确边界：

| 类别 | 范围 | 示例 |
|------|------|------|
| 可推断 | 从约束反推理由 | "5000行上限" → "防止上下文溢出" `[推断: 基于约束反推]` |
| 可推断 | 从行业常识推断 | "检查SQL注入" → "SQL注入是OWASP Top 1风险" `[推断: 行业常识]` |
| 可推断 | 从流程逻辑推断 | "先检测语言" → "不同语言有不同检查规则" `[推断: 基于流程逻辑]` |
| 不可推断 | 设计者主观取舍 | "为什么选4维度而非5" → `[未推断: 需设计者主观取舍]` |
| 不可推断 | 需要项目上下文 | "为什么用这个阈值" → `[未推断: 需项目上下文]` |
| 不可推断 | 超出领域知识 | 具体业务逻辑 → `[未推断: 超出领域知识]` |

兜底规则：当无法确定是否可推断时，不推断，标注 `[未推断: 无法确定推断边界]`。

所有推断内容必须标注 `[推断: 依据]`，与原始信息明确区分。

## 决策规则表转换规范

原始技能中的"条件→动作"决策规则表，按以下规范转换：

- 保留表格形式（不改为文字）
- 增加"设计理由"列（可从约束反推或标注 `[推断: 依据]`）
- 表头从"条件→动作"改为"场景→行为→理由"

转换示例：

原始：
| 条件 | 动作 |
|------|------|
| 文件 > 500 行 | 标记大文件警告 |

转换后：
| 场景 | 行为 | 理由 |
|------|------|------|
| 文件超过 500 行 | 标记大文件警告并建议拆分 | 大文件通常意味着职责不单一，维护成本高 [推断: 基于约束反推] |

## 操作步骤

### 步骤1：按大纲生成各文件

- 单文件输出：按章节顺序生成
- 多文件输出：逐个文件生成

### 步骤2：应用风格指南

- 主动语态
- 第二人称
- 现在时态

### 步骤3：完成视角转换

- 祈使句 → 陈述句
- 步骤清单 → 流程说明
- 约束声明 → 设计理由

### 步骤4：丰富必要内容

- 补充核心概念的定义
- 说明设计理由
- 增加背景信息

### 步骤5：输出最终文档

## 输出格式

按编排阶段确定的文档大纲输出。
