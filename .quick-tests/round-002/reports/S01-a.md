## 步骤 ❹：四步流程之「逐帧跟」（SKILL.md L46-L55）

**文档指令**：
> 按客体消费的顺序逐段走，记录每步的心理反应。不要跳到结论。
> 记录格式（内心独白）：
> - "这里停顿了" → 信息密度过高或措辞模糊
> - "这里被吓到了" → 警告/限制过多，客体可能放弃
> - "这里跳过了" → 结构上没有抓住注意力
> - "这里误解了" → 措辞有歧义
> - "这里不知道该做什么" → 缺少明确的下一步

**模拟操作**：
1. 以「第一次读此提示词的 Agent」身份，逐段读代码审查提示词。
2. 记录每段的内心独白。

**模拟逐帧记录**（对「代码审查 Agent 系统提示词」的推演）：

| 帧 | 提示词内容（模拟） | 内心独白 | 反应类型 |
|----|-------------------|---------|---------|
| 1 | 「你是一个代码审查者」 | OK，角色明确。 |（正常）|
| 2 | 工作流程：读取 diff → 逐文件检查 → 按严重度分类 → 写行内评论 | 清晰，四步流程。但是「严重度分类」的标准只在后面出现——需要跳读确认。 | **这里停顿了** — 依赖后文定义，可能导致跳读 |
| 3 | 禁止事项：不修改代码、不执行重构、不对风格偏好置评 | 三个禁令明确。「风格偏好」的范围——代码风格（lint）还是架构风格？ | **这里误解了** —「风格偏好」可能被窄化或宽化解释 |
| 4 | 审查维度：正确性 > 安全性 > 性能 > 可读性 | 优先级链清楚。但如果一个 PR 既有正确性问题又有安全问题——按链优先处理安全？还是全部报告但排序？ | **这里停顿了** — 未明确「多个问题在同一文件时如何决定先报哪个」 |
| 5 | 输出格式：每条评论包含「文件:行号」「严重度」「问题描述」「建议修复」 | 格式明确，「严重度」用什么标签（P0-P3? 高/中/低?）没说。 | **这里不知道该做什么** — 缺少严重度标量定义 |

**后续推测**：读到第 5 帧时，Agent 可能回退到第 2 帧的「严重度分类」寻找标准，产生来回跳读。

**对技能文档的评价**：
- **亮点**：五个内心独白标签非常实用，覆盖了最常见的受阻模式 ✅
- 示例（行 97-137）给出了具体的推演记录格式，对初学者极有帮助 ✅
- **困惑点**：文档说「按客体消费的顺序逐段走」（行 48），但**没有定义「段」是什么**。对提示词而言，可能是自然段落；对 UI 可能是屏幕；对 API 文档可能是 endpoint。载体不同，「段」的语义不同，文档没有提示如何判断。
- **困惑点**：行 47 的「不要跳到结论」很重要但没有解释「什么是结论」。在推演过程中，什么算「跳到结论」？是在还没读完时就写评价？还是过早推断最终结果？建议加一个例句说明。

## 步骤 ❺：四步流程之「边界刨」（SKILL.md L57-L63）

**文档指令**：
> 故意走非正常路径，至少三类：
> - **信息缺失**：客体缺少某条前置知识，或跳过了某个章节
> - **错误输入**：客体做了错误操作，或误解了某条指令
> - **跨模块跳转**：客体从中间进入，或从另一条路绕过来

**模拟操作**（对代码审查提示词应用三类边界场景）：

| 边界类型 | 模拟场景 | 推演过程 | 发现 |
|---------|---------|---------|------|
| **信息缺失** | Agent 不知道「严重度」的标签体系（提示词只在格式中写了"严重度"但未定义标量） | Agent 读到输出格式要求填写严重度 → 无标量定义 → 自创标签（如「低级」「中级」「高级」 vs 「P0/P1/P2」）→ 输出不一致 | **问题：严重度标量缺失** |
| **信息缺失** | Agent 跳过「禁止事项」段直接开始审查 | Agent 修改了代码（违反禁令）→ 无自检门控 → 行 116 的「自检问题」模式能拦截此情况吗？本提示词没有自检问题区块 | **问题：缺少自检门控** |
| **错误输入** | Agent 将「不执行重构」误解为「不对任何代码提出改动意见」 | Agent 仅标记问题不提供建议修复 → 但输出格式要求「建议修复」→ 输出不符合格式 → 格式约束与「重构」禁令的边界模糊 | **问题：禁令与格式要求存在潜在矛盾** |
| **跨模块跳转** | Agent 从另一个任务中途转来做代码审查 | Agent 保持上一个任务的角色设定混入 → 可能用「教学语气」而非「审查语气」→ 提示词未定义角色切换/上下文复位 | **问题：缺少角色复位机制** |

**对技能文档的评价**：
- 三类边界设计合理。但在我的场景中（提示词推演），**「跨模块跳转」的适用性较低**——Agent 阅读提示词不像用户浏览网站那样有「从中间进入」的路径。Agent 要么被完整注入提示词，要么没有。「中间进入」在提示词场景中更接近「跳过前文直接读后段」的语义。
- **困惑点**（行 62）：「跨模块跳转」的定义「客体从中间进入，或从另一条路绕过来」明显借鉴了 UI/文档的使用场景，对提示词场景映射不自然。文档如果能分类讨论「不同载体的边界刨侧重点」会更好。
- 示例（行 119-127）的边界刨展示了三个具体场景，非常有助于理解 ✅

## 步骤 ❻：四步流程之「打分」（SKILL.md L65-L77）

**文档指令**：
> 分维度给判断，不笼统说"还行"。每个维度用 ★ 打分（1-5），并指出具体缺口。
> 维度根据载体类型选择

**载体类型匹配**：
- 推演对象是「系统提示词」→ 适用「提示词/指令」维度（行 73）

| 维度 | 评分 | 说明（对被测代码审查提示词的模拟打分） |
|------|------|---------|
| 因果方向清晰度 | ★★★☆☆ | 「如果 X 则 Y」的因果链不完整——严重度分类没有标量定义，Agent 不知道用什么标准 |
| 违规模式覆盖 | ★★★☆☆ | 三条禁令明确，但缺少「不知道怎么做时」的回退模式；缺少「多个维度冲突时」的优先级裁决规则 |
| 自检可执行性 | ★★☆☆☆ | 无自检门控问题——Agent 读完不知道「我怎么知道自己做对了」。行 116 示例中的自检问题模式此处不存在 |
| 后果说服力 | ★★★★☆ | 审查维度的重要性顺序（正确性 > 安全 > 性能 > 可读性）明确，逻辑合理 |
| 边界覆盖 | ★★☆☆☆ | 缺少严重度标量定义、缺少角色复位机制、缺少「格式要求 vs 禁令冲突」时的裁决规则 |

**对技能文档的评价**：
- 维度表（行 71-76）针对性很强，不同载体有不同推荐维度 ✅
- **困惑点**：行 73 的维度**缺少每个维度的定义**。例如「因果方向清晰度」——是指提示词中条件→结果的对应关系是否顺畅？还是指 Agent 能清晰理解「如果我怎么判断，会有什么后果」？无定义会导致不同使用者给同一份输入打出不同分。「自检可执行性」同理。
- 好在行 129-137 的示例中每个维度都有「说明」列，可以倒推出维度的含义。但推荐在维度表中直接加「含义」列，降低初学者的认知负担。
- **矛盾点**：主 SKILL.md 行 73 的提示词维度与行 53 的文档维度在名称和数量上不同，但参考文件 `doc-guide-validation.md` 行 48-56 的文档维度又有第三套（入口明确性、步骤可跟踪性、术语一致性、缺失场景覆盖、独立性）。这三套维度表之间没有「用法选择指南」。用户可能困惑：我推演文档时该用主文档的维度还是参考文档的维度？
