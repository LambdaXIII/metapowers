## 1. 工具作为契约

Anthropic 指出：工具定义了 Agent 与其**行动空间**和**信息空间**之间的**契约**。

```
┌─────────────────────────────────────────────────────────┐
│                       Agent                             │
│     ┌──────────────────────────────────────────┐        │
│     │          工具契约 (Tool Contract)          │        │
│     │  工具 = Agent 与外部世界的「API 边界」      │        │
│     └──────────────────────────────────────────┘        │
│                ↕                   ↕                     │
│          ┌──────────┐      ┌──────────────┐             │
│          │行动空间   │      │ 信息空间      │             │
│          │(Action)   │      │ (Data)       │             │
│          └──────────┘      └──────────────┘             │
└─────────────────────────────────────────────────────────┘
```

**为什么是「契约」？** 工具不是代码实现的细节，而是 Agent 能力的公共接口。契约双方的边界清晰程度，直接决定 Agent 行为的可预测性。当工具语义模糊时：
- Agent 会发明参数值
- 会调用错误的工具
- 会在错误的时间重试

> 一个好的工具契约应该让 Agent 不需要「猜测」——猜测就是 bug 的前兆。

## 2. 四项设计需求

每个工具的定义必须满足以下四项需求：

| # | 需求 | 说明 | 反例 | 正例 |
|---|------|------|------|------|
| 1 | **Token 高效** | 定义简洁、自包含，不浪费注意力预算 | 200 字的参数描述，包含冗余说明 | `product_id (string): 产品 UUID，来自 search_products 结果` |
| 2 | **输入参数描述明确** | 消除歧义，让 Agent 知道每个参数的含义和来源 | `id: 产品 ID` | `product_id (string, required): 产品的 36 位 UUID，来自 search_products 返回的 product.id` |
| 3 | **对错误鲁棒** | 返回清晰的、结构化的错误信息 | 返回 `Error 500` | 返回 `{"error": "PRODUCT_NOT_FOUND", "message": "product_id xyz 不存在", "suggestion": "请先调用 search_products"}` |
| 4 | **功能最小重叠** | 如果人类工程师无法确定应该用哪个工具，AI Agent 也不行 | 既有 `send_email` 又有 `send_message_to_user` | 只有一个 `send_email`，职责清晰 |

### 2.1 功能性重叠的判定标准

```text
问：两个工具之间是否存在功能性重叠？
├── 如果人类工程师需要看文档才能区分 → 重叠太大，合并
├── 如果同一任务可以用两个工具完成 → 重叠太大，合并或拆分职责
└── 如果两个工具服务于不同的调用场景 → 合理
```markdown

### 2.2 参数描述的黄金法则

- **描述「什么」，不描述「怎么获取」**：参数来源逻辑应放在调用链中，而非提示词中
- **用具体值约束代替自然语言规则**：`choices: ["read", "write", "admin"]` 优于 `permission: 权限级别，只能是 read、write 或 admin`
- **标记必填/可选**：明确标注 `required` vs `optional`，避免 Agent 跳过关键参数

## 3. 最小可行工具集

### 3.1 核心理念

管理一个**最小可行工具集（Minimum Viable Toolset）**。每个工具必须通过严格的「成本-收益」审查：

```text
问题：这个工具值得占用注意力预算吗？
│
├── 它被调用的频率有多高？
├── 没有它，Agent 能否完成任务（通过其他路径）？
├── 它的定义消耗多少 token？
└── 保留它 vs 移除它，对用户体验的影响是什么？
```markdown

**每多一个工具，Agent 的选择难度就增加一分**——臃肿的工具库会导致：
- Agent 花费更多 token 在工具选择推理上（而非任务本身）
- 工具名相似导致调用错误
- 上下文被大量工具定义污染

### 3.2 Anthropic 的 Tool Search 数据

Anthropic 内部实验数据（延迟加载 vs 全部加载）：

| 指标 | 全部工具加载（Baseline） | 使用 Tool Search（延迟加载） |
|------|------------------------|-----------------------------|
| **上下文保留率** | 23% | **95%** |
| **准确率** | 49% | **74–88%** |
| **Token 开销** | 高（所有工具定义始终在上下文中） | 低（仅加载被搜索到的工具） |

> **结论**：工具不是越多越好。每个工具定义消耗注意力预算，而注意力预算的边际回报递减。

### 3.3 什么时候应该添加一个工具？

只有满足以下**全部**条件时才添加新工具：

1. ✅ 当前工具集**无法**完成某个必要任务
2. ✅ 该任务在用户场景中出现频率 ≥ 5%
3. ✅ 工具定义 ≤ 200 tokens（含参数描述）
4. ✅ 与现有工具的功能重叠 ≤ 20%

## 4. 工具类别（OpenAI 框架）

OpenAI 将 Agent 工具分为三个基本类别：

| 类别 | 英文 | 描述 | 示例 |
|------|------|------|------|
| **数据** | Data | 检索上下文和信息 | 查询 CRM、读取 PDF、搜索网页、数据库查询 |
| **行动** | Action | 与系统交互执行操作 | 发送邮件、更新记录、创建工单、转交人工 |
| **编排** | Orchestration | Agent 本身作为其他 Agent 的工具 | 退款 Agent、研究 Agent、代码审查 Agent |

### 4.1 各类别的设计考量

#### Data 工具
- **关注点**：响应速度、数据新鲜度、结果大小控制
- **最佳实践**：限制返回结果数量（如 top-5），避免 Agent 上下文被大量数据淹没
- **参数设计**：必须有分页/限制参数

#### Action 工具
- **关注点**：幂等性、审批门、审计日志
- **最佳实践**：对破坏性操作要求显式确认；记录每次调用的完整轨迹
- **参数设计**：所有参数必须有合理的默认值或明确的必填标记

#### Orchestration 工具
- **关注点**：任务分解、结果聚合、错误传播
- **最佳实践**：定义清晰的输入/输出契约；子 Agent 的结果需要结构化为父 Agent 可消费的格式
- **参数设计**：输入参数应包含目标、约束和返回格式要求

### 4.2 工具类别与安全等级

| 类别 | 默认安全等级 | 审批门建议 |
|------|------------|-----------|
| Data | 低风险 | 无需审批（但需监控异常查询模式） |
| Action | 中到高风险 | 破坏性操作必须审批 |
| Orchestration | 中风险 | 视子 Agent 权限而定 |

## 5. 工具调用协议

来自生产级 Agent（same.new）的最佳实践——**5 条规则及其设计原理**：

```markdown
<tool_calling>
  1. 始终严格遵循工具调用 schema
  2. 用自然语言解释你将调用哪个工具以及为什么
  3. 一次只调用一个工具（除非明确允许并行调用）
  4. 等待工具结果后再继续
  5. 实现错误恢复（最多 3 次重试）
</tool_calling>
```markdown

### 5.1 每条规则的设计原理

#### 规则 1：始终严格遵循工具调用 schema

**为什么：** LLM 倾向于「创造力」，会在参数中插入额外信息或改变格式。schema 是契约的具体形式——偏离 schema = 契约破裂。

**实现方式：**
- 在系统提示词中明确声明："You MUST follow the tool's JSON schema exactly. Do not add, remove, or reorder parameters."
- 在参数描述中包含类型和格式约束
- 在应用层进行 schema 验证，拒绝无效调用

#### 规则 2：用自然语言解释将调用哪个工具以及为什么

**为什么：** 强制 Agent「说出推理过程」可以减少幻觉调用。当 Agent 需要解释时，它更有可能选择正确的工具。

**示例：**
```markdown
❌ Agent 直接调用工具（无解释）:
  search_products(query="laptop")

✅ Agent 先解释再调用:
  "用户想要查找笔记本电脑。我将使用 search_products 工具搜索相关产品。"
  → search_products(query="laptop")
```markdown

#### 规则 3：一次只调用一个工具（除非明确允许并行调用）

**为什么：** 并行调用引入复杂的依赖关系。一个工具的输出可能影响下一个工具的参数。顺序执行比并行更容易调试和审计。

**例外条件（允许并行）：**
- 工具之间没有数据依赖
- 两个工具都不改变系统状态（只读）
- 用户明确要求并行

#### 规则 4：等待工具结果后再继续

**为什么：** Agent 经常「预判」工具结果继续推理，导致幻觉。必须等待实际结果。

**实现方式：**
- 在系统提示词中声明："After calling a tool, STOP. Wait for the result before continuing."
- 在应用层阻止 Agent 在工具结果返回前进行后续推理

#### 规则 5：实现错误恢复（最多 3 次重试）

**为什么：** 网络错误、限流、暂时性故障是常态。但无限重试会导致无限循环。

**详见 [第 6 节 — 错误恢复策略](#6-错误恢复策略)**

## 6. 错误恢复策略

### 6.1 DeepMind 框架下的错误分类

DeepMind Gemini 3 Pro 系统指令（2025 年 11 月发布）定义了智能错误恢复策略：

```text
错误恢复规则
│
├── 瞬态错误 (Transient) —— 必须重试
│   ├── 网络抖动
│   ├── 429 限流 (Rate Limit)
│   ├── 500 Internal Server Error
│   ├── 超时 (Timeout)
│   └── 策略: 重试，最多 3 次，指数退避
│
└── 其他错误 (Non-transient) —— 必须改变策略，绝不重复
    ├── 404 不存在
    ├── 401/403 无权限
    ├── 400 参数非法
    ├── 逻辑错误（数据不符合预期）
    └── 策略: 改变策略，不重复相同的失败调用
```markdown

### 6.2 系统提示词中的表述

```markdown
## Error Recovery
- Transient errors (network timeouts, 429 rate limits, 500s):
  → MUST retry (up to 3 attempts with exponential backoff)
- All other errors (404, 401, 400, logic errors):
  → MUST change strategy, do NOT repeat the same failed call
  → Try alternative approach or inform the user
```markdown

### 6.3 重试策略的理性边界

| 重试次数 | 适用场景 | 风险 |
|---------|---------|------|
| 0（不重试） | 非关键只读查询 | 偶发失败影响体验 |
| 1-3 次 | 大部分场景 | 平衡可靠性与延迟 |
| 3-5 次 | 关键写入操作 | 可能产生重复写入（需幂等性） |
| 5+ 次 | 不推荐 | 无限循环风险；token 浪费 |

### 6.4 实践陷阱

- **陷阱 1**：Agent 在非瞬态错误上重试 → 每次失败都消耗 token，最终超时
  - **解决**：在系统提示词中清晰区分「可重试」和「不可重试」错误
- **陷阱 2**：重试时使用相同参数 → 对 429 正确，但对 400 错误无意义
  - **解决**：要求 Agent 在重试前检查参数
- **陷阱 3**：Agent 陷入「重试循环」→ 消耗大量 token 和时间
  - **解决**：设置硬上限（最多 3 次），之后必须改变策略

## 7. 上下文工程视角下的工具管理

### 7.1 注意力预算分配

> → 注意力预算的理论基础见 [context-engineering.md §3](context-engineering.md#3-注意力预算attention-budget)

从 Context Engineering 视角（Anthropic, 2025），工具定义是注意力预算的主要消耗者之一：

```text
系统提示词注意力预算分配（典型分布）
│
├── 角色定义与上下文: 10-15%
├── 核心指令与工作流: 15-25%
├── 工具定义: 30-50%  ← 最大开销
├── 输出格式与约束: 10-15%
├── 安全与边界: 10-15%
└── Few-Shot 示例: 5-10%
```markdown

**工具定义是注意力预算的最大消耗者**——这意味着优化工具定义的 token 效率是优化系统提示词整体效率的最佳切入点。

### 7.2 工具定义的优化策略

| 策略 | 方法 | Token 节省 |
|------|------|-----------|
| **缩短参数描述** | 用示例值代替长篇描述 | 30-50% |
| **合并相似工具** | 用 `action` 参数区分分支 | 40-60% |
| **延迟加载** | 仅加载当前场景需要的工具 | 70-90% |
| **使用枚举值** | 用 `choices` 约束代替自然语言描述 | 20-40% |
| **移除低频工具** | 删除调用率 < 5% 的工具 | 变量 |

### 7.3 延迟加载的实现模式

```text
[系统提示词核心] ← 始终加载
  - 角色定义
  - 核心指令
  - 通用工具（文件读写、搜索）
  - 安全规则

[已按需激活的工具集] ← 由 Agent 搜索/触发
  - 场景 A 工具（如：Git 操作）
  - 场景 B 工具（如：数据库查询）
  - 场景 C 工具（如：邮件发送）
```markdown

## 8. 常见反模式

| # | 反模式 | 表现 | 正确做法 |
|---|--------|------|---------|
| 1 | **工具过载** | 50,000+ token 的工具定义，上下文被污染 | 最小可行工具集；延迟加载 |
| 2 | **描述模糊** | 参数描述让人困惑 | 用示例值 + 类型约束消除歧义 |
| 3 | **无错误处理** | 工具失败后 Agent 不知道怎么办 | 明确的错误分类 + 重试策略 |
| 4 | **全部工具同时加载** | 无论场景如何，所有工具定义都在上下文中 | 延迟加载 + 按场景分组 |
| 5 | **忽略工具间依赖** | 同时调用有依赖关系的工具 | 一次一个，等待结果 |
| 6 | **无限重试** | Agent 在失败时不断重试直到超时 | 3 次上限 + 改变策略 |
| 7 | **功能性重叠** | 两个工具做同一件事的不同变体 | 合并或用参数区分 |
