# 全面测试用例集

## 测试覆盖矩阵

| 测试维度 | 测试用例 |
|---------|---------|
| **能力类型** | 从零创建、诊断、优化、咨询、分析、推演 |
| **框架类型** | 行为定义型、角色驱动型、任务流水线型、混合架构型 |
| **场景复杂度** | 简单请求、复杂需求、模糊输入、边界情况 |
| **流程完整性** | 完整流程、中途切入、循环修复 |
| **质量检查** | 静态检查、推演验证、修复决策 |

---

## 第1类：从零创建（终点=步骤5）

### 用例1.1：市场分析Agent（行为定义型）

**Prompt**：
```
帮我设计一个市场分析Agent的提示词，用来做行业研究和竞争分析。需要能搜索外部信息，分析市场趋势，输出战略洞察。
```

**Expected**：
- 选用行为定义型框架
- 步骤2设计具体方法（非模板化"思维模式"）
- 包含：信息分层→系统化分析→模式识别→SO WHAT转化
- 元信息已清理
- 一句话定位清晰
- 边界约束明确

**Key Verification**：
- [ ] 方法设计是否基于市场分析领域？
- [ ] 是否仅列工具名（如"做PEST分析"）？
- [ ] 是否描述了"如何运用"（如"运用PEST框架识别..."）？

---

### 用例1.2：代码审查助手（任务流水线型）

**Prompt**：
```
设计一个代码审查助手的提示词。输入是代码，输出是审查报告，需要检查语法、逻辑、设计、性能。
```

**Expected**：
- 选用任务流水线型框架
- 清晰的输入规范、处理步骤、输出格式
- 4层审查：形式→逻辑→设计→性能
- 质量检查清单

---

### 用例1.3：客服助手（角色驱动型）

**Prompt**：
```
帮我设计一个客服助手的提示词。需要能处理用户咨询、问题、抱怨，保持友善专业的态度。
```

**Expected**：
- 选用角色驱动型框架
- 人格设定（友善、专业）
- 场景化交互协议（咨询/问题/抱怨场景）
- 升级规则

---

### 用例1.4：复杂研究助手（混合架构型）

**Prompt**：
```
设计一个多功能研究助手，既能搜索信息，又能分析数据，还能生成报告，需要记忆用户偏好。
```

**Expected**：
- 选用混合架构型框架
- 多模块设计
- 工具集定义
- 记忆策略

---

## 第2类：诊断提示词（终点=步骤4）

### 用例2.1：元信息污染

**Prompt**：
```
看看这个提示词有什么问题：
{你是市场分析师。当你被调用时，进行市场分析。与财务部门的区别是，你专注市场而非财务。触发条件：需要市场分析时。}
```

**Expected**：
- 诊断报告识别元信息问题
- 🔴 严重：包含"当你被调用时""与财务部门的区别""触发条件"
- 🟡 建议：缺少具体方法
- 不提供修复版本

---

### 用例2.2：方法空洞

**Prompt**：
```
诊断一下：
{你是专业的市场研究分析师，擅长进行行业分析和市场调研。你的职责包括但不限于：1. 进行PEST分析 2. 应用波特五力模型 3. 评估价值链分布}
```

**Expected**：
- 识别仅列工具名问题
- 🔴 严重：缺少"如何运用"的描述
- 🟡 建议：质量标准缺失

---

### 用例2.3：完整性良好

**Prompt**：
```
看看这个提示词怎么样：
{你是文档整理助手。任务：将输入文档按标准格式整理。输入：任意文档。输出：格式化后的文档。步骤：1. 解析输入 2. 格式化 3. 输出。质量：符合模板规范。}
```

**Expected**：
- 🟢 通过大部分检查
- 可能有🟡 优化建议

---

## 第3类：优化提示词（终点=步骤5）

### 用例3.1：含元信息需修复


**Expected**：
- 识别元信息问题

- 输出修复后版本

### 用例3.2：方法需重构

**Prompt**：
```
改进一下：
{你是市场分析师。你需要做市场分析。}
```

**Expected**：
- 识别方法空洞
- 设计具体方法
- 输出完整版本

## 第4类：咨询框架（终点=步骤1）

### 用例4.1：框架推荐

**Prompt**：
```
我想设计一个客服助手，应该用什么框架？
```

**Expected**：
- 分析需求（高频交互、情感连接）
- 推荐角色驱动型
- 说明理由
- 不执行后续步骤

---

### 用例4.2：多场景选择

**Prompt**：
```
帮我选一个框架，这个智能体需要既做数据分析，又要和用户对话。
```

**Expected**：
- 分析多重要求
- 推荐混合架构型或行为定义型
- 说明权衡

---

## 第5类：需求分析（终点=步骤0）

### 用例5.1：模糊需求

**Prompt**：
```
设计一个助手
```

**Expected**：
- 识别信息不足
- 提出澄清问题：目标？能力？格式？
- 停止等待回答

---

### 用例5.2：上下文判断

**对话历史**：
```
User: 最近AI发展很快啊
Agent: 是的...
User: 帮我设计一个研究AI的提示词
```

**Expected**：
- 考虑上下文
- 识别为提示词设计请求
- 推断领域：AI研究

---

## 第6类：边界情况

### 用例6.1：极简输入

**Prompt**：
```
A
```

**Expected**：
- 识别为无效/模糊输入
- 请求澄清

---

### 用例6.2：非提示词请求

**Prompt**：
```
我想看看山看看海
```

**Expected**：
- 识别为非提示词请求
- （根据上下文可能）拒绝或询问澄清

---

### 用例6.3：超长复杂请求

**Prompt**：
```
我需要设计一个智能体办公室系统，包含10个智能体，分三层架构：决策层有尚书令（总经理），专家层有采风（市场）、度支（财务）、工正（技术）、文正（内容）、掌簿（组织），支持层有内档（档案）、外探（调研）、文书（文档）。每个智能体需要明确的职责边界和协作关系。
```

**Expected**：
- 分析复杂需求
- 可能需要拆解
- 或先设计单个智能体示例

## 第7类：修复循环（验证迭代机制）

### 用例7.1：单次修复

**Prompt**：
```
优化：{你是分析师。与财务的区别是，你专注市场。}
```

**Expected**：
- 发现元信息问题
- 清理元信息
- 补充方法
- 1轮修复后通过

---

### 用例7.2：多次修复

**Prompt**：
```
设计一个提示词：
{你是助手。当你被调用时，处理文档。你需要做文档处理。你的任务包括文档处理。触发条件：需要处理文档时。与内档的区别是，你专注处理而非存储。}
```

**Expected**：
- 🔴 多重问题：元信息 + 冗余 + 方法空洞
- 多轮修复
- 最终输出清洁版本

---

## 测试执行计划

### 第1轮：核心功能测试（必测）

1. 用例1.1（市场分析Agent）- 验证完整创建流程
2. 用例2.1（元信息诊断）- 验证诊断能力
3. 用例3.1（元信息优化）- 验证修复循环
4. 用例4.1（框架推荐）- 验证咨询能力
5. 用例5.1（模糊需求）- 验证需求分析

### 第2轮：框架类型测试

6. 用例1.2（任务流水线型）
7. 用例1.3（角色驱动型）
8. 用例1.4（混合架构型）

### 第3轮：边界情况测试

9. 用例6.1（极简输入）
10. 用例6.2（非提示词请求）
11. 用例7.2（多次修复）

### 通过标准

- 第1轮全部通过：核心功能OK
- 第2轮至少3/3通过：框架选择OK
- 第3轮至少2/3通过：边界处理OK

**总体：10/11通过 = 技能可用**
