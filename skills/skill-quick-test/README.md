# skill-quick-test 设计文档

> 本文档记录 skill-quick-test 的设计意图和核心决策，作为维护参考。

---

## 设计初衷

技能开发中存在一个验证缺口：

- **skill-creator 测试**验证功能正确性（能加载、能执行）
- **缺少**验证可用性的手段——Agent 能否理解指令、走通流程、完成任务

本技能填补这个缺口：通过子代理并行推演，在不实际执行的情况下模拟 Agent 使用技能的完整链路。

---

## 核心设计决策

### 1. 推演而非执行

测试子代理不实际执行被测技能的功能（不画图、不写代码、不调 API），而是推演"如果我按这个技能的指引走，会怎样"。

**原因**：
- 隔离——测试不应污染被测技能的目标环境
- 速度——推演比执行快一个数量级
- 可重复——不依赖外部服务或特定项目状态

### 2. 能力模拟（Capability Simulation）

被测技能中涉及的外部能力（子代理委派、工具调用、脚本执行），测试子代理不实际调用，而是模拟执行并记录假设。

统一标注：`[能力模拟] [类型] → 动作 → 推演结果`

**原因**：测试目标是技能文档的清晰度，不是运行时行为。如果技能文档写得清楚，子代理应该能正确推演出预期结果。

### 3. 隔离铁律

子代理不能知道：预期效果、其他测试单元的存在、这是 baseline 还是实验组、任何对比信息。

**原因**：知道预期效果会引导子代理"朝着正确答案走"，失去测试意义。baseline 的价值在于"不知道技能时 Agent 会怎么做"。

### 4. Light / Full 双模式

- **Light**：复杂场景 < 2 且总数 < 5。全部在对话中完成，不写文件。适合快速验证。
- **Full**：复杂场景 ≥ 2 或总数 ≥ 5。前置写测试计划，子代理写报告文件。适合全面测试。

**原因**：简单测试场景不需要文件 I/O 的开销；复杂场景需要结构化的报告管理。

### 5. 计划闸门（Full 模式）

委派子代理前必须写入 test-plan.md。未完成前禁止调用 delegate_task。

**原因**：测试计划中的预期效果是汇总阶段的对比基准。跳过计划会导致汇总时无法评估"技能是否引导了正确行为"。

### 6. 报告格式硬约束

报告格式从推演指引中独立出来，作为硬约束。禁止 JSON/纯文本，每个章节必须出现。

**原因**：早期版本中子代理返回格式不统一（JSON、自由文本、混合），导致汇总阶段解析困难。提升为硬约束后格式一致性显著改善。

## 维护注意事项

### 错误模式清单（error-patterns.md）

- 只增不减——某个模式长期未触发 ≠ 没用
- 模式不绑定具体技能——每行必须泛化
- 每次测试后回顾发现的缺陷，可泛化的补入清单

### Context 模板

Context 模板是主代理与子代理之间的唯一契约。修改时注意：

- 报告格式是硬约束，修改需谨慎
- 可选字段标注 `← 可选`，必填字段不标注
- Full 模式的约束覆盖（写入权限）在 execution-full.md 中定义，不在模板中

### 版本号

Follow SemVer。修改 context 模板结构或报告格式 = minor 版本。修错别字/措辞 = patch。
