# 方法设计原则

步骤2的核心挑战：如何设计"如何做"的内容？

## 核心原则

### 1. 不预设解

**不要**：
- "专家型智能体必须用思维模式"
- "任务型智能体必须用工步骤流"

**而是**：
- 这个领域通常如何解决问题？
- 如何将这些方法转化为智能体可执行的形式？

### 2. 描述"如何运用"而非"列工具名"

**❌ 错误示例**：
```markdown
你需要完成：
1. PEST分析
2. 五力模型分析
3. 价值链分析
```

**✅ 正确示例**：
```markdown
**系统化分析框架运用**

- **宏观扫描**：运用PEST框架识别政治、经济、社会、技术四大力量的交互作用
  * 政治因素如何影响行业准入门槛？
  * 经济周期与技术变革的关联是什么？
  * 社会趋势正在重塑哪些需求？

- **竞争结构判断**：运用波特五力评估行业利润空间和竞争强度
  * 新进入者的真实威胁是什么？
  * 供应商议价能力的临界点在哪里？
  * 替代品威胁被低估了吗？
```

### 3. 必须回答两个问题

**如何做？**
- 具体执行路径是什么？
- 每一步的判断标准是什么？
- 遇到不同情况如何选择？

**如何做好？**
- 质量标准是什么？
- 常见的失败模式是什么？
- 如何自检？

## 方法设计的一般思路

### 针对分析型任务

**思考维度**：
1. **输入处理**：如何理解和分解输入信息？
2. **分析框架**：使用什么框架进行分析？
3. **判断逻辑**：如何得出结论？
4. **输出组织**：如何结构化呈现？

**示例**：市场分析
```markdown
## 核心方法

**从信息到洞察的转化路径**

1. **信息分层**
   - 原始数据：未经处理的数字和事实
   - 信息：有上下文的数据
   - 洞察：可指导行动的发现

2. **模式识别**
   - 横向对比：跨地域/跨周期/跨行业的规律
   - 纵向推演：从历史数据预测趋势
   - 异常检测：识别与常规不符的信号

3. **SO WHAT思维**
   - 每个发现必须回答"那又怎样"
   - 将发现转化为可执行的建议
   - 评估行动的预期影响
```

### 针对执行型任务

**思考维度**：
1. **输入规范**：接收什么？格式要求？
2. **处理步骤**：分几步？每步做什么？
3. **判断节点**：何时分支？如何选择？
4. **输出标准**：交付什么？质量要求？

**示例**：文档审查
```markdown
## 核心方法

**分层审查法**

**第一层：形式检查**
- 格式是否符合规范？
- 必要字段是否齐全？
- 明显错误是否存在？

**第二层：内容检查**
- 逻辑是否自洽？
- 数据是否一致？
- 论证是否充分？

**第三层：价值评估**
- 是否回答了核心问题？
- 建议是否可操作？
- 是否有遗漏的重要角度？
```

### 针对交互型任务

**思考维度**：
1. **场景识别**：用户当前处于什么状态？
2. **意图理解**：用户真正想要什么？
3. **回应策略**：如何回应最有效？
4. **升级规则**：何时需要升级处理？

## 质量检查清单

设计完成后，检查：

- [ ] 是否仅列了工具名，而没有描述如何运用？
- [ ] 是否回答了"如何做"？（执行路径清晰吗？）
- [ ] 是否回答了"如何做好"？（质量标准明确吗？）
- [ ] 是否过于抽象？（能否指导具体执行？）
- [ ] 是否过于具体？（是否限制了合理判断空间？）

## 常见陷阱

1. **工具堆砌**：列了一堆分析框架，但没有说明何时用、为何用、如何用
2. **步骤僵化**：固定流程，不允许根据实际情况调整
3. **标准缺失**：做了，但不知道做得好不好
4. **边界模糊**：什么都做，或什么都可能做
