# metapowers — Agent 规则

本仓库是**帮助 Agent 开发 Skill 的 Skill**。所有 Skill 以 [skill-creator](https://github.com/anthropics/skill-creator) 标准为基线，以下仅列出额外约束。

## 🔴 源码 vs 已安装 — 同一技能，两个世界

本仓库 `skills/` 目录下是**开发中的源码**。这些技能也可能以同名形式
**安装在当前运行时环境**中。两者完全独立：

- **源码**：你可以读、改、删的文件 —— 开发对象
- **已安装**：你可以调用但绝不能改的已发布产品

区分不靠工具名或路径约定，靠**用户意图**：

| 用户意图 | 目标 |
|----------|------|
| 修改、编辑、开发、设计、调试技能本身 | 仓库中的 `skills/<name>/` 目录 |
| 使用、调用、加载技能来完成任务 | 运行时环境中的已安装技能 |

不管当前环境用什么工具读取文件、加载技能，这个区分不变。
如果拿不准意图，看用户是要**产出改动**还是**产出结果**。

**如果无法自行准确判断意图，必须向用户确认后再行动。宁可多问一句，不可猜错目标。**


> **如果当前环境中没有 skill-creator 技能，请提示用户安装后再继续。**

## 通用性门槛

本仓库只收通用 Skill。一个 Skill 只有在"做完全不同项目的人也能用上"时才适合放进来。

| 不适合 | 替代方案 |
|--------|----------|
| 依赖特定第三方工具或服务 | 做成独立插件 |
| 只服务某个团队或工作流 | 做成独立插件 |
| 垂直领域（游戏开发、金融分析等） | 做成独立插件 |
| "这样可能更好"但没有实际问题驱动 | 等遇到真实问题再来 |
| 一次提交多个不相关修改 | 拆分成独立提交 |

## 额外 Skill 规范

以下是在 skill-creator 标准基础上的追加要求。

### metadata 字段（必须）

skill-creator 标准要求 `name` 和 `description`，本项目额外要求 `metadata` 中包含：

| 字段 | 示例 |
|------|------|
| `version` | `1.0.0` |
| `last_updated` | `2026-06-17` |
| `author` | `Ĉalio` |

### CHANGELOG.md（必须）

每次迭代必须记录变更，格式遵循 [Keep a Changelog](https://keepachangelog.com/)。

### README.md（可选，给人读的）

与 SKILL.md（给 Agent 读的指令）互补：
- 设计意图：为什么这样设计？
- 关键架构决策和取舍
- 维护注意事项

> **README.md 和 CHANGELOG.md 不是技能标准文件。** 它们是开发过程中的参考文件，仅供人类阅读。技能的标准文件（SKILL.md、references/、scripts/、assets/、templates/）中**不应引用或引导加载** README.md 和 CHANGELOG.md。

### templates/ 目录

用于存放提示词或文档模板。Agent 按需读取，不占主上下文。

### 设计原则

#### 目录树从技能根开始

SKILL.md 中展示技能目录树时，根目录必须是技能自身（如 `my-skill/`），不得包含上层分类目录（如 `skills/my-skill/` 或 `communication/my-skill/`）。上层路径是磁盘位置细节，不属于技能身份的一部分——技能每次移动目录都得改树，纯属维护噪音。

#### 技能无硬性依赖

技能的设计和运行不得依赖其他技能、项目中的特定文件或运行环境中的特定信息。成套技能之间只能互相推荐（在 description 或"与其他技能的关系"中提及），不能设计成硬性依赖（如调用另一个技能的输出、读取另一个技能的内部文件、假设某个项目路径存在）。

| 允许 | 禁止 |
|------|------|
| 在 description 中推荐相关技能 | SKILL.md 中写"必须先运行 X 技能" |
| 在"与其他技能的关系"中说明协作关系 | references/ 引用另一个技能的文件 |
| 独立运行，不假设环境状态 | 假设项目根目录存在特定文件结构 |
| 根据 `available_skills` 列表动态选择 | 在流程中硬编码"调用 skill-creator" |

## 行为约束

- **真实需求驱动**：不解决理论推测的问题。如果用户没有讲清具体场景，先问清楚。
- **展示改动**：提交前必须让用户看过完整 diff 并得到明确同意。
- **不编造**：问题描述、功能说明、测试结果必须真实。

## 开发中 vs 已安装 Skill

本节的规则是上面总纲的具体落地。

### 判断速查

| 用户说 | 意图 | 目标 |
|--------|------|------|
| "改一下 web-deep-research 的提示词" | 修改源码 | `skills/web-deep-research/` |
| "用 web-deep-research 帮我做研究" | 调用技能 | 运行时技能 |
| "给 agent-prompt-design 加个 reference" | 修改源码 | `skills/agent-prompt-design/` |
| "用 agent-prompt-design 检查我的 prompt" | 调用技能 | 运行时技能 |
| "这个技能的行为不太对" | 调试源码 | `skills/<name>/` |
| "帮我新建一个技能来干 X" | 开发 | `skills/<name>/` (新建) |

关键信号（按优先级从高到低）：

1. **看整段对话在做什么** —— 这是最终裁决
2. 动词是「改、加、删、建、写、调试」→ 源码；动词是「用、调用、帮我做」→ 运行时
3. 动词是「加载、读、打开、看看、检查」→ 回到第 1 条：看对话上下文

💡 **拿不准就问。** 当对话上下文不足以判断意图时——
比如用户只说「看一下 web-deep-research」而无更多线索——
不要猜，直接确认：「你是想修改它的代码，还是用它来做事？」

### 规则

1. **绝不安装开发中的技能。** 开发中的技能只在仓库内迭代，不复制到任何运行时技能目录。
2. **同名不同体。** 同一个技能名在仓库和运行时同时存在时，它们是两份完全独立的副本。所有修改针对仓库中的源码，绝不触碰运行时副本。
3. **调用只走运行时。** 使用一个技能完成任务时，加载的是运行时环境中的已安装版本，不是仓库中的源码。
4. **隔离发布。** 本项目的 Skill 只有通过正式发布/安装流程后才成为运行时技能。在此之前，它们只是仓库中的源文件。
