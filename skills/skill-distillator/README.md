# skill-distillator 设计文档

> 本文档记录 skill-distillator 技能的设计意图和核心决策，作为维护参考。

---

## 设计初衷

在技能开发过程中，我们发现一个矛盾：

- **技能文件为 Agent 优化**：SKILL.md 使用指令型语言，references/ 分散存放，便于 Agent 按需读取执行
- **人类阅读成本高**：想了解一个技能时，需要在多个文件间跳转，难以形成整体认知

**skill-distillator 的核心价值**：将技能的"执行文档"转化为"阅读文档"，降低人类理解成本，便于知识沉淀和传承。

---

## 核心思想

**从"告诉 Agent 怎么做"到"向人解释是什么"**

技能文档（SKILL.md）的默认读者是 Agent，使用指令型语言：
- "你应该这样做..."
- "第一步：读取文件"
- "禁止修改原文件"

方法论文档的默认读者是人类，使用说明型语言：
- "这个技能的逻辑是..."
- "技能首先读取文件以了解核心功能"
- "技能仅读取文件，不执行任何修改操作"

这种转换是 skill-distillator 的核心能力。

---

## 关键决策

| 决策 | 理由 |
|------|------|
| 五步执行框架（理解→解构→编排→表达→质量检查） | 系统化且可操作，每个阶段有明确的方法论支撑，质量检查确保输出可靠性 |
| 多文件输出 | 符合渐进式披露原则，便于选择性阅读 |
| references/ 视角转换 | 从"Agent参考"转为"人类参考"，保留完整内容，转换表达方式 |
| 典型示例选择（2-3个） | 展示用法即可，避免淹没核心 |
| Diátaxis 框架应用 | 基于用户意图而非作者思维组织内容 |
| 知识拆分到独立 reference 文件 | 每个方法论独立维护，SKILL.md 只保留流程和引用 |
| 阶段文档（phases/） | 每个阶段独立文件，包含目标、知识加载、操作步骤、质量检查 |
| 质量检查强制阶段 | 连续两轮无修改停止，确保输出质量 |
| 智能转换原则 | 清晰则摘取，指令型则转换，简略则丰富，不强制重写 |
| 技能类型识别（纯指令型/混合型） | 纯指令型授权推断和特殊转换，混合型使用标准流程；简化为两类而非三类，避免"已可读型"的模糊边界 |
| 源一致性扫描与矛盾处理 | 以 SKILL.md 为准，数值型矛盾并列呈现；在理解阶段扫描，避免矛盾传播到输出 |
| 推断边界定义 | 可推断（约束反推/行业常识/流程逻辑）标注 `[推断: 依据]`，不可推断标注 `[未推断: 原因]`，无法确定时不推断 |
| 类型自适应策略 | 纯指令型：推断授权+检查清单特殊转换+决策规则表转换；混合型：标准流程+选择性转换 |
| 技能标准区分 | 在探索阶段区分 skill-creator 标准文件与 metapowers 项目追加文件，确保仅符合 skill-creator 标准的技能也可蒸馏 |
