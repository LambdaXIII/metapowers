# web-deep-research — 设计说明

> 给人读的。Agent 读 SKILL.md。

## 设计目标

这个技能解决一个问题：**"我要把这个话题研究透——怎么系统地逼近真相？"**

它不解决"怎么搜得快""怎么用某个工具""怎么判断一条信息准不准"——那些是工具使用指南和评估方法论的事。它解决的是**研究本身的流程设计**：从明确意图，到收集线索，到交叉比对，到交付一份结论可由读者独立检验的报告。

三个约束定义了它的边界：

1. **不是快速查询工具**——查一句话就能回答的事实用 `web-search-protocol`，不用这个
2. **不是工具选择指南**——`web_search` vs `web_extract` vs `browser_navigate` 的选择是另一个技能的事
3. **不是纯咨询**——"你觉得我该怎么做"如果没有多信源调查维度，就不是研究

## 核心架构决策

### Phase 0-4 流程：收集和评估严格分离

**决策**：Phase 2（纯收集，记录 URL）和 Phase 3（按信息性质评估）之间有一条硬边界。

**理由**：Agent 在收集信息时最容易犯的错是过早下判断——"这条看起来不可靠，跳过"或者"这个说法很有道理，当成事实"。分离收集和评估迫使 Agent 先完整收集再统一评估，避免锚定效应。

**取舍**：这比"边搜索边判断"慢得多，Token 消耗也更大。所以文档开头声明了——"Use when thorough investigation matters more than speed"。

### 领域特化文档：启发提示，不是结构框架

**决策**：`references/creative-work.md` 和 `references/person-biography.md` 提供领域特定的来源推荐和研究维度提示，但不规定报告结构或必查清单。

**理由**：我们对 creative-work.md 和 person-biography.md 各做了一轮"重写→推演→修复"循环，发现了四条通用原则（详见 `references/domain-reference-design-principles.md`）：

1. **来源组织的轴线应是"寻源目的"，不是"来源权威"**——Agent 需要知道"去哪找什么"，可信度判断留给 Phase 3
2. **维度组织轴线 = 该领域人类对研究对象的自然认知顺序**——人物用叙事逻辑（是谁→经历了什么→跟谁相关→别人怎么看他→留下了什么），作品用客观性梯度（事实→半事实→主观）。硬套相反的逻辑会导致"把人当物体分析"
3. **越详细，越需要收束机制**——每个特化文档必须自带至少一条显式收束规则，否则详细度会逼迫 Agent 全面铺开
4. **加载时机是文档价值的一半**——入口文件不做硬绑定，用两级安全网（SKILL.md 预判 + workflow 复查），且只指回 Content Index 表格而不枚举文件名

### 来源类别命名：按信息生产者，不按信息用途

**决策**：来源分类用"谁说的"界定边界——"学界与评论界""权威来源""公众舆论""一手与见证资料"。

**理由**：用户读到"专业评估"时问了一句"这是啥意思"——这个名字暗示了功能（评估），但 Agent 无法直观判断它和"权威来源"的区别。功能分类需要 Agent 自己理解"这个类别是干嘛的"，而生产者分类（"谁说的"）是客观属性，无需理解。

### 人物特化：身份锁定是硬门控

**决策**：person-biography.md 要求在 Phase 1 结束时必须完成目标人物身份的锁定，穷尽搜索/推理/询问三种手段仍无法锁定时中止并确认。

**理由**：公众人物的姓名冲突率极高——即使用了全名+身份限定词，搜到的资料仍可能属于另一个同名人物。如果在 Phase 2 中途才发现错误，前面的收集工作全部浪费。硬门控把验证成本集中在 Phase 1，保护后面三个阶段不被污染。

```
    └── report-template.md    ← 研究报告模板
```

## 维护指南

### 新增领域特化文档

1. 读一遍 `references/domain-reference-design-principles.md` 的自检清单
2. 用"跟朋友介绍这个东西时自然用什么顺序"来确定维度组织轴线
3. 来源按"想找什么→去哪儿"组织，类别名标定生产者而非用途
4. 主观维度必须配收束规则 + "什么算最显著"的具体示例
5. 如果该领域存在"找错对象"风险（人物/公司/地名等），配 Phase 1 身份锁定硬门控
6. 更新 SKILL.md 的 Content Index 表格（一行），不改 Instructions

### 修改已有特化文档

1. 改完后跑一轮推演：模拟 Agent 首次加载这份文档，按"清空→逐帧跟→边界刨→打分"走一遍
2. 推演发现的修复同样适用于另一份特化文档时，两边一起改——保持结构对齐
3. 更新 CHANGELOG.md
