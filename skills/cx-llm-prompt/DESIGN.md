# 新技能设计规划：cx-prompt-workflow

## 1. 核心定位

**一句话定义**：基于优先级流程的提示词工程工作流技能

**解决的问题**：
- 现有技能像"知识库"，需要用户自己选择和组合
- 用户不知道"现在该做什么"，陷入决策困境
- 诊断、设计、优化等场景没有统一的处理框架

**核心创新**：
- 将提示词工程抽象为**6个优先级步骤**
- 用户可以在任何步骤切入，但**必须按优先级顺序执行**
- 每个步骤有明确的输入、输出和依赖关系
- 自动判断用户场景，从合适的步骤开始

## 2. 流程设计

### 2.1 六步流程（按优先级排序）

```
步骤0: 需求分析（基础层）
├── 输入：用户请求 + 对话上下文
├── 输出：目标 + 用途 + 约束 + 关键决策点 + 终点判断
├── 功能：理解意图（自然判断，非关键词匹配）
├── 核心原则：不明确预设场景，根据实际需求判断
├── 独立性：✓ 可单独执行
└── 必要性：所有后续步骤的依赖

步骤1: 甄选参考框架
├── 输入：需求分析结果
├── 输出：选定框架 + 选择理由
├── 功能：提供结构骨架，不决定内容
├── 独立性：✓ 可单独执行
└── 依赖：步骤0

步骤2: 设计"如何做"（核心设计层）★关键修正
├── 输入：框架 + 需求 + 领域信息
├── 输出：内容大纲（含"如何做"的具体设计）
├── 核心原则：
│   ├── 不预设解（如"思维模式""工作流"模板）
│   ├── 根据领域和任务设计具体方法
│   ├── 必须回答：1) 如何做 2) 如何做好
│   └── 自动应用技巧：元信息清理、一句话定位、边界约束
├── 方法来源：
│   ├── 该领域的通用方法论（需外部知识）
│   └── 转化为智能体可执行的形式
├── 独立性：✗ 必须依赖步骤1
└── 检查：确保描述了具体方法而非仅列工具名

步骤3: 编写提示词
├── 输入：内容大纲
├── 输出：完整提示词文本
├── 独立性：✗ 必须依赖步骤2
└── 可提前结束：生成后不再检查

步骤4: 综合诊断（诊断层）
├── 输入：提示词文本
├── 输出：统一问题清单（含静态+动态问题）
├── 核心检查：是否回答了"如何做"和"如何做好"
├── 模式：
│   ├── 完整模式：静态检查 + 推演验证（默认）
│   └── 仅推演模式：仅推演验证（用户明确要求时）
├── 内容：
│   ├── 静态：七维检查（元信息、完整性、一致性、具体性、安全性、效率、模型适配）
│   └── 动态：推演验证（执行路径、边界情况、潜在问题）
└── 独立性：✓ 可单独执行

步骤5: 修复决策与输出（修复层）
├── 输入：综合诊断报告 + 当前版本
├── 输出：
│   ├── 如终点=步骤4：诊断报告
│   ├── 如终点=步骤5且无问题：最终提示词
│   └── 如终点=步骤5且有问题：修复后提示词
├── 功能：
│   ├── 问题归因（根源判断）
│   ├── 决策跳转（确定修复步骤）
│   ├── 执行修复
│   └── 迭代回步骤4（最多3次）
└── 触发：步骤4完成后自动触发（如终点为步骤5）
```

### 2.2 用户切入与终点判断

**步骤0的自然判断**：不依赖关键词，根据上下文和实际需求判断

| 用户请求示例 | 步骤0判断 | 终点 | 执行范围 | 输出 |
|------------|----------|------|---------|------|
| "帮我设计一个市场研究Agent" | 创建意图 | 步骤5 | 0→1→2→3→4→5 | 最终提示词 |
| "看看这个提示词有什么问题" | 诊断意图（仅诊断） | 步骤4 | 0→4 | 综合诊断报告 |
| "优化/修复一下这个提示词" | 优化意图 | 步骤5 | 0→4→5→（如问题→修复→4→5） | 修复后提示词 |
| "用什么框架比较好" | 咨询意图 | 步骤1 | 0→1 | 框架推荐 |
| "推演一下这个提示词" | 推演意图 | 步骤4（仅推演模式） | 0→4 | 推演验证报告 |

**终点决定输出**：
- 终点=步骤1：输出框架推荐
- 终点=步骤4：输出综合诊断报告
- 终点=步骤5：输出最终提示词（或修复后版本）

### 2.3 步骤依赖与跳转规则

**正常流程**：0 → 1 → 2 → 3 → 4 → 5

**诊断修复流程**：
```
用户切入步骤4（诊断）
    ↓
发现问题（严重程度分级）
    ↓
🔴 严重问题（如含元信息、关键冲突）
    ↓
跳转到步骤2（重新细化）
    ↓
步骤3（重新编写）
    ↓
步骤4（再次检查）
    ↓
步骤5（推演验证）
```

**跳过规则**：
- 用户明确说"只分析需求" → 执行步骤0后停止
- 用户明确说"只推荐框架" → 执行步骤0→1后停止
- 步骤3（编写）完成后，用户说"不用检查" → 直接结束

## 3. 能力覆盖与差异

### 3.1 与现有技能（cx-prompt-engineering）能力对比

| 现有技能能力 | 对应新技能步骤 | 改进说明 |
|-------------|---------------|---------|
| 需求分析 | 步骤0 | 增加"自然判断"，不依赖关键词匹配 |
| 策略设计（框架选择） | 步骤1 | 直接推荐而非提供选项，增加理由说明 |
| 内容编写 | 步骤2+3 | 步骤2专门设计"如何做"，不预设解 |
| 诊断审查 | 步骤4 | 整合静态+动态诊断，增加元信息维度 |
| 迭代优化 | 步骤5 | 自动修复+迭代，最多3次循环 |

### 3.2 核心差异

| 维度 | cx-prompt-engineering | cx-prompt-craft（新） |
|------|----------------------|---------------------|
| **组织方式** | 按能力类型 | 按流程步骤（0-5） |
| **用户交互** | 询问用户想用什么能力 | 场景驱动，自动识别 |
| **框架选择** | 提供选项 | 直接推荐+理由 |
| **内容设计** | 列举技巧供参考 | 步骤2专门设计"如何做" |
| **诊断检查** | 六维检查 | 七维检查（+元信息） |
| **执行顺序** | 依赖补齐 | 优先级流程 |
| **"如何做"** | 技巧模板（思维模式/工作流等） | **不预设解，根据领域设计** |

**关键跃迁**：
- 从"提供模板"到"教人找到解"
- 从"用户选择"到"自动决策"
- 从"技巧列举"到"方法设计"

## 4. 核心设计原则与边界

### 4.0 技能范畴（重要）

**本技能只处理：提示词的设计/诊断/优化**

| 场景 | 示例 | 是否属于本技能 |
|------|------|---------------|
| ✅ 设计提示词 | "帮我设计一个市场研究Agent的提示词" | 是 |
| ✅ 诊断提示词 | "看看这个提示词有什么问题" | 是 |
| ✅ 优化提示词 | "优化一下这个提示词" | 是 |
| ❌ 非提示词请求 | "我想看看山看看海" | **不是** |
| ❌ 业务任务 | "帮我搜索山海图片" | **不是** |

**关键区分**：
- 用户说的是"提示词相关"→本技能处理
- 用户说的是"业务内容"→不属于本技能，由上层Agent判断

### 4.1 优先级原则

### 4.1 优先级原则
- 必须按步骤0→1→2→3→4→5顺序执行
- 不能从步骤2跳到步骤4（必须先有框架才能细化）
- 可以跳过（如执行0→1后停止）

### 4.2 自给自足原则
- 每个步骤必须明确声明输入、输出、依赖
- 独立执行时，依赖必须已满足（用户提供或之前步骤产出）
- 依赖不满足时，自动执行前置步骤

### 4.3 场景识别原则
- 不询问用户"你想用哪个能力"
- 根据用户请求内容自动判断场景
- 从合适的步骤切入

### 4.4 可解释原则
- 每个决策（选什么框架、用什么技巧）必须说明理由
- 让用户理解"为什么这样"，但不把选择负担交给他们

## 5. 步骤详细设计

### 步骤0：需求分析

**输入**：用户请求文本

**处理逻辑**：
1. 识别任务类型（创建/诊断/优化/咨询/分析/推演）
2. 提取关键信息：
   - 目标：要做什么？
   - 用途：一次性使用还是持久化？
   - 约束：格式、平台、特殊要求
3. 确定关键决策点：
   - 是否需要自主决策？
   - 是否涉及多轮交互？
   - 任务边界是否清晰？

**输出**：
```markdown
## 需求分析结果

### 任务类型
[创建/诊断/优化/咨询/分析/推演]

### 核心信息
- **目标**：...
- **用途**：一次性任务 / 持久化智能体
- **约束**：...

### 关键决策点
- [决策点1]：[判断结果]（如"需要自主决策：是"）
- [决策点2]：[判断结果]

### 建议切入点
基于以上分析，建议从**步骤X**开始执行。
```

### 步骤1：甄选参考框架

**决策逻辑**：

| 关键特征 | 推荐框架 | 理由 |
|---------|---------|------|
| 需要自主决策 + 使用工具 | 行为定义型 | 强调决策逻辑和边界 |
| 高频交互 + 情感连接 | 角色驱动型 | 强调人格和交互风格 |
| 明确输入 + 固定步骤 | 任务流水线型 | 强调流程和输出规范 |
| 以上多种组合 | 混合架构型 | 全面覆盖 |

**输出**：
```markdown
## 框架推荐

### 推荐框架：[框架名称]

**选择理由**：
基于需求分析中的关键决策点：
1. [特征1] → 因此需要[框架特性1]
2. [特征2] → 因此需要[框架特性2]

[框架名称]最能满足这些需求，因为...

### 其他可行选项（可选说明）
- [其他框架]：也适用，但[差异点]
```

### 步骤2：根据框架细化指令

**技巧应用清单**（根据需求自动选择）：

| 技巧 | 适用条件 | 自动应用 |
|------|---------|---------|
| 元信息清理 | 所有场景 | ✓ |
| 一句话定位 | 步骤0判断为"持久化智能体" | ✓ |
| 边界约束设计 | 步骤0判断为"需要自主决策" | ✓ |
| 思维模式描述 | 步骤0判断为"专家型分析任务" | ✓ |
| 任务清单描述 | 步骤0判断为"明确边界任务" | ✓ |

**输出**：结构化大纲

### 步骤3：编写提示词

直接输出完整文本。

### 步骤4：综合诊断（诊断层）

**输入**：提示词文本 + 模式指示（完整/仅推演）

**诊断内容**：

| 类型 | 检查项 | 输出 |
|------|--------|------|
| **静态检查（七维）** | 元信息、完整性、一致性、具体性、安全性、效率、模型适配 | 各维度结果 + 问题描述 |
| **动态推演** | 典型输入执行路径、边界情况测试、潜在问题识别 | 场景模拟 + 风险点 |

**问题分级与归因**：

| 严重度 | 定义 | 归因方式 |
|--------|------|---------|
| 🔴 严重 | 会导致执行失败或行为异常 | 必须定位根源步骤 |
| 🟡 建议 | 影响质量或清晰度 | 建议定位根源 |
| 🟢 优化 | 可以更好但当前可用 | 可选归因 |

**根源判断**（针对🔴严重问题）：
```
问题 → 分析根源 → 跳转步骤
├─ 需求理解偏差 → 步骤0
├─ 框架不匹配 → 步骤1  
├─ 结构设计问题 → 步骤2
└─ 具体编写问题 → 步骤3
```

**输出**：
```markdown
## 综合诊断报告

### 静态检查结果
| 维度 | 结果 | 问题描述 |
|------|------|---------|
| 元信息 | ✅/🔴/🟡 | ... |
| 完整性 | ✅/🔴/🟡 | ... |
| ... | ... | ... |

### 推演验证结果
**场景1（典型）**：执行路径... [正常/⚠️问题]
**场景2（边界）**：... [正常/⚠️问题]

### 问题汇总（分级+归因）
🔴 严重：[N] 个
1. [问题描述] - 根源：[步骤X] - 影响：...

🟡 建议：[N] 个
...
```

### 步骤5：修复决策与输出（修复层）

**输入**：综合诊断报告 + 当前提示词版本 + 终点指示

**决策逻辑**：

```
IF 终点 = 步骤4（仅诊断）：
    → 输出：综合诊断报告
    → 结束

IF 终点 = 步骤5（创建/优化）：
    → IF 无🔴严重问题：
        → 输出：最终提示词（简洁版，可选附诊断摘要）
    → IF 有🔴严重问题：
        → 确定跳转步骤（根据根源）
        → 执行修复
        → 回到步骤4重新诊断
        → （最多3次迭代）
        → 输出：修复后提示词
```

**修复执行**：

| 根源 | 跳转步骤 | 修复动作 |
|------|---------|---------|
| 步骤0（需求）| 如已分析过→步骤2；如未→报错 | 基于已有理解重新设计 |
| 步骤1（框架）| 步骤1 | 重新选择框架 |
| 步骤2（细化）| 步骤2 | 重新细化内容 |
| 步骤3（编写）| 步骤3 | 重写相关部分 |
| 无法判断 | 步骤2 | 保守策略：重新设计 |

**输出格式**（终点=步骤5时）：

```markdown
## 最终提示词

```yaml
---
[完整提示词内容]
---
```

---

## 设计说明（可选展开）
- **选用框架**：[框架]（原因：[一句话]）
- **关键技巧**：[一句话定位/元信息清理/...]
- **诊断摘要**：[静态N项+推演N场景，关键问题简述]
```

**迭代控制**：
- 修复后必须回到步骤4重新诊断（形成闭环）
- 最多3次修复循环
- 如3次后仍有严重问题，输出当前最佳版本+风险提示

## 6. 与智元大师经验的整合

智元大师的发现将整合到步骤2（细化指令）和步骤4（要点检查）中：

| 智元大师发现 | 整合位置 | 整合方式 |
|-------------|---------|---------|
| 元信息分离原则 | 步骤2、步骤4 | 步骤2自动清理，步骤4检查 |
| 一句话定位 | 步骤2 | 持久化智能体自动应用 |
| 思维模式vs任务清单 | 步骤2 | 根据任务类型自动选择描述方式 |
| 边界约束重要性 | 步骤2 | 自动添加"你不做什么"章节 |
| 七维检查 | 步骤4 | 扩展六维为七维（+元信息） |

**不作为独立章节，而是内化为流程步骤的自动行为。**

## 7. 已确认设计

### ✅ 流程结构（6步最终版）

| 步骤 | 名称 | 类型 | 核心功能 |
|------|------|------|---------|
| 0 | 需求分析 | 基础层 | 自然判断意图 + 决定终点 |
| 1 | 甄选参考框架 | 设计层 | 推荐框架 + 说明理由 |
| 2 | 根据框架细化指令 | 设计层 | 应用技巧（元信息清理/一句话定位等） |
| 3 | 编写提示词 | 设计层 | 生成完整文本 |
| 4 | 综合诊断 | 诊断层 | 静态检查（七维）+ 推演验证 |
| 5 | 修复决策与输出 | 修复层 | 问题归因 + 修复 + 迭代 + 最终输出 |

### ✅ 核心机制

**终点决定输出**：
- 终点=步骤1：输出框架推荐
- 终点=步骤4：输出综合诊断报告（不修复）
- 终点=步骤5：输出最终提示词（或修复后版本）

**诊断与修复分离**：
- 步骤4：只做诊断，输出统一问题清单
- 步骤5：根据终点决策是否修复

**问题处理**：
- 不预设分类，按根源判断（步骤0/1/2/3）
- 最多3次修复迭代
- 技能不直接询问用户

**步骤0的自然判断**：
- 不依赖关键词匹配
- 根据上下文和实际需求灵活判断

### ✅ 技能范畴

只处理提示词相关请求，不处理纯业务请求。
