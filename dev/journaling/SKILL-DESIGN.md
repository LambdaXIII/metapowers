# journaling 设计文档

> 本文件是 journaling 技能的**设计锚定文档**——记录设计方向、原则理由和关键决策。它独立于技能文件体系，不被运行时加载，仅作为开发者维护和演进技能的参考。

---

## 设计初衷

Agent 的长期记忆需要一个结构化的笔记本系统——不是简单的文件堆，而是一个有 INDEX.md、有条目规范、有维护周期的知识管理体系。

这个技能解决的问题：

- **"写了不读"**：经验记下来了，但下次遇到相关任务时不会先查阅
- **"写了找不到"**：信息存在但无法在需要时快速定位
- **"写了变噪音"**：INDEX.md 膨胀，信号被淹没在日志中
- **"写了不维护"**：条目过期、标签蔓延、结构腐化，但无人清理

Journal 服务于 Agent 自身——用户不审批 journal 操作。Agent 自主维护（归档、合并、删除、标签修复），可以征求建议但不请求许可。这不是协作偏好，是设计要求：用户无法判断 journal 操作的正确性，因此技能必须让 Agent 自给自足。

---

## 核心设计决策

### 1. 渐进式披露——SKILL.md 作为路由

SKILL.md 只保留路由表 + 核心约束 + 操作规则。所有机制细节在 references/ 中按需加载。

**原因**：上下文成本。加载技能写一条笔记和加载技能做全量维护的上下文需求完全不同。路由表让两种场景的成本分离——写笔记只需加载 protocal-write.md 和 spec-note.md，做维护才加载 protocal-maintenance.md。

### 2. 五阶段维护协议——扫描、审查设计定规计划、执行（规则先行）、双向质量检验、收尾

Phase 0 纯扫描（只收集数据不做判断），Phase 1 审查→设计→定规→计划（带着个性化规则文件理解现状、设计目标组织方式、确认规则变更、制定重组计划——不碰文件），Phase 2 执行（先改三规则文件再改条目），Phase 3 双向质量检验（回顾重构微调规则 → 以最终三规则验证条目），Phase 4 收尾记录。

**关键结构**：

- 扫描与决策边界 — Phase 0 输出数据，Phase 1 回答"是否改变规范"
- 设计目标在审查规则之前 — 从"想要什么结构"出发评估规则
- CLASSIFICATION/TAGS/CONVENTIONS 统称个性化规则文件。CONVENTIONS 可选；存在时优先级高于其他两份——特例覆盖默认规则
- 规则修改先于条目重组 — 执行阶段第一步更新三规则，确认制度就位后再操作条目
- 双向质量检查 — 正向：条目是否符合规则；逆向：规则是否反映条目实际状态

**原因**：旧协议将设计决策隐含在规则审查中，规则变更与条目移动同时进行。新结构显式分离"设计→定规则→执行"三步。

### 3. 摘要锚定——scope boundary 而非内容总结

每条笔记的 summary 是其 scope 的边界定义，不是内容的概括。

**三个检查点**：创建时写摘要定义范围→修改前读摘要判断"新内容还在范围内吗"→修改后验证摘要是否仍准确。这让"update vs create"的决策从"我需要想清楚"变成"读摘要→答案自明"。

### 4. Tags 自管理——TAGS.md + 设计方法论文档

Tags 由每个 journal 通过本地的 `TAGS.md` 自管理，技能只提供种子（6 个基础 tag）和设计方法论（`references/design-tags.md`）。

**关键转变**：上一版设计使用技能强制的 4 分类 21 tag 体系（activity/domain/meta/project）。新设计仅初始化 6 个种子 tag（lesson、research、knowledge、documentation、journal、plan），后续 tag 增长遵循命名规则和注册流程。维度分组（activity/domain 等）是可选的，仅在 tag 超过 ~15 个时按需引入。

**先注册后使用**：虽然种子集很小，但新增 tag 必须在 TAGS.md 中注册后才能使用。自由格式标签仍然会退化为噪音，只是约束从"技能强制分类"变为"技能提供方法论 + journal 自执行"。

**规范验证**：维护时 Phase 0 收集 tag 使用数据，Phase 1 审查 TAGS.md 及其规则是否仍然适应当前内容。Tag Worthiness Criteria 决定一个 tag 是否值得保留。

### 5. 读写非对称——读不加载，写才加载

读取 journal（浏览 INDEX.md、查阅条目、搜索内容）**不需要**加载 journaling 技能。只有在写入或维护时才需要加载。

**原因**：journal 是 Agent 每轮启动都要读取的——INDEX.md 是启动协议的第一步。如果每次读 journal 都要加载多个 reference 文件，上下文成本不可接受。读写分离让"读取"保持零成本（INDEX.md 本身就是自描述的状态封面），把技能的复杂性全部推到"写入"侧——写入时才需要规范（格式、目录、标签、摘要锚定），维护时才需要协议（五阶段流程）。

**设计自洽**：INDEX.md 的协议声明行明确标注"读此文件不需要加载 skill · 写入或维护时必须加载"。这和渐进式披露形成配合——读取是零成本高频操作，写入是中等成本低频操作，维护是高成本极低频操作，三者的上下文成本逐层递增。

### 6. Journal 服务于 Agent 而非用户

用户不审批 journal 操作。Agent 自主维护（归档、合并、删除、标签修复），可以征求建议但不请求许可。

**设计约束**：这不是协作偏好，是设计要求——用户无法判断 journal 操作的正确性，因此技能必须让 Agent 自给自足。

### 7. 初始化协议——三阶段明确分工

初始化协议（`references/protocal-init.md`）将原七步流程重构为三阶段：

- **Phase 1（确定位置）**：检查上下文中是否已知 journal-root，未知则询问用户。位置是锚点——不确定则后续无依托。
- **Phase 2（初始化内容）**：逐项检查 CLASSIFICATION.md、TAGS.md、INDEX.md。存在则判断类型、检查格式——合规不动、不合规则保留内容重写格式。同名非同类则改名移入 inbox 后创建。
- **Phase 3（设计发现合约）**：向用户说明重要性、原则和方案，由用户决定是否建立。禁止自行执行。

**Phase 3 演化——协议流与参考分离**：v4.4.0 将 Phase 3 进一步拆分为两层——协议文件保留"方案讨论"的上下文概览（载体类型、核心任务），将 Step 1-4 的详细发现流程独立为 `references/design-discovery-contract.md`。

**原因**：发现合约设计涉及的具体操作步骤较多，挤在协议文件中会削弱"为什么需要"和"有效条件"等背景信息的可读性。分离后协议文件保持远征型协议的节奏感——读完条件后知道方向，需要实操指导时按需加载参考文件。这符合渐进式披露的设计原则。

协议不再内嵌初始模板——模板独立于 `templates/seed/` 目录，协议引用但不包含。`seed/` 子目录强调版本身份，但文件名不加前缀——复制使用时无需改名。
种子目录不再在初始化时创建——由写入操作按需催生，CLASSIFICATION.md 中已定义用途。

### 8. Protocol 操作模式分化——远征型 vs 内嵌型

Protocol 在设计上天然分成两类，核心区别在操作频率和认知上下文。

**远征型（Expedition）**：任务独立、步骤多、边界清晰、cognition budget 充足。每一步需要 why（agent 有空理解原理并处理边缘情况）；强边界约束（"不执行"、"先确认再动手"）；类型识别先行。代表：protocal-init.md、protocal-maintenance.md。

**内嵌型（Embedded）**：嵌入主线工作，高频率、低延迟要求。仅开头提供定位说明（"这个协议是干嘛的"），不提供逐步骤 why；不打断执行节奏；出口质量由维护协议兜底。代表：protocal-write.md、protocal-import.md。

**原因**：两种模式的核心约束不同。用远征型的设计范式套内嵌型不是不够好——是方向错了。设计 protocol 时必须先判断操作场景再落笔。

### 9. 目标前置——每个 protocol 开头应有完成状态描述

每个 protocol 开头应说明"做完这个想达到什么状态"。作用：提供操作终点（知道什么时候算完成）、提供判断依据（以是否朝向目标做取舍）、减少迷失感（序列执行过半时可能忘记为什么做这些步骤）。

**原因**：之前的 write 和 init 协议缺少目标前置，agent 读到中间步骤时无法判断"我还在正确的方向吗"。远征型协议展开为完整节，内嵌型仅保留 2-3 行定位说明。

### 10. 理解验证 ≠ 质量检查

| | 理解验证 | 质量检查 |
|---|---------|---------|
| 时机 | 决策前 | 交付前 |
| 问题 | "我读对了吗" | "我做得对吗" |
| 内容 | 确认数据解读无误 | 确认格式合规、链接完整 |
| 代表 | Phase 1 入口重读 Phase 0 快照 | Self-Check 四条、Phase 3 quality check |

**原因**：只有交付前检查没有决策前验证——如果决策时数据解读有误，质量检查依然会通过（输出格式正确，但方向错了）。

### 11. 加法胜于减法——可逆操作设计模式

任何涉及重组/移动的操作，设计模式应为"写新位置→验证→清理旧位置"的三段加法，而非追踪回退路径。

**原因**：Agent 的执行记录不可靠，依赖 agent 重建状态的任何机制都脆弱。加法模式在中断时天然安全（旧位置还在），终止状态可读（检查文件系统即知进度）。

**具体应用**：maintenance Phase 2 的"新位置先就位，原位置后清理"原则；cascade rename 的"先枚举全部匹配，再从最深到最浅逐一更新"。

### 12. 模板分离的时机

内嵌代码块是否拆到 templates/ 的判断标准：

- **拆**：多 protocol 引用同一内容 | 多种执行方式（cp / write / 手写）
- **留**：单协议引用、单一操作方式的示例或参考

**原因**：protocal-write.md 的 YAML 代码块是单协议引用的示例，留在原地。CLASSIFICATION/TAGS/INDEX 种子模板被 protocal-init.md 引用、可能在不同环境中通过 cp 或 write 执行——拆到了 templates/seed/。

### 13. CONVENTIONS 机制——可选设计模式实例注册表

Journal 中新增 CONVENTIONS.md 文件，作为可选设计模式（如 dashboard、将来的其他模式）的实例注册表。由 agent 在维护时创建/审查/更新/销毁。

核心约束：
- **读取时不感知**：启动协议、日常条目读取不加载此文件——不增加 session 启动负担
- **写入/导入时与三规则一起前置加载**：v4.5.1 调整为统一前置加载，技能规范先确立基线，convention 在后修正
- **仅在维护时修改**：创建、更新、销毁 convention 条目只发生在维护协议阶段

规范原则：
- **不重复**：convention 不应复述技能已有规范
- **允许偏离**：convention 可以与默认规则不同——偏离时需在条目中明确说明理由和预期收益，确保维护时能判断这个偏离是否仍然值得
- 参考 spec-note 格式风格

优先级：CONVENTIONS 是可选文件。但当它存在时，其内容优先级高于 CLASSIFICATION 和 TAGS——特例覆盖默认规则。
同属于"个性化规则文件"（Decision #14），统称统一。

### 14. 个性化规则文件——CLASSIFICATION/TAGS/CONVENTIONS，统称统一，职责不同

CLASSIFICATION.md（目录分类规则）、TAGS.md（标签注册表）、CONVENTIONS.md（设计模式实例注册）统称为"个性化规则文件"。
职责分层：
- CLASSIFICATION 和 TAGS 是基础制度，journal 初始化时创建，定义日常操作的基线
- CONVENTIONS 是可选文件，记录从实际使用中生长的设计模式实例。存在时，内容优先级高于其他两份——特意写上的偏离覆盖默认规则

统一统称的原因：此前三文件没有统称，每次提及只能逐一列名，因此用"三规则平等并列"作为书写纪律防止遗漏。统称确定后，纪律不再需要。


### 15. INDEX.md 协议声明使用标题节而非引用块

INDEX.md 顶部的协议声明区从 `>` 引用块改为 `## 协议声明` 标题 + 列表格式。

原因：`>` 在 Markdown 中语义为引述/引用，适合标注"这是从别处引用的内容"——但协议声明是文件的头部规则，不是引用。`##` 标题具有结构权威性——它是文件的正文章节，不是注脚。

附加效果：标题使 INDEX.md 的协议声明区成为文件扫描时的第一个可见结构块，agent 阅读注意力天然高于缩进形式的内联引用。

**格式兼容**：旧版 INDEX.md 使用 `> ⚠️` 格式。Type Identification 同时识别新旧两种格式（参见 protocal-init.md），初始化时检测到旧版格式 → 自动按新格式重写。维护协议 Phase 4 Step 1 的全部重写也会批量迁移旧格式。

### 16. 维护协议：设计目标在规则审查之前

维护时先设计"目标组织方式"（目录结构、模式选择、标签体系），再审查"现有规则是否支撑这个目标"。

**原因**：旧协议的问题驱动模式（发现分类不一致→修分类，发现 tag 蔓延→修 tag）导致修改是局部的、被动的。先确立全局目标再评估规则，确保每次修改都朝向一致方向。

### 17. 双向质量检查

Phase 3 分两步：Step 1 回顾重构过程反思规则是否合理（逆向——规则是否反映条目状态），Step 2 以确认后的规则验证条目合规（正向——条目是否遵循规则）。

**原因**：单向检查只验证"条目是否遵守规则"，但规则本身可能在重构执行中暴露出不合理。双向闭环让规则和条目互相校正。


### 18. 跨版本防格式漂移——规则文件格式审查 + INDEX.md 全量重写

两条约束确保技能更新后 journal 文件不沿用旧版本格式：

**格式审查**（P1-S1）：对每条存在的规则文件，加载对应的 skill 规范文件，对照格式要求逐项检查。即使技能规范中格式约束不多，也必须重新阅读确认，避免跨版本格式漂移。发现格式不符 → 标记为 P1-S3 变更清单。

**INDEX.md 全量重写**（Phase 4 Step 1）：加载 `references/spec-index.md`，按当前技能规范完全重写 INDEX.md。保留已更新内容（修剪后的最近变更、折叠后的专项工作、去重后的经验摘要、已修复的链接），但格式和结构严格遵循 spec。不增量修补。

**原因**：增量修补累积格式偏差。技能规范在跨版本中可能调整字段顺序、新增必填项、修改协议声明格式。增量修改总是漏掉"旧版本才有的字段顺序"或"旧版本的协议声明行格式"。全量重写是唯一可靠的跨版本一致手段。规则文件格式审查同理——内容审查不覆盖格式合规性。


### 19. 删除即归档——delete 不是合法操作

**设计意图**：删除是不可逆操作，与 journal "可逆维护" 的设计原则冲突。archive/ 目录提供了等价出口——保留历史信息同时移出活跃目录。在 daily operations 中，Agent 看到任何"应该删除"的信号时，执行 `mv to archive/` 而非 `rm`。

**关键约束**：
- 日常操作中不允许硬删除任何 journal 文件。"删除"的语义等于"移入 archive/"
- archive/ 清理仅限维护协议中进行——日常操作不触发
- 新移入 archive 的内容必须经过至少一个维护周期的冷却才能被判断为可硬删除
- 维护协议中 archive 清理的判定标准：恢复（仍有价值）、保留（冷却期或历史参考）、可硬删除（非本轮新移入、确认无用、无交叉引用）

**与现有设计的关系**：
- 是决策 #11（加法胜于减法）在删除场景的直接应用——archive 是加法操作（新位置写入），不是减法操作
- 是决策 #6（Journal 服务 Agent 而非用户）的自然约束——Agent 自主操作范围限定在可逆操作内，不可逆操作需要协议保障或用户确认
- Operating Principles 中"不请求许可"原则因此收紧为"仅可逆操作"——archive 不需要审批（仍是可逆的），hard deletion 需要维护协议条件或用户确认


### 20. description 作为操作范围约束，非场景枚举

**设计意图**：description 的作用不是枚举「用户说什么话时加载我」（场景白名单必定腐烂——总会有新的说法遗漏），而是声明操作边界——「什么类型的操作需要本技能」。操作边界「任何写入性操作」比场景白名单「写条目 / 维护 / 捕获决策」稳健：不管用户说「归档」还是「把那个移走」，只要操作落到 journal-root 写入范围内，约束生效。

**与 SDO 的关系**：本质上就是 SDO 原则「description 写做什么不写触发词」——「做什么」是操作边界，「触发词」是场景枚举。

**与现有设计的关系**：
- 是「Journal 服务 Agent」的自然延伸——操作边界由 Agent 的工作方式（调用 write/edit/mv/bash 工具）定义，不是由用户的措辞定义
- 模式 #6（扩展-约束配对）的应用：扩展触发面（任何写入），用负边界限制（仅读取不需要）

### 21. 信息依赖优于规则闸门

**设计意图**：文本加密系统中，所有文本机制靠 Agent 自觉执行。「将必须读 INDEX.md 写为规则」时 Agent 可选择跳过——跳过规则没有即时后果。但「将 INDEX.md 中有什么写为事实陈述」时，跳过意味着承认自己在信息缺失中操作——这需要更高的认知成本来合理化。信息缺口（「你需要 X 才能做 Y」）比规则约束（「必须先做 X」）更有效，因为前者作用于当下的任务需要，后者作用于过去的失败记忆。

**三层信息依赖结构**：
1. **description**（加载决策点）：操作边界约束——「什么操作需要我」
2. **SKILL.md 正文**（技能加载后）：信息依赖声明——「我需要 INDEX.md 中的什么」
3. **INDEX.md**（读取时）：价值自述——「我包含什么，跳过我的代价」

**与现有设计的关系**：
- 是决策 #6（Journal 服务 Agent 而非用户）的直接应用——技能服务于 Agent 的信息需要，不向 Agent 发号施令
- 是「每个版本修正前一个版本的盲点」的最新迭代——v4.6.0 修正了「删除是合法操作」的盲点，本次修正「Agent 会主动读 INDEX」的盲点

### 22. 合约随技能版本同步——维护协议的一环

**设计意图**：发现合约在初始化时写入载体（RULES.md / AGENTS.md 等），之后技能版本迭代（v4.5→v4.6→v4.7→v4.8）带来的合约措辞改进不会自动回灌。模拟 Phase 3 发现，我的 RULES.md 仍残留 v4.5.1 时代的写操作范围（仅「复制、新建、编辑」）和启动措辞（泛泛的「获取背景知识」），且含拼写错误（`juonral`、`按照其中地`）。维护协议必须包含合约过期检测和更新建议，否则技能进化是纸上进化。

**机制**：
- Phase 0 新增 Step 11——搜索已部署合约并与最新模板按四维度逐项比对
- Phase 1 P1-S1 新增合约审查维度——对每个差异判断 UPDATE/FIX/无操作
- `design-discovery-contract.md` 新增已有合约检测——初始化时先检测再决定是更新还是新建
- 更新策略：内容升级而非格式替换——保留载体格式结构，仅升级内容要素

**与现有设计的关系**：
- 是 Protocol 设计模式「每个版本修正前一个版本的盲点」的本轮迭代：v4.7.0 修正了 Agent 不读 INDEX 的盲点，本版修正「已部署的启动指令不随技能升级」的盲点
- 合约审查与 archive 审查同属 Phase 0→Phase 1 的扫描→审查模式——两者共享相同的发现→评估→标记→执行结构


## 措辞风格约定

- **第一人称**："我记的，我用的"——journal 是 Agent 自己的笔记本，不是外部要求
- **中英混合**：结构性内容（frontmatter、标签）用英文，解释性内容和操作规则用中文
- **指令式而非叙事式**：操作规则用祈使句（"验证..."、"不要..."），不用"曾经犯了X错所以应该Y"的叙事结构。教训吸收为指令，案例留在 journal 中。

## 维护注意事项

- **操作规则是活文档**：SKILL.md 的 Operating Rules 在遭遇新的操作失败时追加。新增规则用指令式措辞，不夹带历史溯源——历史案例属于 journal 经验条目，不属于技能规范。
- **SKILL.md 的路由表和 Linked Files 要同步**：新增或移除 reference 文件时，两个位置都要更新。
- **泛化与脱敏**：本技能从特定运行环境迁移而来。示例中使用"项目 A/B"等通用占位符代替真实项目名。journal 路径用 `<journal-root>` 参数化。维护时保持这种通用性——不要回填特定项目名或硬编码路径。

## references/patterns/ 子目录——推荐式参考与规范的区分

`references/patterns/` 是 `references/` 下的子目录，存放**推荐式参考**文档（非硬规范）。

区分逻辑：`references/` 下的顶层文件为"规范/协议/说明"，`references/patterns/` 下的文档为"推荐模式/设计参考"，提供启发而非约束。

```
references/
├── spec-*.md        ← 规范
├── protocal-*.md    ← 协议/流程
├── design-*.md      ← 设计方法论
├── patterns/        ← 推荐模式（非约束）
│   └── dashboard.md
```

- [`references/patterns/dashboard.md`](references/patterns/dashboard.md) — 项目/领域级次级 INDEX 设计参考。提供创建信号的判断依据和内容组织示例，不是硬性规范。
