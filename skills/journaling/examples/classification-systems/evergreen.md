# Evergreen Notes

> 以概念为中心的笔记体系，强调笔记随时间"生长"而非一次性创建。笔记标题即 API——设计良好的标题让笔记成为整个想法的句柄。偏好关联本体（链接）而非层级分类法（文件夹）。

**创建者**: Andy Matuschak  
**核心分类维度**: 演化概念（笔记从种子开始，随时间精炼和链接）

## 核心原则

1. **原子性**：每条笔记精确封装一个概念
2. **概念导向**：笔记围绕想法而非来源组织
3. **密集链接**：链接即关系表述
4. **渐进式精炼**：笔记从不"完成"，持续更新
5. **标题即声明**：好标题本身就是结论（如"执行细节的笔记不如概念笔记有价值"）

## 笔记类型

| 类型 | 目的 | 示例 |
|------|------|------|
| 执行策略笔记 | 无明确方法论时做什么 | "如何选择要写的概念" |
| 笔记类型分类 | 区分不同笔记 | Taxonomy of note types |
| 元笔记 | 关于笔记系统的笔记 | "笔记写作的执行策略" |

## 工作流

```
阅读收件箱 (reading inbox)
    ↓ 处理外部来源
文献笔记 (literature notes)
    ↓ 提炼概念
永久笔记 (evergreen notes)
    ↑ 持续积累 + 链接
写作收件箱 (writing inbox)
    ↓ 将笔记组装为文章
输出
```

## 分类方式

Evergreen Notes **不在文件夹层面分类**。组织依赖：

- **命名约定**：标题即 API，一致性命名形成隐式分组
- **密集链接**：通过链接网络表达概念结构
- **MOC / Index**：必要时补充索引笔记
- **属性 / 标签**：可选的前端元数据

Andy 的原始笔记库结构极度扁平——所有 `.md` 文件在同一个目录，链接是唯一的组织工具。

## 适用场景

- **长期知识写作**：输出为文章、书籍的作者
- **概念探索**：需要持续思考、更新、迭代想法
- **个人知识体系**：不分享给团队，自己维护

## 不适用场景

- **快速捕获**：Evergreen 的高标准让初次捕获有压力
- **任务管理**：不处理"做什么"
- **需要共享**：个人化的系统难以被他人理解

## 改编要点

- "种子"的概念很有用——允许不完美的笔记存在，给它时间成熟
- 标题设计是最重要的技能——好的标题让你的思考可检索、可引用
- 不必完全扁平——可以在 `notes/` 目录下做一级分类（如 `concepts/`、`literature/`、`projects/`），保持索引密度
- 阅读收件箱和写作收件箱的分离是 Andy 系统的关键设计——两者目的不同，不应混在一个 inbox 中
