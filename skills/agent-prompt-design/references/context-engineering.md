## 1. 系统提示词是什么

系统提示词（System Prompt）不是一次性的请求（Prompt），而是 AI Agent 的**宪法、操作手册和行为锚点**三者的合一。

| 类比 | 含义 | 对 Agent 的影响 |
|------|------|-----------------|
| **📜 宪法** | 不可协商的基本规则和价值观 | 定义信任边界、拒绝场景、安全护栏 |
| **📖 操作手册** | 每天如何执行工作的流程指南 | 定义工具使用方式、任务分解方法、输出规范 |
| **⚓ 行为锚点** | Agent 的「身份 — 专业 — 目标」人格设定 | 决定语气风格、交互模式、决策偏好 |

关键区分（Dust Blog, 2026）:

- **Prompt（提示）** = 一次性任务请求。用户说「帮我写一封邮件」——这是一个 prompt。
- **Instructions（指令）** = 持久行为规则。系统提示词中的「发送邮件前始终请求确认」——这是一条 instruction。
- **System Prompt（系统提示词）** = 一组 persistent instructions + 角色定义 + 工具契约 + 安全策略的集合。

> Prompt is what you ask; System Prompt is who you are.

**第一阶安全变量：** 同一模型在不同系统提示词下，安全性能可从 **<1% 到 97%** 剧烈摆动（arXiv:2603.25056, 220K 次评估）。这意味着系统提示词在生产系统中的影响可能超过模型选择本身。

## 2. 从 Prompt Engineering 到 Context Engineering

### 2.1 范式转变

Anthropic 在 2025 年 9 月提出了一个关键论点：传统「提示词工程」已经不足以支撑生产级 Agent 的设计需求。我们需要的是 **Context Engineering（上下文工程）**。

| 维度 | Prompt Engineering | Context Engineering |
|------|--------------------|---------------------|
| **关注点** | 如何写出好的指令文本 | 如何管理 Agent 推理时的**全部 token 状态** |
| **范围** | 系统提示词本身 | 系统指令 + 工具定义 + MCP + 外部数据 + 消息历史 |
| **时间维度** | 单次推理 | 跨多轮推理、长时间运行 |
| **核心挑战** | 措辞和短语选择 | 注意力预算（Attention Budget）的分配 |
| **评估标准** | 输出质量 | 输出质量 + token 成本 + 推理效率 + 长程一致性 |

### 2.2 为什么需要这个转变

生产级 Agent 的上下文由一个**持续增长的动态系统**构成，而非静态文本：

```
      静态层                         动态层
  ┌─────────────┐           ┌─────────────────────┐
  │ 系统提示词    │           │ 对话历史（不断增长）   │
  │ 工具定义      │           │ 工具调用结果          │
  │ MCP 配置     │   ───→    │ 外部检索信息          │
  │ Few-Shot 示例│           │ 用户上传的文件        │
  └─────────────┘           │ 环境动态数据（OS等）   │
                             └─────────────────────┘
```

传统 Prompt Engineering 只关注**静态层**的文本质量，而 Context Engineering 需要回答：

- **注意力在哪？** 100K token 的上下文中，Agent 真正「看到」了什么？
- **什么值得保留？** 旧消息中的架构决策需要压缩还是丢弃？
- **什么必须靠近？** 关键指令是否被埋没在工具定义中？
- **边际收益在哪？** 再多加 500 token 的示例，值得付出注意力预算吗？

> *"Good context engineering means finding the smallest possible set of high-signal tokens that maximize the likelihood of some desired outcome."*
> — *Anthropic, "Effective Context Engineering for AI Agents" (Sep 2025)*

### 2.3 实际含义

Context Engineering 的核心输出是一个**经过注意力预算优化的上下文包**，而非一篇写得很好的文章。这意味着：

- **位置即优先级**：最重要的指令放在最前面，因为 Transformer 对开头 token 的注意力更集中（primacy bias）
- **结构即信号**：用 XML 标签、Markdown 标题、分隔符创造「注意力锚点」
- **噪信比（Noise-to-Signal Ratio）**：每添加一个 token，都在稀释已有 token 的信号强度
- **动态修剪**：长会话中主动管理上下文窗口，压缩或丢弃不相关信息

## 3. 注意力预算（Attention Budget）

### 3.1 核心概念

Transformer 的注意力机制产生 **n² 的成对关系**。每增加一个 token，所有 token 之间的注意力关系数量以平方级增长。

```
Token 数量 (n)    注意力对数量 (n²)
   1,000            1,000,000
   10,000          100,000,000
   100,000       10,000,000,000
   1,000,000   1,000,000,000,000
```

系统提示词的每个 token 都在消耗注意力预算。随着上下文增长，模型准确回忆特定信息的能力下降——这被称为 **Context Rot（上下文衰减）**。

### 3.2 上下文衰减（Context Rot）

Context Rot 不是二元失效（突然遗忘），而是**渐进的信噪比下降**：

| 上下文位置 | 信号强度 | 典型问题 |
|-----------|---------|---------|
| 开头 2K token | 最高 | Primacy bias – 但可能不是最重要的信息 |
| 中间 80K token | 中等 | 信息容易被「淹没」，尤其是与两端类似的内容 |
| 结尾 2K token | 高 | Recency bias – 最近的工具结果优先被注意 |
| 工具定义区 | 中等 | 当工具数量 > 30 时，每个工具的信号大幅稀释 |

**关键洞察：** 注意力预算是一个**具有递减边际回报的有限资源**。

```text
每块价值的动态变化：
    
   信号强度
      ↑
   高 │  ██████████████████  (系统提示词/角色定义)
      │           ████████  (工具定义，前5个)
      │                ███  (第10-20个工具)
      │                  █  (第20+个工具定义的边际价值)
      └────────────────────────→ 上下文长度
```

### 3.3 对系统设计的实际影响

**1. 最小的系统提示词 = 更好的系统提示词**

不要把所有「可能有用」的内容塞进去。每个额外 token 都在稀释已有指令的信号强度。

**2. 工具数量有隐性成本**

Anthropic 内部数据：使用 Tool Search（延迟加载）后，上下文保留率从 **23% 提升到 95%**，准确率从 **49% 提升到 74-88%**。

- ❌ 把 50 个工具定义一次性塞进上下文（50K+ token 污染）
- ✅ 维护一个最小可行工具集，按需延迟加载
  > → 工具定义在注意力预算中的具体占比（30-50%）参见 [tool-design.md §7.1](tool-design.md#71-注意力预算分配)

**3. 利用 Prompt Caching**

前置静态内容（系统提示词、示例、常用工具）以利用模型的 Prompt Caching 能力：
- xAI Grok 建议将静态内容放在开头以最大化缓存命中率
- 降低每次请求的 token 成本
- 提升响应速度

**4. 旧消息压缩策略**

长会话中，历史对话的边际回报递减。策略：
- 保留最近的 N 轮对话（保证 recency）
- 压缩或丢弃中间的「已完成任务」的 tool call 详情
- 保留关键架构决策的摘要

### 3.4 注意力预算检查清单

设计或审查系统提示词时，问自己：

- [ ] 如果只能保留 2000 token，这 2000 是什么？
- [ ] 当前提示词中哪个部分贡献的边际价值最低？可以删掉吗？
- [ ] 工具定义的总 token 数是多少？能否启用延迟加载？
- [ ] 关键指令是否被埋没在次要内容中（被注意力衰减区域覆盖）？
- [ ] 历史消息是否在吞噬预算？需要压缩或 checkpoint 吗？

---

### 3.5 Token 预算快速参考

| 提示词规模 | 判断 | 说明 |
|-----------|------|------|
| < 500 token | 精简 | 仅含角色 + 核心指令，适合极简场景 |
| 500–2000 token | 正常 | 包含角色、指令、少量边界规则 |
| 2000–5000 token | 偏大 | 需检查是否有冗余指令或示例 |
| > 5000 token | 危险 | 工具定义很可能是主要原因 → 见 [tool-design.md §7.1](tool-design.md#71-注意力预算分配) |

**速算公式：** `总预算 ≈ 指令 token + 工具定义 token × 工具数量 + 安全规则 token + 示例 token`。若总预算 > 3000 token 且工具数 > 15，强烈建议启用延迟加载（[tool-design.md §7.3](tool-design.md#73-延迟加载的实现模式)）。

## 4. 正确的海拔（Right Altitude）

### 4.1 问题空间

Anthropic 提出了系统提示词的「正确海拔」概念——在两种常见失败模式之间找到平衡：

```text
脆弱的硬编码逻辑  ←─────────────────→  模糊的高层指导
（过度具体，       ↑                   （过于笼统，
 维护困难）            正确的海拔：       模型无方向）
                  足够具体以有效引导行为，
                  又足够灵活以提供强启发式规则
```

| 失败模式 | 典型表现 | 后果 |
|---------|---------|------|
| **过度海拔（太具体）** | 硬编码 if-else 链、枚举所有边界情况 | 维护成本高、一个条件变化就需要重写整段、适应性差 |
| **不足海拔（太模糊）** | 「做个好助手」「按最佳实践操作」 | 模型无具体方向、输出不一致、难以调试 |

### 4.2 如何找到正确的海拔

**三问测试：**

1. **给同事看这个指令，他能理解吗？** 如果直觉上需要领域知识才能理解，海拔太高。
   - ❌ `"如果 MQL 的 SPICE 精度超过 3σ 且 DNL < 0.5 LSB，则标记为良品"`
   - ✅ `"如果模拟结果的精度和线性度在容差范围内，则通过校验"`
   - 具体细节由工具定义封装，而非写入指令

2. **指令对一个新模型或新场景仍然适用吗？** 如果高度依赖特定模型行为或特定工具版本，海拔太低。
   - ❌ `"调用 v3.2.1 的 /api/legacy/process 端点"`
   - ✅ `"调用当前系统注册的数据处理服务"`
   - 版本号/端点信息应注入为动态变量，而非硬编码

3. **去掉这条指令，行为会退化吗？** 如果去掉不影响，说明这条指令不够清晰，海拔太高（或完全多余）。

### 4.3 启发式规则 vs 硬编码逻辑

正确的海拔本质上是**从硬编码到启发式的转变**：

| 场景 | 硬编码（太具体） | 启发式规则（正确海拔） |
|------|-----------------|---------------------|
| 错误处理 | "重试 3 次，间隔 1 秒、2 秒、4 秒" | "瞬态错误应重试（指数退避）；逻辑错误应改变策略" |
| 输出格式 | "用 H2 标题，14px 字体，蓝色链接" | "输出应结构化、可读性强，对技术读者有足够的细节" |
| 任务分解 | "先查 CRM，再查 Support DB，然后找 Slack 历史" | "收集所有相关背景信息，再制定行动计划" |

### 4.4 海拔与注意力预算的关系

正确的海拔直接影响注意力预算的效率：

- 太具体的指令：占用了大量 token，但覆盖场景有限 → **注意力 ROI 低**
- 太模糊的指令：占用了少量 token，但未能有效引导 → **注意力 ROI 为 0**
- 正确的海拔：最少 token 覆盖最多场景 → **注意力 ROI 最高**

## 5. 从最小化开始，迭代添加

### 5.1 核心原则

Anthropic 推荐的 Agent 提示词开发流程——也是 Context Engineering 的实践入口：

> **先写最简提示词，然后观察 Agent 在什么场景下失败，再仅针对失败模式添加指令。**

```
最强模型 + 最简提示词
    ↓
建立基线性能（Evals）
    ↓
观察失败模式 → 仅针对失败模式添加指令/示例
    ↓
回归测试（回放历史交互）→ 无退化
    ↓
成本/延迟优化（替换更小模型）
    ↓
持续监控 + 迭代
```

### 5.2 为什么从最小化开始

**1. 避免过度拟合你自己的心智模型**

大部分系统提示词的设计问题是**设计者猜错了边界情况**。与其预先塞入你认为「可能有用」的规则，不如让 Agent 在真实场景中暴露真正的失败模式。

**2. 注意力预算的稀缺性**

每一条看似无害的「预防性」指令都在稀释核心指令的信号。如果 Agent 在 1000 次执行中只有 1 次会遇到某个边缘情况，为其预留 200 token 的指令是浪费。

**3. 维护成本**

外部存储的提示词仍然需要迭代和版本管理。多余的规则会：
- 增加后续修改时引入矛盾的风险
- 让新开发者难以判断哪些规则是必需的
- 增加 A/B 测试的变量空间

### 5.3 何时添加，何时不动

| 场景 | 操作 | 原因 |
|------|------|------|
| Agent 在某个场景下**行为不一致** | 添加一条清晰的指令 | 提供确定性引导 |
| Agent 在某个场景下**行为错误但一致** | 添加示例（Few-Shot） | 模式匹配比指令更有效 |
| Agent **忽略或错误使用工具** | 在工具描述中强化 | 工具定义是更好的修改点 |
| Agent **输出格式混乱** | 添加结构化输出约束 | JSON Schema 或预填响应骨架 |
| Agent **执行顺序错误** | 添加工作流步骤 | 流程性指令 |
| 某个场景**从未出现过** | ❌ 不添加 | 等它出现了再处理 |

### 5.4 最小化实践的注意事项

- **使用强模型验证**：从最强模型开始，排除模型能力不足导致的失败。如果最强模型在最简提示词下表现好，说明问题不在提示词设计。
- **失败案例驱动**：每条指令都应该能追溯到至少一个具体的失败案例。如果无法追溯，考虑去掉它。
- **回归测试是必须的**：每次添加指令后，用历史交互数据回放，验证新指令没有破坏已有行为。迭代失败 → 回滚。

### 5.5 最小化启动模板

```
---
Agent Name: [待定]
Description: [一句话描述]
---

You are a [具体角色]. Your goal is to [核心目标].

## Tools
[仅列出最小必要工具，每个工具一个句子描述]

## Boundaries
- [3-5 条最关键的边界，而不是所有可能出问题的地方]
```

从这开始运行，观察失败，然后**仅针对失败**添加以下内容（按优先级）：

1. 失败模式对应的指令
2. 失败模式对应的 Few-Shot 示例
3. 必要但遗漏的工具描述
4. 额外的边界规则
