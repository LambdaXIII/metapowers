# 测试场景选择与准备指南

> 本指南帮助 Agent 发现和准备测试场景，包含基础方法和扩展方法。

---

## 一、基础方法（必读）

### 1.1 场景发现的核心思路

测试的终点不是穷举所有可能性，而是确认技能的核心功能能正确运作。

**核心思路**：启发思路，而非穷举场景。

### 1.2 发散-收敛思维方法

**发散期**

**目标**：尽可能多地列出可能的测试场景。

**操作**：
1. 不要急于筛选，先尽可能多地列出场景
2. 使用"两个思考维度"（正向/反向）辅助发散
3. 记录所有想到的场景，哪怕看起来不太重要

**发散话术**：
- "这个技能在什么情况下会被用到？"
- "用户最常说什么样的话来触发这个技能？"
- "这个技能有什么功能分支或条件判断？"
- "什么输入可能导致技能失效或出错？"

**收敛期**

**目标**：从发散结果中筛选出最有价值的测试场景。

**筛选标准**：
- 是否覆盖核心功能？（必测）
- 是否覆盖功能分支？（必测）
- 是否有实际用户场景支持？（优先）
- 是否能发现潜在问题？（优先）

**收敛话术**：
- "哪些场景最能验证技能的核心功能？"
- "哪些场景能覆盖不同的功能分支？"
- "哪些场景有实际用户需求？"
- "哪些场景可能暴露问题？"

### 1.3 两个思考维度

**正向维度：典型使用场景**

**核心问题**：技能的典型使用场景是什么？什么情况下会用到？

**思考方向**：
- 用户最常说什么话来触发技能？
- 技能的主要功能是什么？
- 用户使用技能的典型流程是什么？
- 技能解决了什么问题？

**示例（以"cx-asset-design"为例）**：
- 用户说"帮我设计资产"
- 用户说"根据这个概念，设计资产"
- 用户说"我需要一份资产设计稿"

**反向维度：失效或出错场景**

**核心问题**：什么会让技能失效或出错？什么输入会破坏预期？

**思考方向**：
- 输入缺失时会发生什么？
- 输入格式错误时会发生什么？
- 输入内容冲突时会发生什么？
- 边界条件是什么？

**示例（以"cx-asset-design"为例）**：
- 用户只提供"概念"，没有提供"资产列表"
- 用户提供的"概念"过于简短或模糊
- 用户提供的"资产列表"与"概念"不匹配

### 1.4 场景准备操作步骤（5步法）

| 步骤 | 操作 | 输出 |
|------|------|------|
| 1. 技能分析与正向思考 | 分析能力结构，列出典型使用场景 | 场景清单（草稿） |
| 2. 反向思考 | 列出可能导致技能失效的情况 | 场景清单（草稿） |
| 3. 整合 | 筛选出最有价值的场景（3-5个） | 场景清单（精简） |
| 4. 分类 | 确定每个场景的类型 | 场景清单（带分类） |
| 5. 整理 | 为每个场景生成 eval_metadata.json | eval_metadata 文件 |

### 1.5 技能分析技巧（步骤1详解）

在列出典型场景之前，先快速分析技能结构：

**分析维度**：
- 能力有哪些？（确保每个能力都有测试）
- 执行模式有哪些？（确保每种模式都有测试）
- 有什么依赖关系或决策规则？（考虑是否需要测试）

**分析方法**：
1. 读取被测试技能的 SKILL.md
2. 查找"能力"、"功能"、"执行模式"、"决策规则"等章节
3. 在正向思考时，确保覆盖这些维度

**示例（cx-asset-design）**：
- 能力：概念提炼、资产识别、资产设计
- 执行模式：完整流程、单项能力、增量更新
- 正向思考时列出：完整流程场景 + 3个单项能力场景 + 增量更新场景

### 1.6 场景分类

| 类型 | 说明 | 优先级 |
|------|------|--------|
| **核心场景** | 技能的核心用途、最常见请求 | 必测 |
| **扩展场景** | 核心场景的变化/组合 | 建议测 |
| **边界场景** | 输入在边缘的情况 | 按需 |
| **失败场景** | 可能导致失效的情况 | 按需 |

### 1.7 实战示例

#### 示例：测试"cx-asset-design"技能

**发散期**：

正向思考（5-10个）：
1. 用户说"帮我设计资产"
2. 用户说"根据这个故事，设计资产"
3. 用户说"我需要一份资产设计稿"
4. 用户说"设计一个场景的资产"
5. 用户说"根据这个故事，提炼概念"
6. 用户说"识别这个故事需要哪些资产"
7. 用户说"设计这个角色的外观"

反向思考（5-10个）：
1. 用户只提供"概念"，没有提供"资产列表"
2. 用户提供的"概念"过于简短
3. 用户提供的"资产列表"与"概念"不匹配
4. 用户没有提供任何输入
5. 用户提供的输入格式异常

**收敛期**：

筛选 3-5 个最有价值的场景：
1. 核心场景：用户提供故事内容，要求完整资产设计（概念提炼 → 资产识别 → 资产设计）
2. 扩展场景：用户只提供故事片段，技能自动识别关键资产
3. 边界场景：用户提供的故事内容过于简短

**分类**：
- 场景1：核心场景（必测）
- 场景2：扩展场景（建议测）
- 场景3：边界场景（按需）

## 二、扩展方法（按需阅读）

> 当用户要求更全面的测试覆盖时，阅读本章节。

### 2.1 何时使用扩展方法

**触发条件**：
- 用户说"测试要全面一些"
- 用户说"扩展测试范围"
- 用户说"测试所有能力/模式"
- 用户要求覆盖更多场景

**目标**：从基础测试（3-5个场景）扩展到全面测试（覆盖所有能力/模式）。

### 2.2 能力覆盖分析方法

#### 步骤 1：提取技能能力清单

**方法**：
1. 读取被测试技能的 SKILL.md
2. 查找"能力"、"功能"、"功能模块"等章节
3. 列出所有能力及其职责

**示例（cx-asset-design）**：
```
能力清单：
- 概念提炼：从故事内容中提炼视觉美学方向
- 资产识别：从故事内容中系统提取需要设计的资产
- 资产设计：为每个资产生成具体可执行的设计规范
```

#### 步骤 2：为每个能力设计测试场景

**方法**：
- 每个能力至少一个测试场景
- 测试场景应能验证能力的核心功能

**示例（cx-asset-design）**：

| 能力 | 测试场景 | Prompt 示例 |
|------|---------|-------------|
| 概念提炼 | 单项能力-仅概念提炼 | "帮我提炼这个故事的视觉概念" |
| 资产识别 | 单项能力-仅资产识别 | "帮我识别这个故事需要设计哪些资产" |
| 资产设计 | 单项能力-仅资产设计 | "帮我设计这个角色的外观" |

### 2.3 执行模式覆盖分析方法

#### 步骤 1：提取执行模式清单

**方法**：
1. 读取被测试技能的 SKILL.md
2. 查找"执行决策"、"触发规则"、"能力触发规则"等章节
3. 列出所有执行模式及其触发条件

**示例（cx-asset-design）**：
```
执行模式清单：
1. 完整流程：用户意图为新项目做完整规划 → 概念提炼 → 资产识别 → 资产设计
2. 单项能力：用户意图只做某一项 → 只执行对应能力（如需要，先补齐依赖）
3. 增量更新：用户意图增加新设定/修改 → 只执行资产设计（如需要，先执行资产识别）
```

#### 步骤 2：为每种执行模式设计测试场景

**方法**：
- 每种执行模式至少一个测试场景
- 测试场景应能验证模式的触发和执行

**示例（cx-asset-design）**：

| 执行模式 | 测试场景 | Prompt 示例 |
|---------|---------|-------------|
| 完整流程 | 核心场景-完整资产设计 | "帮我设计这个故事的资产" |
| 单项能力 | 单项能力-仅概念提炼 | "帮我提炼这个故事的视觉概念" |
| 增量更新 | 扩展场景-增量更新 | "在现有设计基础上，增加一个新角色" |

### 2.4 依赖补齐测试方法

#### 步骤 1：提取依赖关系

**方法**：
1. 读取被测试技能的 SKILL.md
2. 查找"依赖"、"依赖补齐"、"依赖关系"等章节
3. 列出所有依赖关系及补齐逻辑

**示例（cx-asset-design）**：
```
依赖关系：
- 执行资产识别时 → 检查是否有概念提炼 → 没有则先执行概念提炼
- 执行资产设计时 → 检查是否有资产清单 → 没有则先执行资产识别
```

#### 步骤 2：设计依赖补齐测试场景

**方法**：
- 测试场景应触发依赖补齐逻辑
- 验证补齐后的执行是否正确

**示例（cx-asset-design）**：

| 依赖关系 | 测试场景 | Prompt 示例 | 预期行为 |
|---------|---------|-------------|---------| 
| 资产识别依赖概念提炼 | 依赖补齐-资产识别 | "帮我识别这个故事的资产"（未提供概念） | 先执行概念提炼，再执行资产识别 |
| 资产设计依赖资产清单 | 依赖补齐-资产设计 | "帮我设计这个角色"（未提供资产清单） | 先执行资产识别，再执行资产设计 |

### 2.5 决策规则测试方法

#### 步骤 1：提取决策规则

**方法**：
1. 读取被测试技能的 SKILL.md
2. 查找"决策规则"、"输出决策"、"判断规则"等章节
3. 列出所有决策规则及判断条件

**示例（cx-asset-design）**：
```
决策规则：
- 无需保存文件：少量资产（1-3个）、用户未要求保存、内容简单
- 需要保存文件：多资产（4个以上）、用户明确要求保存、内容复杂
```

#### 步骤 2：设计决策规则测试场景

**方法**：
- 测试场景应覆盖不同的决策分支
- 验证决策结果是否符合预期

**示例（cx-asset-design）**：

| 决策规则 | 测试场景 | Prompt 示例 | 预期行为 |
|---------|---------|-------------|---------|
| 输出决策-少量资产 | 边界场景-少量资产 | "帮我设计一个角色"（只有一个资产） | 无需保存文件，对话中完成 |
| 输出决策-多资产 | 边界场景-多资产 | "帮我设计这个故事的所有资产"（多个资产） | 需要保存文件 |

## 三、常见问题

### Q1：如何判断一个场景是否值得测试？

A：使用以下判断标准：
- 是否覆盖核心功能？（是 → 优先测试）
- 是否有实际用户场景支持？（是 → 优先测试）
- 是否能发现潜在问题？（是 → 优先测试）
- 是否覆盖功能分支？（是 → 建议测试）

### Q2：应该测试多少个场景？

A：建议 3-5 个场景：
- 最少 3 个：确保覆盖核心功能
- 最多 5 个：避免过度测试，影响效率
- 根据技能复杂度调整：复杂技能可适当增加

### Q3：如何处理边界场景？

A：
- 边界场景不是必测项
- 如果有明确的边界条件，建议测试
- 如果边界条件不明确，可以省略

### Q4：如何处理失败场景？

A：
- 失败场景不是必测项
- 如果有明确的错误处理逻辑，建议测试
- 如果失败场景不常见，可以省略

### Q5：何时使用扩展方法？

A：
- 用户明确要求"全面测试"或"扩展范围"
- 技能有多种能力或执行模式，需要全面覆盖
- 测试发现基础场景遗漏了重要功能

## 四、总结

**场景发现的核心**：
1. 使用发散-收敛思维
2. 用正向/反向两个维度辅助发散
3. 筛选 3-5 个最有价值的场景
4. 确保覆盖核心功能和主要分支

**扩展分析的核心**：
1. 提取技能的能力、执行模式、依赖关系、决策规则
2. 为每个维度设计测试场景
3. 使用检查表验证覆盖情况

**记住**：
- 基础方法：测试的目标是验证技能能否正常工作，而非穷举所有可能性
- 扩展方法：测试的目标是全面覆盖技能的所有特性，确保没有遗漏
