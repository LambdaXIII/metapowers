## 1. 五条铁律 (Five Iron Rules)

内容编写有 **五条铁律**，违反了任何一条都会导致 Agent 行为不可控。每条铁律后面附有 ❌ 错误 / ✅ 正确 的对比示例。

### 1.1 铁律一：清晰直接 (Be Clear and Direct)

**原则：** 指令必须足够具体，让一个没有语境上下文的同事也能理解。如果有任何歧义，LLM 也会困惑。

> *"把提示词给一个没有上下文的同事看。如果他困惑，Claude 也会困惑。"*
> —— *Anthropic Prompt Engineering Guide*

**为什么：** LLM 不会主动填补模糊信息的空白。模糊的指令意味着模型用自己的假设来填补——而这些假设往往不是你想要的。

| ❌ 模糊指令 | ✅ 清晰指令 |
|-------------|-------------|
| "写一封营销邮件" | "为目标受众（100-500 人中型科技公司）写一封 Q3 AcmeCloud 功能发布邮件。突出三个功能：高级加密、跨平台同步、实时协作。语调：专业但平易近人。CTA：免费 30 天试用。主题行：<50 字符，包含 '安全' 和 '协作'" |
| "优化这段代码" | "重构这个函数，使时间复杂度从 O(n²) 降至 O(n log n)。保持现有 API 签名不变。添加类型注解和单元测试。" |
| "检查这个文档" | "检查这份合同的第三条赔偿条款是否符合加州法律 SB-1234 的要求。如果不符合，标记具体差异并推荐修正措辞。" |

**自我检查：** 把你的指令读给身边的人听。如果他开始问"你说的 XX 是什么意思？"，你就需要重写了。

---

### 1.2 铁律二：描述「做什么」而非「怎么做」 (Describe WHAT not HOW)

**原则：** 告诉 Agent **要达成的目标**（what），而不是**实现步骤**（how）。业务逻辑、数学计算、确定性流程应该封装在工具定义中，而不是写在提示词里。

**为什么：**
- Agent 不擅长数学、复杂计算或确定性流程——这些应该移入工具/函数
- 把"怎么做"硬编码在提示词中会导致**脆弱的 if-else 链条**，维护困难
- 工具才是 Agent 与外部世界之间的**契约**——工具定义应当承担操作细节

| ❌ 描述怎么做 (HOW) | ✅ 描述做什么 (WHAT) |
|---------------------|---------------------|
| "调用 CreateCase API，使用上一个查询中的客户 ID，将 case_type 设为 'support'，然后调用 AssignAgent API 分配工单" | "为这个客户创建支持工单" |
| "查询 users 表获取 email，然后 format_email() 格式化，再调用 sendmail() 发送" | "给这个用户发送重置密码邮件" |
| "先读取文件，base64 编码，然后检查文件大小是否 < 10MB，再上传到 S3" | "上传这份文档到知识库" |

**核心准则：** 业务逻辑 → 工具定义；目标指令 → 系统提示词。

---

### 1.3 铁律三：正面表述 (Affirmative Phrasing)

**原则：** 始终使用**正面指令**描述期望行为，而不是用**负面指令**描述禁止行为。

**为什么：** LLM 对正面指令的理解更可靠。负面表述（"不要做 X"）在复杂场景下容易被忽略或错误处理——模型需要先理解"不要做"的内容，再推断"应该做什么"，多了一层认知负担。正面指令直接告诉模型目标状态，执行的路径更短、更稳定。

| ❌ 负面表述 | ✅ 正面表述 |
|-------------|-------------|
| "未经用户批准不要发送邮件" | "发送邮件前始终请求确认" |
| "不要修改原始数据" | "在修改前创建数据的副本" |
| "不要跳过验证步骤" | "在每个步骤完成后运行验证" |
| "不要把 API key 写进代码" | "将 API key 存储在环境变量中，通过 `os.getenv()` 读取" |

**原理机制：** 当 LLM 看到"不要做 X"时，它在推理过程中仍然会在 token 空间中对"做 X"进行表征激活——负面指令本质上是用一个较强的激活抑制另一个激活。而正面指令直接建立期望的激活路径，更不容易在长推理链中衰减。

---

### 1.4 铁律四：一个思想，一条指令 (One Thought, One Instruction)

**原则：** 每条指令应该表达**一个独立的逻辑单元**。如果逻辑需要分支，为每个分支编写独立指令。混合同一逻辑维度的多个条件会导致**矛盾指令**——这是"沉默的杀手"。

**为什么：** 两条矛盾的指令共存时，LLM 不会报错，不会提醒你，只会默默地做出不一致的行为。这对调试极为困难，因为问题不是"出错了"，而是"不一致"。

| ❌ 矛盾指令 | ✅ 一致指令 |
|-------------|-------------|
| "总是回复中文" 和 "对技术术语保留英文" 在同一级别 | "默认回复中文。对没有标准中文译名的技术术语保留英文原文，并在括号中给出首次出现的术语的中文解释。" |
| "优先使用 Markdown 格式" 和 "输出纯文本格式" | "输出格式为 Markdown，使用 `##` 标题、`-` 列表和 `` ` `` 代码标记" |
| "尽可能详细" 和 "保持简洁" | "输出控制在 3-5 段。每个观点用一段话阐述，不要超过 3 个主要观点。" |

**如何处理逻辑分支：**
```
✅ 正确做法：
如果你遇到 A 情况 → 执行指令 X
如果你遇到 B 情况 → 执行指令 Y
（不要写成：遇到 A 和 B 时，做 X 但也要考虑 Y）

✅ 更精细的做法：
- If [传输协议是 HTTP]: 警告用户不安全，询问是否继续
- If [传输协议是 HTTPS]: 直接传输，无需额外确认
```

---

### 1.5 铁律五：使用可验证的条件 (Use Verifiable Conditions)

**原则：** 条件判断必须基于**可观察、可验证、边界明确**的事实，而不是基于模糊的语义概念。

**为什么：** LLM 对模糊边界的理解与人类不同。你眼中"数据缺失"和"数据不完整"的区别，在 LLM 看来可能完全一样。使用**可在代码层面直接判定**的条件来消除歧义。

| ❌ 模糊条件 | ✅ 可验证条件 |
|-------------|---------------|
| "如果数据缺失，询问。如果数据不完整，请求澄清。"（缺失 vs 不完整的边界在哪？） | "如果必填字段为空 → 要求提供。如果可选字段为空 → 跳过继续。" |
| "如果用户看起来不高兴，安抚一下" | "如果用户输入中包含 '生气'、'失望'、'投诉'、'差评' 等关键词 → 先道歉，然后提供补偿方案" |
| "如果响应时间太长，考虑超时" | "如果 API 调用在 30 秒内未返回 → 触发重试机制（最多 3 次）" |
| "如果内容不合适，拒绝生成" | "如果用户请求的内容命中以下任一拒绝类别：非法、有害、仇恨、隐私侵犯、恶意软件、自残、虚假信息 → 拒绝请求并不提供解释" |

**边界明确化对比：**

```
❌ 模糊：
  如果字段是空的 → 处理
  如果字段不完整 → 也处理
  （两个条件重叠，Agent 无法区分）

✅ 可验证：
  必填字段为空 → 要求用户提供该字段的值
  可选字段为空 → 使用默认值或跳过
  字段已填写但格式无效 → 提示格式要求，请用户重填
  （每个条件互斥、可验证）
```

---

> **💡 记不住五条铁律？试试优先级栈（Priority Stack）进行简化：** 如果五条铁律同时出现让你觉得难以权衡，可以用**优先级栈**这个心智模型来简化决策——把规则按优先级排序，冲突时高优先级自动覆盖低优先级。P1（安全/法律/隐私，永不违反）→ P2（准确性/事实正确，强烈优先）→ P3（语调/格式/长度，默认行为）→ P4（个性/幽默，锦上添花）。这样一来，当"清晰直接"和"正面表述"看似矛盾时，你只需判断哪条优先级更高即可，无需编写复杂的分支逻辑。详见 [anti-patterns.md 中的优先级栈完整说明](anti-patterns.md#替代方案优先级栈priority-stack)。

> **🔄 模型迁移提示：** 从模型 A 迁移到模型 B 时，复查以下铁律是否需要调整：
> - 铁律二（描述 WHAT 非 HOW）：推理模型的步骤容忍度不同——非推理模型可能需要更明确的 HOW，推理模型则更倾向 WHAT
> - 铁律四（一思想一指令）：不同模型对复杂指令的消化能力不同，迁移时可能需要拆分或合并指令
> - 铁律五（可验证条件）：条件触发阈值可能因模型变更而偏移——某个模型能准确判断的条件，另一个模型可能误判

## 2. Few-Shot 示例策略

### 2.1 基本原则

- **不要把所有边界情况塞入提示词**。提示词中塞入 50 个示例不会让模型更好——只会消耗注意力预算。
- 精选一组**多样化、具有代表性**的示例。2-5 个精心设计的示例通常优于大量平庸的示例。
- 示例应封装在 `<example>` 标签中（与指令区域隔离）。

### 2.2 代表性四维检查（操作标准）

> **来源：** SurePrompts' "Few-Shot Example Selection Guide" (April 2026)

选择示例时，不能仅笼统要求"有代表性"。以下是四个可操作、可自检的维度：

| 维度 | 检查项 | 不合格的信号 |
|------|--------|------------|
| **相似性** | 示例的输入特征和实际查询是否接近？（领域、长度、措辞风格） | 示例全是短句但实际查询是长段落 |
| **多样性** | 是否覆盖了不同输出类别、不同长度、至少一个边界情况？ | 3 个示例都是同一种输出格式 |
| **排序** | 最强的示例是否放在最后（最靠近查询的位置）？ | 示例顺序随机，无策略 |
| **格式精确性** | 示例的输出格式是否和目标格式完全一致？ | 示例用 JSON 但指令要求 YAML |

**相似性陷阱：** 纯相似性不足——最少包含一个边界情况（如包含否定的查询），避免近重复（cosine > 0.92 → 去掉一个）。

**Rule of thumb：** 如果 3 个示例就能代表全部任务 → 使用固定集。如果不断想加示例 → 改为动态选择。

> ⚠️ **警告：** 当指令和示例冲突时，示例往往胜出。确保示例的输出格式与你的格式指令**精确一致**。

### 2.3 结构化输出任务中 Few-Shot 尤其有效

对于需要特定输出格式的任务（JSON、表格、报告模板），Few-Shot 示例的效果显著——模型通过**模式匹配**学习格式，比自然语言描述格式更可靠。

```
✅ 推荐格式：

<example>
User: 帮我写一封请假邮件给经理
Assistant: 好的，以下是请假邮件草稿：

**主题：** 年假申请 - [日期范围]

尊敬的[经理姓名]：

您好！我计划于 [开始日期] 至 [结束日期] 休年假，共计 [天数] 天...
</example>

<example>
User: 将这段文字转成 Markdown 表格
Assistant: | 名称 | 数量 | 单价 |
|------|------|------|
| 物品A | 10   | ¥25  |
| 物品B | 5    | ¥48  |
</example>
```

### 2.4 何时增加示例

遵循 Anthropic 推荐的迭代式流程：先使用最强模型 + 最简提示词 → 观察失败模式 → **仅针对失败模式**添加示例。不要提前把所有边界情况塞入。

## 3. 角色定义

### 3.1 三要素模型

Dust Blog（2026）提出精确角色定义的**三要素**——缺少任何一个，角色定义都是不完整的：

| 要素 | 说明 | ❌ 弱定义 | ✅ 强定义 |
|------|------|-----------|-----------|
| **身份 (Identity)** | 你是谁？在什么组织/场景中？ | "你是一个助手" | "你是面向中型销售团队的 SaaS 平台的客户支持专家" |
| **专业领域 (Expertise Domain)** | 你擅长什么？能提供什么价值？ | "你擅长回答问题" | "你专精于 CRM 数据迁移、API 集成故障排除和报告定制" |
| **目标 (Goal)** | 你要达成什么结果？如何衡量成功？ | "帮助用户" | "你的目标是在首次交互中解决至少 80% 的客户问题" |

### 3.2 公式

```
你是一个 [身份]，专注于 [专业领域]。你的目标是 [可衡量的目标]。
```

### 3.3 完整示例对比

```
❌ 弱定义：
"你是一个有帮助的 AI 助手。"

✅ 强定义：
"你是 AcmeCloud 面向医疗行业客户的高级解决方案架构师。
你专精于 HIPAA 合规架构设计、EHR 系统集成、医疗数据迁移方案。
你的目标是在 30 分钟内为客户设计出符合合规要求的架构方案，
并在首次会议中回答至少 90% 的技术问题。"
```

**注意：** 角色定义不要超过 3-4 句话。过长会模糊核心定位。

> **交叉引用：** 此处的"角色定义三要素"对应 [structure-design.md 的 8 大组件 §3](structure-design.md#3-生产级-agent-的-8-大组件)。结构设计决定了角色定义**放在哪里**（提示词最前面，利用 primacy bias），内容编写决定了角色定义**写什么**（身份+专业+目标）。

## 4. 输出格式控制

### 4.1 核心原则

- **结构化输出优先使用 JSON Schema**（比自然语言描述更节省 token，更精确）
- **预填响应骨架**：开始一个 JSON 块或报告大纲，让模型补全
- **明确约束**：长度、结构、语调、必须元素

### 4.2 JSON Schema vs 自然语言描述

| 维度 | ❌ 自然语言描述格式 | ✅ JSON Schema 控制格式 |
|------|--------------------|------------------------|
| Token 消耗 | 高（需要用文字描述结构） | 低（Schema 本身就是结构） |
| 精确度 | 低（模型可能遗漏字段） | 高（Schema 强制执行） |
| 可验证性 | 低（需要人类检查） | 高（可自动验证 JSON） |
| 示例 | "请以 JSON 格式返回，包含 name、age、email 字段" | `{"type":"object","properties":{"name":{...},"age":{...},"email":{...}}}` |

### 4.3 预填响应骨架

**原理：** 给模型一个已经开始的输出骨架，模型会自然地补全而不是重新生成格式。这是引导输出格式最可靠的方式之一。

```
❌ 不预填：
"请返回一个 JSON 对象，包含标题、摘要和三个要点"

✅ 预填骨架：
{
  "title": "
  （模型从这里开始补全）
```

实际应用中：

```
✅ 预填骨架示例：

{
  "summary": "",
  "key_points": [],
  "action_items": [],
  "risk_level": "",
  "recommendation": ""
}
```

### 4.4 明确长度、结构、语调和必须元素

所有的格式约束应当**量化**而不是**定性**。

| 维度 | ❌ 模糊约束 | ✅ 量化约束 |
|------|-------------|-------------|
| **长度** | "简明扼要" | "不超过 200 字" 或 "3-5 段，每段 2-3 句" |
| **结构** | "结构化输出" | "使用 ## 标题分段，每个建议用 - 列表条目" |
| **语调** | "专业一点" | "语调：正式、技术向，使用被动语态。避免第一人称。" |
| **必须元素** | "要有来源" | "必须包含：① 事件日期 ② 来源链接 ③ 影响范围" |

### 4.5 输出格式完整模板

```markdown
## Output Format

- **Length:** 3-5 paragraphs, each 2-3 sentences max
- **Structure:** 
  ```
  ## [Section Title]
  
  **Key Point:** ...
  
  - Supporting detail 1
  - Supporting detail 2
  ```
- **Tone:** Professional, technical, use third person
- **Required elements:** 
  - Cite sources for all factual claims [format: (Author, Year)]
  - Include a "Next Steps" section at the end
  - Flag any uncertainty with ⚠️ marker
- **Forbidden:** 
  - Do NOT include placeholder text like "[Insert date]"
  - Do NOT use emojis
```

## 附录：快速检查表

写作完成后，用这份清单检查每条铁律：

- [ ] **铁律一**：指令给一个没有上下文的同事看，他能理解吗？
- [ ] **铁律二**：我是在描述"做什么"还是"怎么做"？业务逻辑应该放在工具定义中吗？
- [ ] **铁律三**：所有指令都是正面表述吗？有没有"不要做 X"的负面表述可以改成"始终做 Y"？
- [ ] **铁律四**：每条指令表达一个独立逻辑吗？有没有矛盾的指令对？
- [ ] **铁律五**：所有条件判断都基于可验证的事实吗？边界够清晰吗？
- [ ] **Few-Shot**：2-5 个多样化、有代表性的示例？封装在 `<example>` 标签中？
- [ ] **角色定义**：身份 + 专业领域 + 目标 三要素齐全？
