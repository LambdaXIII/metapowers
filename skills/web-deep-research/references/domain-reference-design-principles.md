# 领域参考文档设计原则

📍 `web-deep-research` · 从 creative-work.md 和 person-biography.md 优化中归纳

---

新增任何 `references/` 下的领域特化参考文档时，以下原则可以作为自检清单。它们来源于 creative-work.md 和 person-biography.md 两次完整的「重写 → 推演 → 修复」循环——每条原则在两个文档上都交叉验证过。

## 1. 来源组织的轴线应是"寻源目的"，不是"来源权威"

**旧的错误模式**：按可信度分档（优先/次选/谨慎使用）。这是 Phase 3 评估逻辑的前置——收集阶段不需要"谁更可信"，需要"去哪儿找什么"。

**正确的做法**：
- 每个来源标注它擅长产出**哪类信息**，可信度问题留给 Phase 3
- 来源列表按"信息需求"分组，不按"权威等级"分组
- "优先来源（必查）"选的是覆盖信息维度最广的来源，不是最权威的来源

## 2. 维度组织轴线 = 该领域人类对研究对象的自然认知顺序

**旧原则（v1）**：按信息的客观性梯度分层（事实→半事实→主观）。这个原则对 creative-work.md 有效，但对 person-biography.md 造成了问题——用户指出"你把人物当成物体了"。客观性梯度适合分析"物"（作品），不适合理解"人"。

**修正后的做法**：

维度组织的第一问不是"这个信息的客观性如何"，而是"**人们描述这个对象的时候，自然用什么顺序？**"

- **人物**：身份→生平→关系→外界评价→成果。这是叙事逻辑——先知道是谁，再了解经历了什么，再看他跟谁相关，再看别人怎么评价，最后看留下了什么
- **创意作品**：本体→商业→主创→创作背景→艺术特色→批评→影响。客观性梯度在此仍然有效，因为作品是"物"——分析"物"时区分事实和判断比理解"人"时更自然

如果发现自己在用一个领域适用的维度轴线强行套到另一个领域——停下来，先问：我一口气跟朋友介绍这个东西的时候，会用什么样的顺序？那个顺序就是这个领域的组织轴线。

**自检问题**：读一遍维度列表。每一步都让你觉得"接下来当然应该了解这个"——那就是对的轴线。如果某个步骤让你觉得"为什么这个在这儿"——轴线错了。

## 3. 越详细，越需要收束机制

**问题**：特化参考的价值在于比通用流程更有的放矢——但如果细节铺开了没有收束，会逼迫 Agent 全面铺开。

**正确的做法**：
- 每个特化文档必须自带至少一条**显式收束规则**
- Tier 3（主观维度）的收束格式：声明"如果 Phase 0 未要求，仅纳入最显著的内容"，并给出 3-5 个具体示例说明"什么算最显著"
- 收束规则不否定详细度——它保护详细度不被滥用

## 4. 加载时机 = 文档价值的一半

**问题**：Agent 在错误的阶段读到特化文档，等于没读。

**正确的做法**：
- 入口文件（SKILL.md）不绑定"必须先加载"或"必须 Phase X 加载"
- 两级安全网：SKILL.md 做预判（能看出来就提前加载），workflow 做复查（Phase 0 完成意图拆解后检查是否漏了特化文档）
- 入口文件不枚举特化文档名，只指回 Content Index 表格——避免每加一个文档就要改两处

## 5. 来源分类命名：按信息的生产者，不按信息的用途

**错误模式**：用功能命名——"专业评估"\"深度分析"\"辅助参考"。Agent 读到"专业评估"时无法直观判断这和"权威来源"的区别——两者都涉及"专业的人"。

**正确的做法**：
- 类别名标定信息的生产者圈子：学界与评论界、权威机构、公众舆论、一手见证者
- 这样 Agent 不需要理解"这个类别是干嘛的"——看到"学界与评论界"就知道是学者和评论人写的，看到"权威来源"就知道是官方/机构发布的
- 类别之间的边界由"谁说的"区分，而非"说的是什么"——"谁说的"是客观属性，"说的是什么"需要 Agent 自己判断

**案例**：person-biography.md 最初用"专业评估"命名学者传记+行业史+批评研究的聚合类别，用户读后问"专业评估是啥意思"——因为这个名字暗示了功能（评估），而非来源（学界与评论界）。改为"学界与评论界"后边界自明。creative-work.md 的"专业评论"同理——虽然当时没触发困惑，但同一逻辑适用。

## 6. 身份消歧是硬门控，不是"尽量"

**适用场景**：人物类特化文档。但原理可泛化到任何存在"找错对象"风险的领域（公司同名、地名同名、术语多义等）。

**问题**：即使是全名+身份限定词，公众人物的姓名冲突率仍然极高。Agent 可能在 Phase 2 中途才发现搜到的资料属于另一个同名人物——浪费了前面的收集工作。

**正确的做法**：
- 在 Phase 1 设置硬门控：**Phase 1 结束时必须完成目标对象身份的锁定**——通过搜索验证、上下文推理或向用户询问，三种手段任一均可
- 给出具体的消歧属性清单（人物：性别、国籍/地区、活跃时期、职业领域、关键生平节点），让 Agent 有可操作的比对手段
- 声明穷尽手段仍无法锁定时中止并向用户确认——不猜测
- 贯穿全程：此后每打开一个新来源，先扫一眼基本信息确认是目标对象

**案例**：person-biography.md 的"重名验证（贯穿全程）"段落给出了 5 个消歧属性 + 持续核对指令 + 中止条件。creative-work.md 的"标题策略"虽然没有硬门控措辞，但"多语言变体 + 年份标注"本质是同一模式的轻量版本——对象的身份消歧。未来新增公司/组织/地名类特化文档时，这个模式应被复用。

## 自检清单

新增领域参考文档时，逐条检查：

- [ ] 来源分类用的是"信息需求→去这里"结构，不是权威分级？
- [ ] 维度组织轴线是否匹配该领域的自然认知顺序（人物→叙事逻辑，作品→客观性梯度）？用"跟朋友介绍时会用什么顺序"自测？
- [ ] 维度参考的开头是否声明了"启发提示，不是结构框架，不是核对清单"的软语气？
- [ ] 主观部分（人物：外界评价+成果展开；作品：Tier 3）有收束规则 + 具体示例说明"什么算最显著"？
- [ ] 文档开头声明了"可以参考"的软语气？
- [ ] SKILL.md 的 Content Index 表格已更新（一行），Instructions 不需要改？
- [ ] 新文档和已有特化文档（creative-work.md, person-biography.md）的术语和结构对齐？
- [ ] 来源类别名标定的是信息生产者（谁说的），不是信息用途（干嘛用的）？
- [ ] 如果该领域存在"找错对象"风险，是否有 Phase 1 身份锁定的硬门控 + 具体消歧属性清单 + 中止条件？
