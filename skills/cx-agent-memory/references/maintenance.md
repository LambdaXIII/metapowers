# 整理维护指导

> **触发时机**：Agent 在感知到记忆库存在查找困难、命名混乱、信息过载等问题时触发，不要求定期整理。用户明确要求时也可以触发。

---

## 核心原则

### 按需维护

**只在感知到混乱时进行**，不要求定期整理。

维护是**演化过程**，不是修正错误——每次维护都基于当前使用状况，制定适合当下的约定。

### Agent自主决策

本指导提供**观察维度和决策原则**，不预设具体方案。

Agent需根据实际情况判断，前瞻性地制定适合当前记忆库的方案。

### 最小干预

只在必要时修改，避免"顺便"优化——每次维护只解决决策清单中的问题。

---

## 维护流程：诊断-决策-执行-验证

### 第一阶段：诊断（观察维度）

**目标**：从多个维度观察记忆库现状，识别**真实困扰**（而非"应该整齐"的审美追求）。

**不是"问题清单"，而是"观察视角"**——描述你发现的感受，而非勾选。

#### 维度1：查找效率

**观察角度**：
- 想找一个内容时，第一反应是去哪里找？
- 脑海中是否有清晰的"记忆库地图"？
- 还是每次都要搜索或遍历才能定位？

**可能的发现**：
- 某些内容总是想不起来在哪
- 需要同时打开多个文件才能确认信息
- 对记忆库的整体结构缺乏"心理模型"

**如果存在这些感受** → 结构或索引需要优化

#### 维度2：命名模式

**观察角度**：
- 文件名是否能快速传达内容主题？
- 同类文件的命名风格是否一致？
- 多种命名风格并存是否造成查找时的"脑内转换"负担？

**可能的发现**：
- 有的文件有前缀，有的没有
- 有的用日期开头，有的用主题开头
- 查找同一类内容时需要尝试多种命名模式

**如果存在这些困扰** → 可以考虑制定轻量的命名约定

#### 维度3：信息密度

**观察角度**：
- index.md 打开后，需要滚动多久才能看到常用内容？
- 不同类型的信息（最新动态、配置说明、使用指南）是否混杂？
- 新加入的Agent能否快速理解记忆库结构？

**可能的发现**：
- index.md 很长，核心内容被淹没
- 维护说明和使用指南 buried 在大量文件列表中
- 每次更新 index 都要小心翼翼避免破坏其他部分

**如果存在这些困扰** → 可以考虑拆分 index 的结构

#### 维度4：演化痕迹

**观察角度**：
- 是否还保留着早期"试验性"的临时内容？
- 是否有内容已经过时，但没有归档？
- 当前的分类是否还符合近期的使用模式？

**可能的发现**：
- 某些目录/文件已不再使用，但还在显眼位置
- 分类逻辑和使用习惯已经变化，但结构未跟上
- "临时"内容已经长期存在，造成干扰

**如果存在这些困扰** → 需要整理和归档

---

### 第二阶段：决策（设计原则）

**目标**：基于诊断观察，运用决策原则，**自主设计方案**。

**不是"选择选项"，而是基于原则权衡后制定方案。**

#### 决策原则1：解决真实困扰

**核心问题**：这个决策要解决什么具体困扰？

**引导思考**：
- 如果不做改变，3个月后这个困扰会加剧吗？
- 是"审美上的不一致"，还是"实际使用中的摩擦"？

**警示**：不要为了"更规范"而规范——如果混乱没有造成查找困难，就不必强行统一。

#### 决策原则2：保持演化空间

**核心问题**：这个决策会给未来留下调整余地吗？

**引导思考**：
- 如果3个月后需求变了，这个方案容易调整吗？
- 是否引入了难以撤销的结构性改变？

**推荐做法**：
- 优先选择"容易调整"的方案（如约定优于配置）
- 避免过早固化结构（除非已经稳定使用很久）

#### 决策原则3：成本与收益平衡

**核心问题**：这个投入值得吗？

**引导思考**：
- 整理所需的时间 vs 未来节省的时间
- 结构复杂度 vs 实际内容数量

**参考标准**：
- **小记忆库**（<20文件）：保持扁平，命名清晰即可
- **中等记忆库**（20-50文件）：考虑轻量分类
- **大记忆库**（>50文件）：考虑结构拆分

#### 决策原则4：入口有效性

**核心问题**：这个决策会影响 AGENTS.md 中"启动时读取 index.md"的协议吗？

**约束条件**：
- 即使拆分结构，index.md 仍必须是唯一入口
- 任何结构变化都要在 index 中明确标记
- 新加入的Agent必须能通过 index 找到所有内容

---

#### 决策输出示例

```markdown
## 基于观察的决策

### 问题1：项目文件命名混乱
**是否真实困扰**：是，查找项目文件时需要尝试多种命名模式
**演化空间**：命名约定可以随时调整，风险低
**成本收益**：重命名10个文件约需5分钟，未来每次查找节省10秒
**启动协议影响**：无，index中只需更新链接

**决策**：制定轻量命名约定
- 约定内容：项目文件统一放 `projects/[项目名]/` 目录
- 命名模式：`[主题]_[可选日期].md`（简洁为主）
- 不强制：通用文档（方法论/经验）保持现有命名
- 可调整：此约定仅为当前参考，下次维护可重新评估

### 问题2：index.md 过长
**是否真实困扰**：是，维护指南被淹没在文件列表中
**演化空间**：拆分后仍可调整，config/目录内容可随时增减
**成本收益**：创建config/并移动内容约需10分钟，index变清爽
**启动协议影响**：需要在index顶部明确标记新结构

**决策**：拆分index结构
- 拆分方式：创建 `config/` 目录，放入维护指南和约定文档
- index顶部标记：使用表格形式展示"快速入口"
- 不拆分：文件清单保留在index中（仍需经常浏览）

### 问题3：projects/目录文件增多
**是否真实困扰**：暂时不严重（目前15个文件），但3个月后可能增至30+
**演化空间**：现在不拆，未来拆也可以，低优先级
**成本收益**：暂不投入，但需标记为"下次观察"
**启动协议影响**：无

**决策**：暂不处理
- 观察：下次维护时评估是否需要按子项目拆分
- 标记：在index中注明"projects/目录待观察"
```

---

### 第三阶段：执行（按需实施）

**目标**：根据决策，选择性地执行必要的整理操作。

**执行原则**：
- 一次只执行决策清单中标记"执行"的项目
- 不要超出决策范围（避免"顺便"修改）
- 每个操作后立即验证（如链接是否有效）
- 如遇意外问题，暂停并重新评估决策

#### 3.1 文件命名整理（如决策需要）

- 重命名文件，符合新约定
- 更新文件内的交叉引用（链接）
- 在 index.md 中更新链接

#### 3.2 文件位置整理（如决策需要）

- 创建必要的子目录
- 移动文件到合适位置
- 更新 index.md 中的路径引用

#### 3.3 Index结构整理（如决策需要）

- 创建 `config/` 子目录（如决策拆分）
- 移入配置和指南文档
- 在 index.md 顶部添加**重要标记**：

```markdown
## 快速入口
⚠️ **本项目使用多文件配置**

| 查看内容 | 位置 |
|---------|------|
| 最新更新 | [最新更新](#最新更新) |
| 完整文件清单 | [config/inventory.md](config/inventory.md) |
| 命名约定 | [config/convention.md](config/convention.md) |
| 维护指南 | [config/maintenance.md](config/maintenance.md) |
```

**重要**：即使拆分，index.md 仍必须是第一入口，通过标记让用户快速找到 config/

#### 3.4 归档整理（如决策需要）

- 识别过时内容（agent 判断以后不需要的内容）
- 如果整个文件都过时：移至 `archive/` 目录
- 如果文件部分过时：归档整个文件，提取有效部分编写新文件
- 在 index.md 中说明归档内容

---

### 第四阶段：验证（检查效果）

**目标**：确认问题已解决，入口有效，记录本次维护。

#### Index入口有效性验证（关键）

**模拟"新Agent视角"，重新加载 index.md：**

- [ ] **首次加载测试**：能否在**10秒内**理解记忆库结构？
- [ ] **重要标记可见性**：index 顶部的重要标记是否**足够显眼**？（位置、格式、符号 ⚠️）
- [ ] **关键入口清晰度**：最新更新、配置文档、维护指南是否**一目了然**？
- [ ] **自解释性**：能否仅凭 index 找到所有内容？
- [ ] **完整性**：是否存在 index 中**未提及**但记忆库中**存在**的内容？

**如果任何一项为否**，返回调整 index 的引导信息：
- 重要标记不够显眼 → 调整位置和格式（表格、符号、标题层级）
- 新 Agent 无法理解 → 添加"快速开始"或"如何使用本记忆库"指引
- 内容未提及 → 在 index 中补充链接或说明

**原因**：index.md 是启动协议的核心——如果 index 不能有效引导，整个记忆库就失效了。维护后的验证必须确保 index 入口有效。

#### 问题解决验证

- [ ] 之前查找困难的内容，现在是否容易找到？
- [ ] 新命名/新结构是否比旧的更清晰？
- [ ] index.md 是否加载更快/浏览更顺畅？
- [ ] 所有链接是否有效？
- [ ] 如果 3 个月后回看，这个决策还会觉得正确吗？

## 与记忆库其他部分的关系

- **在 index.md 中**：
  - 添加"维护历史"章节（或链接到独立记录文件）
  - 顶部标记当前配置结构（如已拆分）

- **约定文档**（可选）：
  - 将当前有效的规范写入约定文档
  - 命名示例：`经验_记忆库约定_v1.0.md`
  - 保留历史版本在 `archive/`

- **维护记录**：
  - 可独立成文档，也可作为 index.md 的章节
  - 重点是记录"为什么这样决策"
