# agent-prompt-design — 设计意图与维护参考

> 本文档是**给人读的**设计说明。Agent 请阅读 `SKILL.md`。

## 设计意图

### 为什么需要这个 Skill

Hermes 生态中已有 `prompt-engineering` 技能，但它面向的是**单次 LLM 调用**的提示词编写——角色扮演、Few-Shot、CoT、结构化输出等。而 Agent 系统提示词是一个不同的品类：它是**持久的行为宪法**，需要考虑跨多轮对话的一致性、工具调用协议的稳定性、安全边界的完整性、以及版本化运营。

本 Skill 填补了"Agent 系统提示词设计"这一垂直领域的方法论空白。它不是 `prompt-engineering` 的替代品，而是针对不同场景的互补工具。

### 为什么这样设计结构

1. **SKILL.md 极简** — 仅含内容索引 + 使用方法 + 能力边界。触发条件由 YAML `description` 承载，不在正文中重复。
2. **references/ 细粒度拆分** — 9 个独立文件，每文件聚焦一个主题。Agent 按需加载，避免一次性吞入全部内容造成注意力过载。
3. **templates/ 与 references/ 分离** — 模板是可直接复制使用的产出物，参考文档是方法论知识。两者的读写模式不同：模板 → 复制输出，参考 → 加载到上下文推理。
4. **渐进式加载** — SKILL.md 中的决策树帮助 Agent 快速定位需要的文件，而非线性通读全部。

### 关键架构决策

| 决策 | 理由 |
|------|------|
| 保留 RESEARCH.md | 作为溯源资料，当方法论需要深入引用时加载。不嵌入 SKILL.md 正文。 |
| 推理模型专题独立成文件 | 2026 年的推理模型策略变化是一个范式级别的转变（CoT 从推荐变为有害），值得独立文件而非埋在 content-writing 中。 |
| 安全独立成文件（safety.md） | 安全是 Agent 提示词设计的第一阶变量（arXiv:2603.25056），且内容量大，独立加载更合理。 |
| 反模式与诊断独立（anti-patterns.md） | 诊断是高频场景——Agent 出问题时用户首先需要的就是反模式排查。独立文件减少加载成本。 |

## 维护注意事项

- **新增 reference 文件时**，同步更新 SKILL.md 中的内容索引表和使用方法决策树。
- **修改模板时**，确保占位符 `[...]` 风格一致，注释标明使用场景和不适用的场景。
- **CHANGELOG.md 必须同步更新**，记录每次迭代的变更。
- **RESEARCH.md 尽量不改**——它是原始调研的快照。新的研究发现应补充到对应的 reference 文件中，而非追加到 RESEARCH。
- 如果某个 reference 文件超过 500 行，考虑进一步拆分为子主题文件。
