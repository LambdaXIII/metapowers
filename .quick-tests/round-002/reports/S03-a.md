## 第三部分：Persona 矩阵设计（模拟推演）

### 3.1 指令记录（L17-L27）

> 找 4-6 个典型用户场景。每个场景定义：人格 + 路径

设计原则：
- 覆盖决策树的**每条主要分支**
- 至少 1 个跨文件场景
- 至少 1 个诊断/修复场景
- 至少 1 个边界场景

**清晰度判断:** ✅ 原则明确，示例表（L83-L89）提供了格式模板。

### 3.2 根据模拟背景设计 Persona 矩阵

基于 `code-review-agent` 的模拟结构（SKILL.md 含 3 分支决策树：A=新项目审查, B=遗留代码审查, C=安全审查），设计 5 个 Persona：

| # | 场景 | 人格 | 决策树路径 | 预期文件链路 |
|---|------|------|-----------|-------------|
| P1 | 新项目审查 | 中级开发者，第一次使用代码审查Agent，想审查一个新写的 PR | 分支 A | SKILL.md → references/workflow.md → templates/review-checklist.md → templates/comment-template.md |
| P2 | 遗留代码审查 | 维护者，接手旧代码库，需要大规模审查 | 分支 B | SKILL.md → references/workflow.md → references/anti-patterns.md → templates/review-checklist.md |
| P3 | 上架前安全审查 | DevOps 工程师，上线前必须做安全审计 | 分支 C | SKILL.md → references/safety.md → references/workflow.md → templates/review-checklist.md |
| P4 | 反模式诊断（跨文件） | 高级开发者，发现代码出现特定问题模式，想定位修复 | 分支 B（偏离） | SKILL.md → references/anti-patterns.md → (需跳转到 workflow.md 或 safety.md 确认上下文) |
| P5 | 从侧门进入（边界） | 新用户，直接从搜索结果打开 workflow.md，没看 SKILL.md | 无分支 | references/workflow.md（无入口上下文） |

### 3.3 设计阶段的发现

| 发现 | 类型 | 说明 |
|------|------|------|
| 3 条分支但只有 1 个 workflow.md | **内容深度风险** | 分支 A、B、C 的 workflow 是否完全相同？如果相同，为什么要分成3条分支？如果不同，workflow.md 需要分段或分支指示。 |
| safety.md 只在分支 C 引用 | **内容重叠排查** | 安全审查的 checklist 是放在 safety.md 还是 review-checklist.md？如果两者都包含 checklist，分工不清。 |
| anti-patterns.md 的入口从分支 B 进入 | **交叉引用断裂风险** | 分支 A 或 C 的读者遇到反模式时，文档是否告诉"跳到 anti-patterns.md"？ |

### 场景 P1：新项目审查（分支 A）

```
SKILL.md（入口）
  → 读到决策树，选择分支 A "新项目审查"
  → 指引：加载 references/workflow.md
  → 阅读 workflow.md → 获得审查流程
  → 指引：使用 templates/review-checklist.md 和 templates/comment-template.md
```

**推演步骤:**

| 步骤 | 模拟心理 | 停顿/困惑 |
|------|----------|-----------|
| 1. 打开 SKILL.md | "这是入口，看到了3个分支，我选A" | ✅ 清晰 |
| 2. 决策树指引→workflow.md | "它说'新项目审查参考 workflow.md'，我打开了" | ⚠️ SKILL.md 是否存在对 workflow.md 中与新项目相关的具体 section 的锚点指引？如果没有，用户需要全文搜索"新项目" |
| 3. 阅读 workflow.md | "这是审查的5个步骤..." | ❓ 步骤中的"新项目特有环节"（如首次引入检查）和"通用环节"是否明确标注？ |
| 4. workflow.md→review-checklist.md | "用这个清单做实际审查" | ❓ checklist 是否区分 A/B/C 分支？还是通用？ |

**关键判断:**
- SKILL.md 中的分支选择后，是否需要区分 workflow.md 中的 section？如果 workflow.md 是通用文件，分支 A 的读者会被迫阅读不相关的 B/C 内容。
- checklist 和 comment-template 的输入输出关系不清晰——是先填 checklist 再写 comment，还是 checklist = comment？

## 第九部分：总结

### 核心结论

1. **推演中止（针对被测技能）**：code-review-agent Skill 的文件在磁盘上不存在，无法进行实际的逐文件内容级推演。以上推演基于模拟背景描述的目录结构和文件用途推断，属于"结构级预推演"而非"内容级全量推演"。

2. **侧门入口的可行性**：从 `skill-validation.md` 直接进入设计多场景验证是 **部分可行但条件不足**：
   - ✅ Persona 设计原则清晰，可独立完成
   - ⚠️ 逐场景推演需要假设文件内容（因为文件不存在）
   - ❌ 没有 SKILL.md 的决策树原文，分支的精确描述未知

3. **skill-validation.md 自身的可用性**：作为验证方法论文档：
   - ✅ 结构完整，步骤清晰
   - ✅ 边界检查设计全面
   - ⚠️ 依赖读者了解"标准 L3"背景
   - ❌ 缺少归属声明，侧门读者容易迷失
   - ❌ 示例仅展示了"发现了12个缺陷"但没有展示完整推演链路，降低了方法第二步的参考价值

### 建议
