## 复杂场景

### S01: 提示词推演验证
| 字段 | 值 |
|------|-----|
| **需求明确度** | 抽象 |
| **目标类型** | 诊断 |
| **背景上下文** | 半知（有提示词文本，用过推演概念但不熟练） |
| **入口** | 正门 |
| **目标** | 用推演方法验证一份系统提示词对Agent的效果 |

**模拟背景：**
- 用户刚写了一份代码审查Agent的系统提示词（约500字）
- 提示词定义了Agent角色（代码审查者）、工作流（读diff→逐文件检查→写评论）、禁止事项（不修改代码、不执行重构）
- 用户想验证：Agent首次读这份提示词时会不会误解或遗漏关键约束
- 用户对推演方法只有模糊概念（听说过"清空→逐帧跟→边界刨→打分"），没实际操作过

**预期效果：**
- 子代理从SKILL.md入口，定位到「提示词/指令」载体类型
- 选择对应打分维度（因果方向清晰度、违规模式覆盖、自检可执行性、后果说服力、边界覆盖）
- 基于模拟背景逐帧推演提示词，至少发现1个结构性问题
- baseline不加载技能，直接凭经验评估

**测试单元：** S01-a（加载技能）+ S01-b（baseline）

### S02: 多文档L3全流程推演
| 字段 | 值 |
|------|-----|
| **需求明确度** | 精确 |
| **目标类型** | 诊断 |
| **背景上下文** | 全知（有完整知识库，知道L1/L3概念，需要全流程推演指导） |
| **入口** | 正门 |
| **目标** | 为一个多文档知识库执行L3全流程跨文档推演 |

**模拟背景：**
- 有一个"车间知识库"项目：5个车间文档（casting, editing, mixing, mastering, delivery）+ 1个通用参考文档
- 每个文档约1000-2000字，描述该车间的工作流程
- 车间之间有上下游关系（casting→editing→mixing→mastering→delivery）
- 用户在casting文档中提到"见通用参考文档的音频格式规范"，但从未验证过这个引用是否断裂
- 用户想按推演的L3方法，从casting开始逐车间走，检查模块流转、术语统一、引用完整性

**预期效果：**
- 子代理定位到 L3 推演（doc-guide-validation.md 或 document-qa-pattern.md）
- 用 L3 的七维度打分体系
- 至少发现"跨模块引用""术语一致性"等结构性问题
- baseline无技能引导，可能只做简单审阅

**测试单元：** S02-a（加载技能）+ S02-b（baseline）

### S03: Skill多场景验证（侧门进入）
| 字段 | 值 |
|------|-----|
| **需求明确度** | 精确 |
| **目标类型** | 诊断 |
| **背景上下文** | 半知（有Skill目录，知道推演可用于Skill验证，跳过SKILL.md直接进入skill-validation.md） |
| **入口** | 侧门 — references/skill-validation.md |
| **目标** | 为一个Skill做多场景验证推演 |

**模拟背景：**
- 有一个"code-review-agent" Skill：SKILL.md（入口+决策树）+ 3个reference文件（workflow.md, anti-patterns.md, safety.md）+ 2个template文件
- Skill的设计者刚完成重构，想用推演的多场景Skill验证方法检查完整性
- 决策树有3条分支：新项目代码审查(A)、遗留代码审查(B)、安全审查(C)
- 用户从搜索直接打开了skill-validation.md，没读过SKILL.md

**预期效果：**
- 子代理从skill-validation.md独立理解Skill验证推演方法
- 基于模拟背景设计3个persona场景
- 发现交叉引用断裂、文件归属缺失等问题
- baseline无技能，凭经验做验证

**测试单元：** S03-a（加载技能）+ S03-b（baseline）
