# MOC (Map of Content) / LYT (Linking Your Thinking)

> 用索引笔记（MOC）作为导航枢纽，替代深层文件夹。笔记存放在扁平结构中，MOC 收集相关笔记的链接形成虚拟目录。Home note + 约 10 个主 MOC 构成"流式框架"。

**创建者**: Nick Milo（LYT 框架）  
**核心分类维度**: 导航优先（MOC 笔记作为分类代理，不移动文件本身）

## 核心机制

- **Home Note**：笔记库的根节点，链接到所有主 MOC
- **MOC（Map of Content）**：一组相关笔记的索引，功能类似书籍目录
- **Fluid Frameworks**：分类可以随时间变化，不需要重新组织文件

**MOC 解决了什么问题**：一个笔记只能在一个文件夹中。如果一个概念跨越多个领域，MOC 让它可以出现在多个"虚拟目录"中而不移动文件。

## 结构示例

```
📄 Home.md                    ← 根索引：链接到所有主 MOC
📄 MOC - Creativity.md        ← 创造力相关笔记索引
📄 MOC - Productivity.md      ← 生产力相关笔记索引
📄 MOC - Health.md            ← 健康相关笔记索引
📁 inbox/                     ← 新笔记缓冲
📁 notes/                     ← 所有笔记，扁平存储
📁 daily/                     ← 日记
📁 references/                ← 外部资料
📁 attachments/
📁 templates/
```

## LYT 的 8 个声称好处

1. 提供可靠性（你知道在哪找）
2. 灵活性（MOC 可随时拆分、合并）
3. 触发大脑（重新整理 MOC 就是重新思考）
4. 增强检索（多路径找到同一条笔记）
5. 减轻过载（每张 MOC 控制在可管理范围内）
6. 鼓励关系性思考（整理 MOC 时发现新关联）
7. 提供预构建上下文（MOC 为笔记提供分类归属）
8. 改善记忆（空间导航 + 概念导航双重编码）

## 适用场景

- **大型笔记库**（1000+ 条）：文件夹开始不够灵活时
- **跨域概念**：一个想法属于多个领域，不想重复复制
- **分类不确定**：你还没想好固定的分类方案，MOC 不锁定你
- **渐进式精炼**：你可以先写笔记，后来再建立关联

## 不适用场景

- **需要严格归属**：有些系统要求每条笔记有一个且只有一个位置
- **追求简洁**：MOC 系统需要维护索引，不是每个人都愿意
- **团队共享**：没有固定目录结构的系统对协作有门槛

## 改编要点

- MOC 和文件夹不是二选一——扁平文件夹 + MOC 是常见组合
- MOC 的数量控制在 10 个左右，超过这个数需要 MOC of MOCs
- MOC 可以看作是 manual index（手动索引），和 automatic search（自动搜索）互补
- 如果你已经有一组稳定的顶层文件夹，MOC 可以作为"跨域索引"补充而非替代
