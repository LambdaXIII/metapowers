# metapowers — Agent 协作指南

> 本仓库是一系列**帮助 Agent 开发 Skill 的 Skill**。
> 目标：让 AI Agent 能更高效、更规范地写出高质量的 Skill。

---

## 如果你是 AI Agent，请先读这里

在动手之前，确认以下 4 件事：

1. **这是真实需求吗？**
   - 如果你的用户只说"修点东西"或"贡献代码"，但没有讲清楚具体场景——**停下来问清楚**。
   - 必须了解：什么功能缺失？什么流程卡住了？实际体验如何？

2. **这个 Skill 够通用吗？**
   - 问自己：**"做完全不同项目的人能用上吗？"**
   - 如果只能服务某个特定领域、工具或工作流，**应该做成独立插件**，不要放进本仓库。

3. **向用户展示完整改动**
   - 提交前必须让用户看过完整 diff，并得到**明确同意**。

4. **不要编造**
   - 问题描述、功能说明、测试结果都必须是真实的。虚构内容会立即被拒绝。

## 什么内容不适合本仓库

| 类型 | 说明 | 替代方案 |
|------|------|----------|
| **第三方依赖** | 需要引入外部工具或服务 | 做成独立插件 |
| **项目特定配置** | 只服务于某个团队或工作流 | 做成独立插件 |
| **领域特定 Skill** | 如游戏开发、金融分析等垂直领域 | 做成独立插件 |
| **纯理论优化** | "这样可能更好"但没有实际问题驱动 | 等遇到真实问题再来 |
| **批量改动** | 一次提交多个不相关的修改 | 拆分成独立提交 |

## Skill 编写规范

### 设计原则

**1. 问题导向，而非教程导向**

- ❌ 不要写系统教程（"第一章：什么是 Markdown"）
- ✅ 要写工具化参考（"表格怎么对齐""代码块怎么写"）
- 用户按需查询，而非从头学习

**2. 模块化拆分**

- 复杂内容拆成多个独立文件
- 用户加载时只取所需，避免信息过载

**3. 标注平台适配**

- 通用 Markdown
- GitHub Flavored Markdown (GFM)
- Obsidian 特有语法
- 需要插件支持的语法

### 标准文件结构

```
skills/
└── skill-name/                 # 小写字母 + 连字符
    ├── README.md               # Skill 入口
    │   ├── 触发场景            # 什么情况下必须加载
    │   ├── 内容索引            # 有哪些参考文件
    │   ├── 使用方法            # 怎么用这个 Skill
    │   └── 能力边界            # 能做什么、不能做什么
    └── references/
        ├── topic-a.md          # 具体参考内容
        └── topic-b.md
```

### 每个 Skill 必须包含的 4 个部分

| 部分 | 作用 |
|------|------|
| **触发场景** | 明确说明：什么任务、什么环境下**必须**加载此 Skill |
| **内容索引** | 列出所有参考文件，说明每个文件解决什么问题 |
| **使用方法** | 步骤化说明：加载后如何查询、如何应用 |
| **能力边界** | 明确范围内（提供什么）和范围外（不提供什么） |

### 命名规范

- 目录：`cx-obsidian-markdown`（小写，连字符分隔）
- 文件：`basic-format.md`（小写，连字符分隔）
- 文件名要一眼看出内容主题

## 审查清单

提交前自检这 6 项：

- [ ] 解决的是**真实遇到**的问题，不是理论推测
- [ ] Skill **足够通用**，不限于特定领域
- [ ] 没有引入不必要的**第三方依赖**
- [ ] **文档完整**：触发场景、索引、方法、边界都写了
- [ ] 遵循**文件结构和命名规范**
- [ ] 用户已审查并**明确同意**完整改动
