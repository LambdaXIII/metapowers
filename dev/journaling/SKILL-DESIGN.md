# journaling 设计文档

> 本文件是 journaling 技能的**设计锚定文档**——记录设计方向、原则理由和关键决策。它独立于技能文件体系，不被运行时加载，仅作为开发者维护和演进技能的参考。

---

## 设计初衷

Agent 的长期记忆需要一个结构化的笔记本系统——不是简单的文件堆，而是一个有 INDEX.md、有条目规范、有维护周期的知识管理体系。

这个技能解决的问题：

- **"写了不读"**：经验记下来了，但下次遇到相关任务时不会先查阅
- **"写了找不到"**：信息存在但无法在需要时快速定位
- **"写了变噪音"**：INDEX.md 膨胀，信号被淹没在日志中
- **"写了不维护"**：条目过期、标签蔓延、结构腐化，但无人清理

Journal 服务于 Agent 自身——用户不审批 journal 操作。Agent 自主维护（归档、合并、删除、标签修复），可以征求建议但不请求许可。这不是协作偏好，是设计要求：用户无法判断 journal 操作的正确性，因此技能必须让 Agent 自给自足。

---

## 核心设计决策

### 1. 渐进式披露——SKILL.md 作为路由

SKILL.md 只保留路由表 + 核心约束 + 操作规则。所有机制细节在 references/ 中按需加载。

**原因**：上下文成本。加载技能写一条笔记和加载技能做全量维护的上下文需求完全不同。路由表让两种场景的成本分离——写笔记只需加载 protocal-write.md 和 spec-note.md，做维护才加载 protocal-maintenance.md。

### 2. 五阶段维护协议——扫描、规则审查、重组、质量检验、收尾

Phase 0 纯扫描（只收集数据不做判断），Phase 1 规则审查（基于扫描结果决定分类规则、tags 是否需要修改），Phase 2 重组（基于确认后的规范执行结构化调整），Phase 3 质量检验，Phase 4 收尾记录。

**关键结构**：扫描与决策之间有一道明确边界——Phase 0 的输出是"当前状态的数据"，Phase 1 才回答"这个状态是否需要改变规范"。这防止了"扫描到一半就开始动手"的倾向。

**规则 vs 条目分离**：规则审查（Phase 1）和条目重组（Phase 2）是独立的。Phase 1 决定分类规则、TAGS.md、CLASSIFICATION.md 是否应该修改。Phase 2 才基于确认后的规则移动条目、更新 frontmatter。如果规则不变，Phase 2 仍然可以执行条目级别的归档和清理。这避免了一个常见错误——在规则尚未确认的情况下就开始搬文件。

### 3. 摘要锚定——scope boundary 而非内容总结

每条笔记的 summary 是其 scope 的边界定义，不是内容的概括。

**三个检查点**：创建时写摘要定义范围→修改前读摘要判断"新内容还在范围内吗"→修改后验证摘要是否仍准确。这让"update vs create"的决策从"我需要想清楚"变成"读摘要→答案自明"。

### 4. Tags 自管理——TAGS.md + 设计方法论文档

Tags 由每个 journal 通过本地的 `TAGS.md` 自管理，技能只提供种子（6 个基础 tag）和设计方法论（`references/design-tags.md`）。

**关键转变**：上一版设计使用技能强制的 4 分类 21 tag 体系（activity/domain/meta/project）。新设计仅初始化 6 个种子 tag（lesson、research、knowledge、documentation、journal、plan），后续 tag 增长遵循命名规则和注册流程。维度分组（activity/domain 等）是可选的，仅在 tag 超过 ~15 个时按需引入。

**先注册后使用**：虽然种子集很小，但新增 tag 必须在 TAGS.md 中注册后才能使用。自由格式标签仍然会退化为噪音，只是约束从"技能强制分类"变为"技能提供方法论 + journal 自执行"。

**规范验证**：维护时 Phase 0 收集 tag 使用数据，Phase 1 审查 TAGS.md 及其规则是否仍然适应当前内容。Tag Worthiness Criteria 决定一个 tag 是否值得保留。

### 5. 读写非对称——读不加载，写才加载

读取 journal（浏览 INDEX.md、查阅条目、搜索内容）**不需要**加载 journaling 技能。只有在写入或维护时才需要加载。

**原因**：journal 是 Agent 每轮启动都要读取的——INDEX.md 是启动协议的第一步。如果每次读 journal 都要加载多个 reference 文件，上下文成本不可接受。读写分离让"读取"保持零成本（INDEX.md 本身就是自描述的状态封面），把技能的复杂性全部推到"写入"侧——写入时才需要规范（格式、目录、标签、摘要锚定），维护时才需要协议（五阶段流程）。

**设计自洽**：INDEX.md 的协议声明行明确标注"读此文件不需要加载 skill · 写入或维护时必须加载"。这和渐进式披露形成配合——读取是零成本高频操作，写入是中等成本低频操作，维护是高成本极低频操作，三者的上下文成本逐层递增。

### 6. Journal 服务于 Agent 而非用户

用户不审批 journal 操作。Agent 自主维护（归档、合并、删除、标签修复），可以征求建议但不请求许可。

**设计约束**：这不是协作偏好，是设计要求——用户无法判断 journal 操作的正确性，因此技能必须让 Agent 自给自足。

### 7. 初始化协议——三阶段明确分工

初始化协议（`references/protocal-init.md`）将原七步流程重构为三阶段：

- **Phase 1（确定位置）**：检查上下文中是否已知 journal-root，未知则询问用户。位置是锚点——不确定则后续无依托。
- **Phase 2（初始化内容）**：逐项检查 CLASSIFICATION.md、TAGS.md、INDEX.md。存在则判断类型、检查格式——合规不动、不合规则保留内容重写格式。同名非同类则改名移入 inbox 后创建。
- **Phase 3（设计发现合约）**：向用户说明重要性、原则和方案，由用户决定是否建立。禁止自行执行。

协议不再内嵌初始模板——模板独立于 `templates/seed/` 目录，协议引用但不包含。`seed/` 子目录强调版本身份，但文件名不加前缀——复制使用时无需改名。
种子目录不再在初始化时创建——由写入操作按需催生，CLASSIFICATION.md 中已定义用途。

### 8. Protocol 操作模式分化——远征型 vs 内嵌型

Protocol 在设计上天然分成两类，核心区别在操作频率和认知上下文。

**远征型（Expedition）**：任务独立、步骤多、边界清晰、cognition budget 充足。每一步需要 why（agent 有空理解原理并处理边缘情况）；强边界约束（"不执行"、"先确认再动手"）；类型识别先行。代表：protocal-init.md、protocal-maintenance.md。

**内嵌型（Embedded）**：嵌入主线工作，高频率、低延迟要求。仅开头提供定位说明（"这个协议是干嘛的"），不提供逐步骤 why；不打断执行节奏；出口质量由维护协议兜底。代表：protocal-write.md、protocal-import.md。

**原因**：两种模式的核心约束不同。用远征型的设计范式套内嵌型不是不够好——是方向错了。设计 protocol 时必须先判断操作场景再落笔。

### 9. 目标前置——每个 protocol 开头应有完成状态描述

每个 protocol 开头应说明"做完这个想达到什么状态"。作用：提供操作终点（知道什么时候算完成）、提供判断依据（以是否朝向目标做取舍）、减少迷失感（序列执行过半时可能忘记为什么做这些步骤）。

**原因**：之前的 write 和 init 协议缺少目标前置，agent 读到中间步骤时无法判断"我还在正确的方向吗"。远征型协议展开为完整节，内嵌型仅保留 2-3 行定位说明。

### 10. 理解验证 ≠ 质量检查

| | 理解验证 | 质量检查 |
|---|---------|---------|
| 时机 | 决策前 | 交付前 |
| 问题 | "我读对了吗" | "我做得对吗" |
| 内容 | 确认数据解读无误 | 确认格式合规、链接完整 |
| 代表 | Phase 1 入口重读 Phase 0 快照 | Self-Check 四条、Phase 3 quality check |

**原因**：只有交付前检查没有决策前验证——如果决策时数据解读有误，质量检查依然会通过（输出格式正确，但方向错了）。

### 11. 加法胜于减法——可逆操作设计模式

任何涉及重组/移动的操作，设计模式应为"写新位置→验证→清理旧位置"的三段加法，而非追踪回退路径。

**原因**：Agent 的执行记录不可靠，依赖 agent 重建状态的任何机制都脆弱。加法模式在中断时天然安全（旧位置还在），终止状态可读（检查文件系统即知进度）。

**具体应用**：maintenance Phase 2 的"新位置先就位，原位置后清理"原则；cascade rename 的"先枚举全部匹配，再从最深到最浅逐一更新"。

### 12. 模板分离的时机

内嵌代码块是否拆到 templates/ 的判断标准：

- **拆**：多 protocol 引用同一内容 | 多种执行方式（cp / write / 手写）
- **留**：单协议引用、单一操作方式的示例或参考

**原因**：protocal-write.md 的 YAML 代码块是单协议引用的示例，留在原地。CLASSIFICATION/TAGS/INDEX 种子模板被 protocal-init.md 引用、可能在不同环境中通过 cp 或 write 执行——拆到了 templates/seed/。

## 措辞风格约定

- **第一人称**："我记的，我用的"——journal 是 Agent 自己的笔记本，不是外部要求
- **中英混合**：结构性内容（frontmatter、标签）用英文，解释性内容和操作规则用中文
- **指令式而非叙事式**：操作规则用祈使句（"验证..."、"不要..."），不用"曾经犯了X错所以应该Y"的叙事结构。教训吸收为指令，案例留在 journal 中。

---

## 维护注意事项

- **操作规则是活文档**：SKILL.md 的 Operating Rules 在遭遇新的操作失败时追加。新增规则用指令式措辞，不夹带历史溯源——历史案例属于 journal 经验条目，不属于技能规范。
- **SKILL.md 的路由表和 Linked Files 要同步**：新增或移除 reference 文件时，两个位置都要更新。
- **泛化与脱敏**：本技能从特定运行环境迁移而来。示例中使用"项目 A/B"等通用占位符代替真实项目名。journal 路径用 `<journal-root>` 参数化。维护时保持这种通用性——不要回填特定项目名或硬编码路径。
