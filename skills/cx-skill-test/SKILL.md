# cx-skill-test：技能快速测试方法

> 本技能提供结构化的快速测试流程，帮助验证技能的实际工作输出是否符合预期。

---

## 一、触发决策规则

### 应该触发

- 用户明确要求测试某技能
- 用户询问"这个技能能不能正常工作"
- 用户要求验证技能效果
- 用户说"测试一下这个技能"

### 不应触发

- 用户只是询问技能的用法（应直接回答）
- 用户要求创建技能（应使用 skill-creator）
- 用户没有明确测试意图
- 用户只是讨论测试方法论（不是要执行测试）

### 与 skill-creator 的选择

| 场景 | 选择 | 理由 |
|------|------|------|
| 开发迭代、快速验证 | cx-skill-test | 快速、轻量 |
| 正式发布前、需要量化数据 | skill-creator | 完整、量化 |

---

## 二、边界与约束

### 只读

- 技能源文件（skills/）
- 测试素材（test_subjects/）
- 项目文件
- eval_metadata.json（测试 SubAgent 不可读取）

### 可写

- 仅用户指定的输出目录

### 禁止

- 修改项目文件
- 安装技能到运行时环境
- 调用记忆库技能
- 修改被测试技能的源文件

---

## 三、核心定位

### 解决什么问题？

在技能开发过程中，需要快速验证技能的实际工作输出是否符合预期。本技能提供：

- **测试方案框架**：解决"不知道该测什么"的问题
- **输入输出分离**：解决"测试指令暴露答案"的问题
- **结构化测试指令**：解决"SubAgent 执行不稳定"的问题
- **元数据记录**：解决"无法复现和对比"的问题
- **客观评估依据**：解决"纯靠主观判断"的问题

### 核心原则

- **快速**：单次验证，快速迭代
- **隔离**：测试阶段不安装到运行时环境，使用独立 SubAgent
- **真实**：使用真实场景，不虚构用例
- **盲测**：测试 SubAgent 不知道预期效果，只执行任务
- **可复现**：通过 eval_metadata.json 和 inputs 目录记录测试信息

---

## 四、测试流程

测试流程分四个阶段：

```
阶段一：制定测试方案（含场景选择和 eval_metadata）
↓
阶段二：征求用户确认
↓
阶段三：执行测试（每项测试使用独立 SubAgent）
↓
阶段四：评估结果
```

### 阶段一：制定测试方案

**操作步骤**：

1. **确认被测试技能**
   - 询问用户要测试哪个技能
   - 读取技能的 SKILL.md 了解其功能

2. **准备测试场景**
   - 使用发散-收敛思维发现场景（见"场景选择与准备"）
   - 筛选 3-5 个最有价值的场景
   - 为每个场景生成 eval_metadata.json

3. **询问基线对比**
   ```
   是否需要启用基线对比？（Y/N）
   - Y: 启用基线，将分别执行 with-skill 和基线模式
   - N: 仅执行 with-skill 模式（默认）
   ```

4. **询问测试目录位置**
   - 默认：`test_space/<skill-name>-test/`
   - 用户可指定其他位置

5. **创建测试目录**
   - 为每个测试用例创建 `case-XX-<简要描述>/` 子目录
   - 在每个用例目录下创建 `eval_metadata.json`
   - 创建 `inputs/` 目录并写入 `prompt.md`

### 阶段二：征求用户确认

**展示内容**：
- 测试场景列表（场景名称、类型、数量）
- 是否启用基线对比
- 输出目录位置

**确认话术**：
```
测试方案已准备完成：

- 被测试技能：[skill-name]
- 测试场景：[N] 个
  - [场景1名称]（核心）
  - [场景2名称]（扩展）
  - [场景3名称]（边界）
- 基线对比：[是/否]
- 输出目录：[test-dir]/test-01/

是否开始执行测试？
```

### 阶段三：执行测试

**准备工作**（主 Agent 执行）：

1. 为每个用例创建 `inputs/` 目录
2. 创建 `inputs/prompt.md`，记录用户 prompt
3. 如有其他输入文件，拷贝或记录到 `inputs/` 目录

**执行方式**：
- 每个测试用例使用独立的 SubAgent
- 根据技能特点选择合适的 SubAgent（见"SubAgent 选择规则"）

**测试指令要素**：

```markdown
## 测试任务

### 被测试技能
- 技能路径：[skill-path]
- 技能描述：[skill-description]

### 测试场景
- 用户 prompt：见 inputs/prompt.md

### 执行要求
1. 加载技能：[skill-path]/SKILL.md
2. 按技能指引响应用户 prompt
3. 输出响应结果到：[output-path]/outputs/result.md

### 可写目录
- [output-path]/outputs/

### 注意事项
- 不要修改技能源文件
- 不要修改项目文件
- 不要读取 eval_metadata.json（评估阶段才使用）
- 只输出任务结果，不输出过程记录
```

**可选过程记录**：

当主 Agent 判断需要过程记录时，在测试指令中追加：

```markdown
### 额外要求
- 输出过程记录到：[output-path]/outputs/test-record.md
- 记录内容应包含：执行步骤、关键决策、使用的参考文档
```

**触发条件**（主 Agent 判断）：
- 技能首次测试，需要了解执行细节
- 上次测试失败，需要诊断问题
- 用户明确要求详细记录

### 阶段四：评估结果

**评估流程**：

1. **主 Agent 整理评估指令**
   - 读取 eval_metadata.json 获取 expected
   - 读取 outputs/result.md 获取实际输出
   - 如有基线，读取两个目录的 result.md

2. **发送评估指令给 SubAgent（general）**

3. **SubAgent（general）执行评估**
   - 读取 outputs/result.md
   - 如有基线，对比两个结果
   - 对照 expected 判断是否符合
   - 输出评估报告

4. **主 Agent 汇总判断**
   - 收集所有测试用例的评估报告
   - 向用户报告测试结果

**评估指令要素**：

```markdown
## 评估任务

### 测试用例信息
- 用例名称：[case_name]
- 用例目录：[case-path]

### 评估依据
- 预期效果：见 eval_metadata.json 的 expected 字段
- 实际输出：见 outputs/result.md

### 执行要求
1. 读取 eval_metadata.json 获取 expected
2. 读取 outputs/result.md 获取实际输出
3. 对照 expected 判断是否符合
4. 输出评估报告到：[case-path]/assessment-report.md

### 可写目录
- [case-path]/

### 注意事项
- 评估必须基于 outputs/result.md 中的实际输出
- 如有基线对比，需同时读取两个目录的 result.md
- 评估结论必须明确：通过/失败
- 如有部分符合，需说明具体差异
```

## 五、两种指令

本技能使用两种指令，分别对应不同阶段：

| 指令 | 用途 | 执行者 | 包含内容 | 不包含内容 |
|------|------|--------|----------|------------|
| **测试指令** | 让 SubAgent 执行测试任务 | SubAgent（根据技能特点选择） | 技能路径、prompt、输出目录 | expected、eval_metadata |
| **评估指令** | 让 SubAgent 评估测试结果 | SubAgent（**统一使用 general**） | eval_metadata 路径、outputs 路径 | - |

### SubAgent 选择规则

| 任务类型 | SubAgent | 理由 |
|----------|----------|------|
| 测试执行 | 根据技能特点选择 | 匹配技能领域，提高执行质量 |
| 结果评估 | general | 稳定可靠，客观评判 |

**测试执行 SubAgent 选择建议**：

| 技能类型 | 建议 SubAgent | 说明 |
|----------|---------------|------|
| 通用技能 | general | 适用于大多数场景 |
| 代码相关技能 | code 或 general | 需要代码理解能力 |
| 文档相关技能 | general | 通用处理即可 |

## 六、场景选择与准备

### 场景类型

| 类型 | 说明 | 优先级 |
|------|------|--------|
| **核心场景** | 技能的核心用途、最常见请求 | 必测 |
| **扩展场景** | 核心场景的变化/组合 | 建议测 |
| **边界场景** | 输入在边缘的情况 | 按需 |
| **失败场景** | 可能导致失效的情况 | 按需 |

### 场景准备操作步骤（5步法）

| 步骤 | 操作 | 输出 |
|------|------|------|
| 1. 技能分析与正向思考 | 分析能力结构，列出典型使用场景 | 场景清单（草稿） |
| 2. 反向思考 | 列出可能导致技能失效的情况 | 场景清单（草稿） |
| 3. 整合 | 筛选出最有价值的场景（3-5个） | 场景清单（精简） |
| 4. 分类 | 确定每个场景的类型 | 场景清单（带分类） |
| 5. 整理 | 为每个场景生成 eval_metadata.json | eval_metadata 文件 |

### 步骤 1 详解：技能分析与正向思考

**技能分析**（融入正向思考）：

在列出典型场景之前，先快速分析技能结构：
- 能力有哪些？（确保每个能力都有测试）
- 执行模式有哪些？（确保每种模式都有测试）
- 有什么依赖关系或决策规则？（考虑是否需要测试）

**分析方法**：
1. 读取被测试技能的 SKILL.md
2. 查找"能力"、"功能"、"执行模式"、"决策规则"等章节
3. 在正向思考时，确保覆盖这些维度

**示例**：
- 技能有 3 种能力 → 正向思考时列出：完整流程、单项能力1、单项能力2、单项能力3
- 技能有 2 种执行模式 → 正向思考时列出：模式 A 场景、模式 B 场景

### 发散-收敛思维方法

**发散期**：多提问题，把可能的场景都列出来

**收敛期**：筛选，哪些场景值得测

### 两个思考维度

| 维度 | 核心问题 |
|------|----------|
| **正向** | 技能的典型使用场景是什么？什么情况下会用到？ |
| **反向** | 什么会让技能失效或出错？什么输入会破坏预期？ |

### 扩展测试范围

当用户要求更全面的测试覆盖时：
- 用户说"测试要全面一些"
- 用户说"扩展测试范围"
- 用户说"测试所有能力/模式"

**行动**：读取 `references/test-scenario-guide.md` 的"扩展方法"章节，获取能力覆盖、执行模式覆盖、依赖补齐测试、决策规则测试的详细方法。

> 💡 **提示**：场景准备的目标是"启发思路"，不是"穷举场景"。最终筛选 3-5 个最有价值的场景进行测试即可。

详细方法见 `references/test-scenario-guide.md`。

## 七、eval_metadata 规范

### 格式要求

```json
{
  "case_name": "场景名称",
  "prompt": "用户实际会说的 prompt",
  "expected": "预期效果（描述性）",
  "baseline": false
}
```

### 字段说明

- `case_name`：测试用例名称（简洁描述场景）
- `prompt`：用户实际会说的 prompt（真实场景）
- `expected`：预期效果（描述性，用于评估）
- `baseline`：是否启用基线对比（true/false）

### expected 编写原则

**使用可验证的描述性语言，避免主观形容词**。

**编写模板**：
```
"输出必须包含 [核心要求]，满足 [质量标准]"
```

**示例对比**：

| ❌ 错误示例 | ✅ 正确示例 |
|------------|------------|
| "输出结果很好" | "输出包含字段A、B、C，字段A和字段B必须非空" |
| "内容完整" | "文件包含标题、日期、作者、正文四个部分，每个部分格式正确" |
| "格式正确" | "JSON 格式有效，包含 case_name、prompt、expected、baseline 四个字段" |

**编写技巧**：
1. **指定核心要求**：明确指出输出必须包含哪些内容
2. **定义质量标准**：说明内容必须满足什么条件
3. **使用可验证语言**：避免"好"、"完整"、"正确"等主观形容词
4. **提供判断依据**：让评估 SubAgent 能明确判断是否符合预期

详细指南见 `references/eval-metadata-guide.md`。

## 八、基线对比

基线对比是**可选模块**，用于对比"有技能"和"无技能"的输出差异。

### 两种模式

| 模式 | 说明 |
|------|------|
| **with-skill** | 加载 SKILL.md 和 references，执行技能功能 |
| **基线** | 完全不加载 skill，执行相同任务 |

### 触发方式

在阶段一（制定测试方案）时，向用户询问：
```
是否需要启用基线对比？（Y/N）
- Y: 启用基线，将分别执行 with-skill 和基线模式
- N: 仅执行 with-skill 模式（默认）
```

### 目录结构

启用基线后，测试用例目录结构：

```
case-01-<简要描述>/
├── eval_metadata.json
├── inputs/
│   └── prompt.md
├── with-skill/
│   └── outputs/
│       └── result.md
└── without-skill/
    └── outputs/
        └── result.md
```

## 九、评估流程

### 评估依据原则

| 情况 | 评估依据 | 具体要求 |
|------|----------|----------|
| 技能生成文件 | 评估基于 outputs/result.md | 读取 result.md 内容 |
| 有基线对比 | 同时评估两个结果 | 对比 with-skill 和 without-skill 的差异 |

### 评估报告格式

```markdown
## 评估报告

### 测试用例：[case_name]
- **Prompt**：[prompt]
- **Expected**：[expected]
- **基线**：[是否启用]

### 评估结果
- [是否符合 expected]
- [如有基线，对比 with-skill 和基线的差异]
- [如有不符合，说明具体差异]

### 结论
- **通过/失败**：[结论]
- **说明**：[如有特殊情况，补充说明]
```

## 十、测试失败处理

当评估结果显示"失败"时：

### 1. 向用户报告失败原因

- 说明具体不符合预期的部分
- 引用 expected 描述与实际输出的对比

### 2. 提供可能的问题分析

- 技能逻辑可能的问题
- 测试场景可能的问题
- expected 描述可能的问题

### 3. 询问用户下一步操作

- 修改技能后重新测试
- 调整测试预期（expected）
- 终止测试，记录问题

## 十一、测试报告生成

### 何时生成

- 所有测试用例执行完毕后
- 用户要求生成报告时

### 报告内容

```markdown
# 测试报告

## 测试概述
- **技能名称**：[skill-name]
- **测试日期**：[date]
- **测试场景数量**：[N]
- **测试轮次**：test-XX

## 结果汇总

- ❌ 失败：[N] 个

## 详细结果
| 用例编号 | 场景名称 | 结果 | 评估报告 |
|---------|---------|------|---------|
| case-01 | [名称] | ✅/❌ | [链接] |

## 建议
- [基于测试结果给出的改进建议]
```

### 保存位置

`<test-XX>/test-report.md`

## 十二、测试终止条件

### 正常终止

- 所有计划的测试用例执行完毕
- 用户确认终止

### 异常终止

- 用户主动取消
- 发现技能严重缺陷，无法继续测试
- 测试环境异常

### 终止后处理

- 保存已执行的测试结果
- 记录终止原因
- 向用户报告当前进度

### 目录创建规则

- 测试目录位置不限制，默认 `test_space/<skill-name>-test/`
- 每次测试会创建 `test-XX/` 子目录（XX 为递增编号）
- 目录不存在时，先创建目录再写入文件

## 十四、关键设计决策

### 输入输出分离

**问题**：之前的测试指令包含 expected，等于告诉测试 SubAgent "标准答案"，失去测试意义。

**解决方案**：
- 测试指令只包含：技能路径、prompt、输出目录
- expected 只存在于 eval_metadata.json
- 评估阶段才读取 expected 进行判断

### tester 输出规范

**问题**：之前要求 tester 输出混杂了结果和过程报告，职责不清。

**解决方案**：
- `outputs/result.md`：任务结果（必需），对应 prompt 的响应
- `outputs/test-record.md`：过程记录（可选），主 Agent 明确要求时才输出

### inputs 目录

**目的**：记录测试输入，便于复现和审查。

**内容**：
- `prompt.md`：用户 prompt（必需）
- 其他输入文件（按需）

### 盲测原则

**核心思想**：测试 SubAgent 执行测试时，不应该知道"什么是正确答案"。

**实现方式**：
- 测试指令不包含 expected
- 测试 SubAgent 不读取 eval_metadata.json
- 评估阶段由独立的评估 SubAgent 对照 expected 判断
