# 发现合约设计参考

> 本文件提供系统化的发现合约设计方法——从载体清查、过滤评估、方案推荐到用户呈报的完整流程。
> 配合 `protocal-init.md` Phase 3 的"方案讨论"节使用，在该节识别出可行方向后，按此流程细化方案。

---

## 合约发现流程

### 已有合约的检测（先于 Step 1 执行）

在开始载体清查之前，先检查当前环境是否已部署发现合约：

1. **搜索已有合约**：在以下常见载体路径下用 grep 搜索合约特征文本（不区分大小写 `case: false`）：
   `index\.md|journal-root|写入操作.*journaling|读取.*INDEX`
   搜索范围：agent 配置目录（`.omp/agent/RULES.md`、`.omp/agent/AGENTS.md`、`.omp/agent/APPEND_SYSTEM.md`）> 项目根 AGENTS.md > 用户目录 `.agents/` > 全局默认路径。由近及远——agent 配置目录中的载体优先级最高。

2. **如果找到合约内容** → 读取并与本文件 Step 3 的最新合约模板逐项比对：

| 比对维度 | 检查内容 |
|---------|---------|
| 启动措辞强度 | 是否含「第一步」+ ⚠️ 等强调元素 |
| 读 INDEX 理由具体性 | 是否点名维护信号/活跃工作区/经验陷阱/信息盲区，而非泛泛的「获取全局背景知识」 |
| 写操作范围完整性 | 举例是否含移动/归档/删除（v4.6.0+ 要求），而非仅创建/编辑 |
| 拼写与语法 | 是否有明显拼写错误（如 `juonral`）或语法错误（如 `按照其中地`） |

3. **判断结果**：
   +- 有差异 → 输出差异清单 + 每条差异对应的更新建议。更新策略：**内容升级而非格式替换**——保留载体已有结构和格式，仅升级内容要素
   +- 完全匹配 → 报告「合约已是最新版本」，继续正常 Step 1 流程

4. **如果未找到合约** → 继续正常的 Step 1 流程（载体清查和全新设计）


### Step 1: 清查可用注入机制

清查当前运行环境中的所有可能载体。按以下分类逐项检查：

**A. 项目级配置文件**——注入到 agent 上下文的文件
- 检查当前工作目录（及父目录）是否存在以下文件：`AGENTS.md`、`CLAUDE.md`、`.cursorrules`、`CONTEXT.md` 等自动加载文件
- 确认哪些文件你拥有读取和写入权限

**B. 运行时配置机制**
- 检查是否有持久化的环境变量机制
- 检查是否有启动脚本（Hook）、初始化文件或配置模板

**C. 环境内置约定**
- 检查当前运行时是否有已知的约定加载路径
- 检查是否已有其他配置文件中包含了 session 启动时自动执行的指令

### Step 2: 逐项评估

对每个候选载体，用三个过滤性问题做评估：

| # | 问题 | 通过条件 | 检查方法 |
|---|------|---------|---------|
| 1 | 是否每次启动自动注入？ | 载体内容在 session 启动时自动成为上下文的一部分 | 根据运行时经验或文档判断 |
| 2 | 我能否写入？ | 文件路径在写入权限范围内，且不是运行时只读 | 尝试打开文件检查权限 |
| 3 | 是否跨 session 持久？ | 写入后不会在 session 结束时自动清理 | 根据文件生命周期判断 |

三个问题都是"是" → 候选。有任何一个是"否" → 排除。

**评估示例**（仅作格式参考，实际结果取决于当前环境）：

| 候选载体 | 自动注入 | 可写入 | 持久 | 结果 |
|---------|---------|-------|------|------|
| AGENTS.md（全局路径） | ✅ 每次 session 自动加载 | ✅ | ✅ | **候选** |
| 项目本地 AGENTS.md | ❌ 非本次 session 工作目录则不加载 | ✅ | ✅ | 排除 |
| 环境变量 | ❌ 仅读取，无写入接口 | ❌ | ❌ | 排除 |

> 此表仅用作格式参考。每次执行时必须基于实际运行时环境重新清查和评估，不得照搬示例。

### Step 3: 推荐方案

从通过过滤的候选中选择最佳方案：

- **优先级原则**：自动注入范围最广、维护成本最低的载体优先
- 如果无候选通过过滤 → 参照 `protocal-init.md` Phase 3 的"回退方案"节

推荐方案应包含：
1. **载体路径**——确切的文件路径或机制名称
2. **合约文本**——需要注入的具体内容。建议包含以下信息：

下方代码块中的 `>` 前缀是模板自身的展示格式（适合直接注入系统提示词 generic-rules 段的场景），不是对载体格式的强制要求。实际写入时须适配载体的已有格式：如载体使用 Markdown 标题和列表（如 RULES.md），合约文本的内容要素（journal-root 声明、读 INDEX 指令、写操作前加载规则）不变，但需转换为标题+列表格式。格式一致可降低 Agent 将合约识别为「外来片段」而跳过的概率。如果是全新载体（无已有格式参考），使用模板的 `>` 引用块格式——它在系统提示词注入场景中最可移植。

   ```
   > Journal root: <journal-root>
   > ⚠️ 启动后第一步：读取 <journal-root>/index.md。INDEX.md 包含维护信号、活跃工作区、经验陷阱等关键上下文——跳过意味着在信息盲区中操作
   > 读取 INDEX.md 不需要加载 journaling skill
   > 向 <journal-root> 目录中进行任何写入操作（移动、修改、编写、删除等）时，必须加载 journaling 技能并按其指令操作
   ```

合约中「读取 INDEX.md 不需要加载 skill」和「写入前需加载 journaling」与 INDEX.md 协议声明行（种子模板第一、二行）内容重复。这是有意冗余——双重防线确保即使一层被忽略另一层仍生效。不建议删减其中任何一层。此设计选择对应技能设计决策 #6（Journal 服务 Agent——多层防线比单层规则更可靠）和决策 #20（信息依赖优于规则闸门——三层信息依赖结构中的 contract→INDEX 链路）。

3. **写入方式**——使用哪个工具或命令写入

### Step 4: 呈报用户

向用户说明以下内容，征得同意后再执行：

1. **推荐的载体**和选择理由（为什么用它而不是其他选项）
2. **合约内容**——即将注入的文本
3. **风险说明**——是否影响现有上下文结构、是否覆盖已有内容
4. **备选方案**——如不同意推荐方案，是否有其他可接受选项

用户确认后，由用户或后续 session 执行写入。**Agent 在此阶段不执行任何写入操作。**
