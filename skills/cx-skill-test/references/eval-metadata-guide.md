# eval_metadata.json 编写指南

> 本指南帮助 Agent 编写清晰、可验证的 eval_metadata.json 文件。

---

## 一、eval_metadata.json 的作用

eval_metadata.json 是测试用例的核心文件，记录：
- 测试场景名称（case_name）
- 用户实际会说的 prompt
- 预期效果（expected）
- 是否启用基线对比（baseline）

---

## 二、必填字段说明

### case_name（测试用例名称）

**作用**：标识测试场景，便于追踪和报告。

**要求**：
- 简洁明了，一眼看出测试什么
- 使用"场景类型-具体描述"格式
- 避免过长，建议 5-15 个字符

**示例**：
- ✅ "核心触发-测试技能"
- ✅ "扩展场景-自动补全"
- ✅ "边界场景-输入过短"
- ✅ "失败场景-无输入"

### prompt（用户 prompt）

**作用**：用户实际会说的话，用于触发技能。

**要求**：
- 真实、自然，符合用户习惯
- 不要过度简化或美化
- 包含必要的上下文信息

**示例**：
- ✅ "帮我测试一下 cx-asset-design 技能"
- ✅ "验证这个技能能不能正常工作"
- ❌ "测试"（过于简化）
- ❌ "请帮我执行技能测试流程"（过于正式）

### expected（预期效果）

**作用**：描述预期输出，用于评估测试结果。

**要求**：
- 使用可验证的描述性语言
- 避免主观形容词（好、完整、正确等）
- 指定核心要求和质量标准
- 提供明确的判断依据

**示例**：
- ✅ "输出包含字段A、B、C，字段A和字段B必须非空"
- ✅ "文件包含标题、日期、作者、正文四个部分，每个部分格式正确"
- ✅ "JSON 格式有效，包含 case_name、prompt、expected、baseline 四个字段"
- ❌ "输出结果很好"（主观，不可验证）
- ❌ "内容完整"（模糊，不可验证）
- ❌ "格式正确"（抽象，不可验证）

### baseline（是否启用基线对比）

**作用**：决定是否执行基线对比测试。

**取值**：
- `true`：启用基线对比，将执行 with-skill 和基线两种模式
- `false`：仅执行 with-skill 模式

**建议**：
- 默认使用 `false`
- 当需要对比技能的实际价值时，使用 `true`

## 三、expected 编写技巧

### 技巧 1：使用"核心要求 + 质量标准"模板

**模板**：
```
"输出必须包含 [核心要求]，满足 [质量标准]"
```

**示例**：
- "输出必须包含字段A、B、C，字段A和字段B必须非空"
- "输出必须包含测试报告，报告必须包含测试概述、结果汇总、详细结果三个部分"

### 技巧 2：指定具体的内容要求

**明确指出输出必须包含哪些内容**。

**示例**：
- ✅ "输出包含：1) 测试概述 2) 结果汇总 3) 详细结果"
- ❌ "输出包含完整内容"

### 技巧 3：定义可验证的质量标准

**说明内容必须满足什么条件**。

**示例**：
- ✅ "字段A和字段B必须非空"
- ✅ "JSON 格式有效"
- ✅ "每个部分格式正确"
- ❌ "内容质量高"

### 技巧 4：避免主观形容词

**主观形容词清单（避免使用）**：
- 好、很好、优秀
- 完整、全面
- 正确、准确
- 合理、恰当
- 清晰、清楚

**替代方案**：
- "好" → "满足 X 标准"
- "完整" → "包含 A、B、C 三个部分"
- "正确" → "符合 X 格式"
- "清晰" → "包含明确的 Y 信息"

### 技巧 5：提供判断依据

**让评估 SubAgent 能明确判断是否符合预期**。

**示例**：
- "输出包含字段A、B、C，字段A和字段B必须非空" → 判断依据：检查字段是否存在，检查字段A和B是否非空
- "JSON 格式有效" → 判断依据：检查 JSON 是否能正常解析

## 四、不同测试类型的 metadata 示例

### 示例 1：核心场景测试

```json
{
  "case_name": "核心触发-测试技能",
  "prompt": "帮我测试一下 cx-asset-design 技能",
  "expected": "Agent 正确加载技能，询问被测试技能名称，准备测试方案",
  "baseline": false
}
```

### 示例 2：扩展场景测试

```json
{
  "case_name": "扩展场景-生成报告",
  "prompt": "测试 cx-asset-design 技能，并生成测试报告",
  "expected": "输出包含测试报告文件（test-report.md），报告包含测试概述、结果汇总、详细结果三个部分",
  "baseline": false
}
```

### 示例 3：边界场景测试

```json
{
  "case_name": "边界场景-输入过短",
  "prompt": "测试",
  "expected": "Agent 询问用户要测试哪个技能，不直接执行测试",
  "baseline": false
}
```

### 示例 4：失败场景测试

```json
{
  "case_name": "失败场景-无输入",
  "prompt": "",
  "expected": "Agent 不触发技能，直接回答用户或询问用户需求",
  "baseline": false
}
```

### 示例 5：基线对比测试

```json
{
  "case_name": "基线对比-设计资产",
  "prompt": "根据这个概念，设计资产",
  "expected": "with-skill 输出包含完整的资产设计，基线输出不包含资产设计或设计质量较低",
  "baseline": true
}
```

## 五、常见错误

### 错误 1：expected 过于主观

❌ 错误：
```json
{
  "expected": "输出结果很好"
}
```

✅ 正确：
```json
{
  "expected": "输出包含字段A、B、C，字段A和字段B必须非空"
}
```

### 错误 2：expected 过于模糊

❌ 错误：
```json
{
  "expected": "内容完整"
}
```

✅ 正确：
```json
{
  "expected": "文件包含标题、日期、作者、正文四个部分，每个部分格式正确"
}
```

### 错误 3：prompt 过于简化

❌ 错误：
```json
{
  "prompt": "测试"
}
```

✅ 正确：
```json
{
  "prompt": "帮我测试一下 cx-asset-design 技能"
}
```

### 错误 4：case_name 不够清晰

❌ 错误：
```json
{
  "case_name": "测试1"
}
```

✅ 正确：
```json
{
  "case_name": "核心触发-测试技能"
}
```

### 错误 5：缺少判断依据

❌ 错误：
```json
{
  "expected": "输出格式正确"
}
```

✅ 正确：
```json
{
  "expected": "JSON 格式有效，包含 case_name、prompt、expected、baseline 四个字段"
}
```

## 六、最佳实践

### 实践 1：使用模板编写 expected

```
"输出必须包含 [核心要求]，满足 [质量标准]"
```

### 实践 2：使用列表明确内容要求

```json
{
  "expected": "输出包含：1) 测试概述 2) 结果汇总 3) 详细结果，每个部分格式正确"
}
```

### 实践 3：使用"必须"强调必要条件

```json
{
  "expected": "字段A和字段B必须非空"
}
```

### 实践 4：提供具体的判断标准

```json
{
  "expected": "JSON 格式有效（能正常解析），包含 4 个必需字段"
}
```

### 实践 5：使用真实的用户 prompt

```json
{
  "prompt": "帮我测试一下 cx-asset-design 技能"
}
```

---

## 七、总结

**eval_metadata.json 编写核心**：
1. case_name：简洁明了，一眼看出测试什么
2. prompt：真实自然，符合用户习惯
3. expected：可验证，避免主观形容词
4. baseline：按需启用，默认 false

**记住**：expected 的目标是让评估 SubAgent 能明确判断是否符合预期。
