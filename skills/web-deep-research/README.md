# web-deep-research — 设计说明

> 给人读的。Agent 读 SKILL.md。

## 设计目标

这个技能解决一个问题：**「我要把这个话题研究透——怎么系统地逼近真相？」**

它不解决"怎么搜得快""怎么用某个工具""怎么判断一条信息准不准"——那些是工具使用指南和评估方法论的事。它解决的是**研究本身的流程设计**：从明确意图，到收集线索，到交叉比对，到交付一份结论可由读者独立检验的报告。

三个约束定义了它的边界：

1. **不是快速查询工具**——查一句话就能回答的事实用 `web-search-protocol`，不用这个
2. **不是工具选择指南**——`web_search` vs `web_extract` vs `browser_navigate` 的选择是另一个技能的事
3. **不是纯咨询**——"你觉得我该怎么做"如果没有多信源调查维度，就不是研究

## 核心架构决策

### Phase 0-4 流程：收集和评估严格分离

**决策**：Phase 2（纯收集，记录 URL）和 Phase 3（按信息性质评估）之间有一条硬边界。

**理由**：Agent 在收集信息时最容易犯的错是过早下判断——"这条看起来不可靠，跳过"或者"这个说法很有道理，当成事实"。分离收集和评估迫使 Agent 先完整收集再统一评估，避免锚定效应。

**取舍**：这比"边搜索边判断"慢得多，Token 消耗也更大。所以文档开头声明了——"Use when thorough investigation matters more than speed"。

### 领域特化文档：各走各的路

每个领域参考文档的组织逻辑由该领域的研究对象决定，不套统一模板。已有的文档恰好展示了这种多样性：

| 文档 | 领域 | 组织逻辑 | 独特设计 |
|------|------|----------|----------|
| creative-work | 影视/动漫/游戏/书籍 | 客观性梯度（事实→半事实→主观） | 三层维度 + Tier 3 收束机制 |
| person-biography | 公众人物 | 叙事逻辑（是谁→经历了什么→跟谁相关→别人怎么看他→留下了什么） | Phase 1 身份锁定硬门控 |
| policy-law | 政策/法规 | 规范层次（条文→背景→适用→评价），从圆心向外扩散 | 硬约束体系（官方原文+地域+时效三维匹配） |
| academic-research | 学术/高专业壁垒话题 | 不限定领域，针对研究方法论 | 论文置信度评估逻辑 + 跨领域关联探索 |
| controversial-topics | 争议性话题 | 立场标定与框架分离，争议真实性判断 | 硬约束（立场标定+声称框架分离+争议真实性） + 争议层次分析 |
| historical-events | 历史事件 | 时间分层 + 多叙事并行 | 硬约束（事实/因果分离+来源年份）+ 事件参数锁定 |
| competitive-research | 竞品/市场横向对比 | 领域边界→实体发现→单实体多维度追踪→关联→并列对比 | 硬约束（领域边界与实体清单重建） |

**设计经验**：

- **每个领域先问"这个研究对象的知识结构是怎样的"**——人物适合叙事逻辑，法规适合规范层次，创意作品适合客观性梯度。硬套不对的轴线会出问题（policy-law 第一版用人物模板写就全错了）
- **硬约束和启发参考要截然分开**——法规有不可妥协的采信规则，Agent 需要明确知道"做不到这个就不能往下走"
- **纯背景知识可以独立成章**——法律体系介绍、法规/政策区分这类基础知识，Agent 看了有助于理解研究课题，但不属于操作指引
- **越详细越需要收束，但收束的方式因领域而异**——创意作品收束主观维度，法规收束"从圆心走多远"，学术研究则不收束维度而是收束"不要深挖单点"
- **文档长度和领域复杂度匹配即可**——creative-work 150 行、academic-research ~60 行、policy-law ~160 行，不用削足适履

### 来源类别命名：两种逻辑

不同领域适用不同的来源组织方式：

- **按信息生产者（"谁说的"）**——适用人物和创意作品研究，因为同一事实来自"自述""同时代见证""学界评价"性质完全不同
- **按信息需求（"去哪儿找什么"）**——适用法规研究，"条文的唯一来源是官方数据库"比"全国人大发布的信息"更直接有用
- **学术研究不适合按来源分类**——跨学科、跨来源性质差异太大，按"获取途径"（摘要、综述、同行讨论）组织更实用

## 维护指南

### 新增领域特化文档

1. **先问"这个领域的研究对象是什么知识结构"**——不要假设能套已有文档的结构。想一想：你向朋友介绍这个东西时用什么顺序？那个顺序就是这个领域的组织轴线
2. **决定是否需要硬约束**——只有像法规这样"采信条件不可妥协"的领域才需要。大部分领域只需要启发参考
3. **来源组织方式由领域决定**——不预设"按生产者分类"或"按需求分类"哪个对，看哪个对 Agent 找信息更直接
4. **启发提示用软语气**——"可以参考""试试看""不是必须步骤"。不要把启发写成结构要求
5. **更新 SKILL.md 的 Content Index 表格**（一行），不改 Instructions
6. **更新本 README 的核心架构决策表**

### 修改已有特化文档

1. 改完后跑一轮推演：模拟 Agent 首次加载这份文档，看它在实际研究场景中能否正确理解和运用
2. 推演发现的修复如果同样适用于其他特化文档，考虑是否需要同步——但不要为了"结构对齐"而硬套
3. 更新 CHANGELOG.md
