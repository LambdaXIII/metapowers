# web-entity-search — Design Notes

## 设计意图

web-entity-search 填补的是"直接搜索"和"深度研究"之间的空白。
直接搜索快但结构松散——Agent 可能只抓到首条结果的碎片，漏掉关键维度。
web-deep-research 全面但重——Phase 0-4 的链式追踪和交叉比对对"XXX 是啥"过于奢侈。

本技能只加了两层东西：**维度模板**（告诉 Agent "这类实体该看哪几个口袋"）和**停止纪律**（"≥2 个来源就填，填完就停"）。模板不是研究报告框架——是搜索时的 checklist。

## 关键架构决策

### 模板放 templates/ 而非 references/

templates/ 语义是"Agent 按需读取的执行模板"，references/ 是"协调者读的知识文件"。维度模板是前者——Agent 在搜索过程中拿着它当 checklist 用。

### 不硬编码其他技能名

SKILL.md 中不出现其他技能的名称——边界用功能描述而非点名来表达（"链式线索追踪超出本技能范围"而非"用 web-deep-research"）。技能独立运行，不假设环境中安装了任何特定技能。

### 输出是自然语言而非表单

维度是给 Agent 看的，用户看到的是自然段落。如果 Agent 把"维度 1：身份领域 ✔️"的检查表呈现给用户，说明 Agent 误解了技能的本意——模板是工作工具，不是交付格式。

### 置信回检只抽样，不全量

Step 4 不是 mini 版 Phase 3——它只抽查 2-3 个关键维度，发现矛盾就标注。重新打开搜索链路就变成了研究报告，违背"薄框架"的原则。

## 维护注意事项

- 新增实体类型时：写新模板 → 更新 SKILL.md Content Index 表格 → 更新 CHANGELOG
- 修改现有模板的维度时：检查输出结构是否需要调整
- 如果某个模板的"搜索指引"需要新增来源类型，同步检查其他模板是否有类似需求
