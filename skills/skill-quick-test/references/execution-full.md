# Full 模式执行

> 📍 本文件是 skill-quick-test skill 的参考文件。

---

## 适用条件

复杂测试场景 ≥ 2 或总测试数 ≥ 5，且用户已确认场景设计。

---

## 【铁律】计划闸门（委派前必过）

委派子代理前**必须**完成以下全部步骤：

- [ ] 用 `templates/test-plan.md` 写入 `test-plan.md` 到临时目录
- [ ] 计划中包含每个场景的预期效果
- [ ] 计划路径告知用户

**未完成此闸门前，禁止调用 delegate_task。** 计划文件是汇总阶段的预期对比基准——跳过此步骤会导致汇总时无法评估技能表现与预期的偏差。

---

## 目录结构

1. 检查环境变量 `SKILL_TEST_DIR`
2. 如果未设置，使用默认路径：`<cwd>/.quick-tests/`
3. 创建本轮目录：`<temp-dir>/round-<id>/`

```
<temp-dir>/round-001/
├── test-plan.md
└── reports/
    ├── S01-U1.md
    ├── S01-U2.md
    └── ...
```

---

## 测试计划文档

使用 templates/test-plan.md 模板写入 `test-plan.md`。

计划中包含场景的完整字段（使用场景描述、模拟背景、入口路径、用户想达成什么、预期效果、测试单元列表）、覆盖检查结果和模式判定摘要。预期效果写在计划中，但**不通过子代理 context 传递**。

---

## 子代理委派格式

Context 字段结构与 Light 模式相同（框架声明、硬约束、使用场景、模拟背景、入口路径、用户想达成什么、加载规则、推演指引、失败行为、能力模拟、禁止事项——全部见 `execution-light.md`），**但有一处覆盖**：

Light 模式 context 中的 `约束: 禁止写入任何文件。所有发现通过返回值报告。` 在 Full 模式下**替换为**：

```
约束:
  允许写入: <temp-dir>/round-<id>/reports/<单元ID>.md（如 B01.md, S01-U1.md, S02-U2.md）
  报告严格按场景卡片格式输出（路径追踪、停顿/困惑、断裂点、意外发现）。
  其他路径禁止写入。
  写入后返回文件名。
```

---

## 委派后中止检查

与 Light 模式相同（见 `execution-light.md` 的「委派后中止检查」节）。

---

## 并发控制

与 Light 模式相同。各子代理写入不同文件，无冲突风险。

子代理返回文件名后，主代理用 `read_file` 读取所有报告内容。

---

## 计划闸门验证

全部子代理完成后，进入汇总前验证：临时目录中 `test-plan.md` 是否存在？
- 存在 → 进入 summarization.md
- 不存在 → 补写 test-plan.md 后再进入汇总（计划中的预期效果是汇总的对比基准）
