# 记录原则详细指导

## 核心约束

确保下次阅读时能准确理解主旨。

## 什么值得记录

- 重要决策及其理由
- 解决方案和踩过的坑
- 关键发现和洞察
- 上下文依赖（为什么这样做）

## 什么不值得记录

- 临时状态（会快速变化）
- 可重建信息（可从代码或文件推导）
- 冗余细节（与主旨无关的细节）

## 场景建议

| 场景 | 建议记录内容 |
|------|-------------|
| 完成复杂任务后 | 决策、理由、关键步骤 |
| 解决棘手问题后 | 解决方案、踩过的坑、参考资料 |
| 会话结束前 | 当前进展、下一步计划、待解决问题 |
| 用户明确要求 | 按用户意图记录 |

## 关联其他文件

当记录内容涉及项目中已有的其他文件时，如有必要且顺手，可以在记录中插入链接。

**判断标准**：

- **有必要**：该链接能帮助未来快速定位相关内容
- **顺手**：不需要专门查询，直接在理解内容时判断

**格式**：

- `[[文件名]]` 或 `[描述](相对路径)`
- 每条记录最多 2-3 个链接

**示例**：

```markdown
Agent-memory 的核心理念参考了 [[方法论_参考文档优化]]
```

## 信息粒度

> 记录是给未来的 Agent 看的。我们定义原则而非具体实施方式，让 Agent 根据实际情况自行判断。

### 文件命名原则

文件名应取自内容本身，而非来源或过程。

- ✅ `方法论_技能类型与设计策略.md` — 反映内容主题
- ❌ `从note-taking到agent-memory重构.md` — 反映过程，未来阅读无意义

### 主题分离原则

每个主题一个文件，避免混合不同领域的内容。

### 文件大小控制

根据项目特点控制文件大小，当内容涉及多个独立主题、或单个主题变得复杂时，考虑拆分。具体阈值由 Agent 根据记忆库规模和阅读习惯自行判断。

### 行长度建议

根据阅读习惯控制行长度，长句断成多行，便于 Agent 渐进式读取。具体数值由 Agent 根据内容特点和阅读场景自行判断。

### index.md 职责边界

`index.md` 只放摘要和入口，详细内容放单独文件。

### 拆分时机判断

当文件内容涉及多个独立主题、或单个主题变得复杂时，考虑拆分。

## index.md 编写

### 强制要求

`index.md` 必须在开头声明：

```markdown
> ⚠️ **强制协议**：编辑或写入本记忆库任何文件前，必须先加载 `cx-agent-memory` 技能。
```

### 板块建议

Agent 需自行判断需要哪些板块。以下是常见板块建议：

| 板块 | 内容 |
|------|------|
| 项目概述 | 项目目标和当前状态 |
| 关键信息摘要 | 重要决策、关键发现的摘要 |
| 详细内容入口 | 指向详细文件的入口列表 |
| 最近进展 | 最近的工作进展 |

### 更新时机

| 记录类型 | 更新方式 |
|----------|----------|
| 重要信息 | 摘要直接写入 index.md |
| 次要信息 | 创建单独文件，在 index.md 添加入口 |
| 项目状态重大变化 | 更新项目概述 |

### 编写原则

- 内容简洁，确保下次阅读能快速理解主旨
- 可根据项目特点调整结构
- 详细示例见 `examples/` 目录
