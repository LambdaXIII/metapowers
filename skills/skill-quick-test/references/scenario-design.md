# 场景设计

> 📍 本文件是 skill-quick-test skill 的参考文件。主入口和决策树请见 SKILL.md。

---

测试用例设计基于 ISTQB 框架，六步方法论。本文件覆盖**步骤 2-5**。步骤 1 在 `scope-analysis.md`，步骤 6 在 `error-patterns.md`。

每一步同时产出：**设计动作** + **覆盖标准**。

---

## 步骤 2：等价类划分

### 设计

将使用场景的输入空间划分为等价类。每类中 Agent 应得到相同的引导结果。

三个维度，各取值如下：

| 维度 | 等价类 | 适用范围 |
|------|--------|---------|
| **入口路径** | 正门 / 侧门 / 回访 / 引用跳转 | **仅加载技能的单元** |
| **任务明确度** | 精确 / 抽象 / 模糊 | 全部单元 |
| **起始上下文** | 裸 / 半知 / 全知 | 全部单元 |

> **入口路径对 baseline 不适用。** baseline 单元 agent 不加载技能，不存在"从哪个文件进入"的问题。baseline 单元只给"用户想达成什么"，不给入口路径。

### 选取原则

不需要笛卡尔积。从范围分析的输出中，为每个等价类取至少一个**有意义的代表**。

- 正门 + 抽象 + 裸 → 最典型的复杂测试场景
- 侧门 + 精确 + 全知 → 验证文件自标识和回链
- 回访 + 精确 + 半知 → 验证独立可读性

### 覆盖标准（检查用）

- [ ] 入口路径：每个等价类 ≥ 1 个测试用例（**仅适用于加载技能的单元**）
- [ ] 任务明确度：每个等价类 ≥ 1 个测试用例
- [ ] 起始上下文：每个等价类 ≥ 1 个测试用例

这三项不要求每两两组合覆盖——要求每个单维度都有代表。

## 步骤 3：边界测试

### 设计

从范围分析的「能力边界」输出中，为**每条**边界声明生成两个测试：

| 测试 | 位置 | 预期行为 |
|------|------|---------|
| 边界内 | 声称范围内的最后一项/最后一个值 | 正常处理 |
| 边界外 | 刚好超出声称范围的一项/一个值 | 给出清晰的"不支持"或降级信号 |

边界测试是简单测试——验证技能对边界输入的行为是否符合预期。虽然测试逻辑简单，仍须委派子代理执行，避免主代理的上下文偏见影响结果。

### 覆盖标准（检查用）

- [ ] 每条边界声明 → 生成了双向测试（内 + 外）
- [ ] 边界外测试的预期不是"成功"，是"降级/拒绝"

## 步骤 4：决策逻辑覆盖

### 触发条件

仅当范围分析输出中「决策点」为"有"时触发。

### 设计

如果技能有显式决策树或条件分支：

- **基础覆盖**：每个分支至少被走一次
- **组合覆盖**：选择 1-2 条经过连续两个决策点的典型路径

不需要测试所有组合路径——选代表性的热点路径。

### 覆盖标准（检查用）

- [ ] 基础分支覆盖：每个分支 ≥ 1 个路径测试
- [ ] 组合覆盖：如 ≥ 2 个决策点，≥ 1 条多分支路径

## 步骤 5：端到端用例

### 设计

每个用例场景 = 抽象需求描述 + 不告知具体操作路径 + baseline 对比。

| 元素 | 要求 |
|------|------|
| 需求描述 | 用自然语言，不告知图表类型/文件路径/预期操作 |
| 目标类型 | 生成 / 查询 / 诊断 |
| baseline | 每个场景拆两个单元：加载技能 vs 不加载 |
| 起点 | 正门为主 |

### 场景卡片格式

```
场景 ID: S01
简述: [一句话]
需求明确度: [精确 / 抽象 / 模糊]
目标类型: [生成 / 查询 / 诊断]
背景上下文: [裸 / 半知 / 全知]
入口路径: [路径] — 正门/侧门/回访（仅 a 单元，baseline 不填）
目标: [自然语言，不告知图表/路径]

测试单元:
  - S01-U1: 加载被测技能
  - S01-U2: Baseline（不加载被测技能）

预期效果（不告知子代理）:
  - 预期路径: ...
  - 预期不会卡住: ...
```

### 覆盖标准（检查用）

- [ ] 每个用例场景有 baseline
- [ ] 场景数 ≥ 2
- [ ] 覆盖 ≥ 2 种任务明确度
- [ ] 覆盖 ≥ 2 种目标类型（生成/查询/诊断 中至少两种）

> **注意**：`场景数 ≥ 2` 是覆盖检查的硬要求。如果范围分析只产出 1 个有意义的复杂场景，
> 应主动补充或向用户说明原因，而非降低标准。

## 覆盖闸门（步骤 2-6 全部完成后）

进入模式判定前，必须全部通过：

1. **步骤 2-5 覆盖标准**：逐项检查上方各步骤的覆盖标准清单，全部 ✅
2. **步骤 6 错误模式检查**：error-patterns.md 逐条检查完成
3. **用户确认**：向用户输出测试用例列表，获得批准

### 用户确认

闸门通过后，向用户输出简洁的测试用例列表：

| ID | 类型 | 简述 | 使用场景 | 预期效果 |
|----|------|------|---------|---------|
| B01 | 边界 | [边界内测试描述] | — | [预期正常处理] |
| B02 | 边界 | [边界外测试描述] | — | [预期降级/拒绝信号] |
| S01 | 复杂 | [场景简述] | [明确度·目标·上下文] | [Agent 应达成的效果] |
| S01-U2 | baseline | 同上，不加载技能 | 同上 | [对照：无技能时的表现] |

模式：[Light / Full]

等待用户确认。用户要求修改 → 回退到对应步骤重新设计。

---

## 模式判定

统计**复杂测试场景数**和**总测试数**：

| 条件 | 模式 |
|------|:--:|
| 复杂测试场景 ≥ 2 **或** 总测试数 ≥ 5 | **Full** |
| 其他 | **Light** |

- 复杂测试场景数不计 baseline（S01-U1 + S01-U2 = 1 个场景）
- 总测试数 = 所有简单测试 + 所有复杂测试单元（含 baseline）
- 简单测试 = 边界测试 + 错误模式中需要实际执行验证的条目

判定后告知用户当前模式。执行流程见：

- [execution-light.md](execution-light.md)
- [execution-full.md](execution-full.md)
