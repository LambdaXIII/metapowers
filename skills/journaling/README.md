# journaling 设计文档

> 本文档记录 journaling 技能的设计意图和核心决策，作为维护参考。

---

## 设计初衷

Agent 的长期记忆需要一个结构化的笔记本系统——不是简单的文件堆，而是一个有仪表盘、有条目规范、有维护周期的知识管理体系。

这个技能解决的问题：

- **"写了不读"**：经验记下来了，但下次遇到相关任务时不会先查阅
- **"写了找不到"**：信息存在但无法在需要时快速定位
- **"写了变噪音"**：仪表盘膨胀，信号被淹没在日志中
- **"写了不维护"**：条目过期、标签蔓延、结构腐化，但无人清理

---

## 核心设计决策

### 1. 两闸门模型——被动存储变主动拦截

行动闸门和写入闸门的设计核心是把"被动知识存储"转化为"主动行为拦截"。

**关键洞察**：知识记下来不等于会被使用。Agent 的默认行为模式是"刺激→反应"，没有"先查已知"的中间步骤。闸门通过在启动协议中嵌入强制扫描，把"我知道这个"转化为"我必须现在读这个"。

- **设计约束**：闸门必须位于入口点——那个唯一保证在行动前被读到的文件。对 journal 来说是 index.md，对项目来说是项目的入口规范文件。如果闸门埋在子页面里，它就变成了更多不会被读的内容。

**闸门是 journal 的活文档**：技能提供闸门的设计框架（格式、列设计、选择标准、维护规则），但闸门的具体内容——哪些行动触发哪些回顾、哪些自问过滤哪些写入——由 Agent 根据实际失败动态创建和更新。闸门从真实失败中生长，不从预设规范中复制。这意味着初始化时闸门表是空的，随着失败积累逐渐填充。

### 2. 渐进式披露——SKILL.md 作为路由

SKILL.md 只保留路由表 + 核心约束 + 操作规则。所有机制细节在 references/ 中按需加载。

**原因**：上下文成本。加载技能写一条笔记和加载技能做全量维护的上下文需求完全不同。路由表让两种场景的成本分离——写笔记只需加载 note-spec.md，做维护才加载 maintenance.md。

### 3. 四阶段维护协议——诊断与决策解耦

Phase 1 纯诊断（不决策），Phase 2 重组（先归档保留历史完整，再调结构），Phase 3 质量检查，Phase 4 收尾。

**关键约束**：Phase 1 的标签是状态描述不是决策。"partially stale"说的是"这个文件部分过时了"，不是"应该归档它"。决策在 Phase 2 做，基于 Phase 1 的完整诊断。这防止了"扫描到一半就开始动手"的倾向。

### 4. 摘要锚定——scope boundary 而非内容总结

每条笔记的 summary 是其 scope 的边界定义，不是内容的概括。

**三个检查点**：创建时写摘要定义范围→修改前读摘要判断"新内容还在范围内吗"→修改后验证摘要是否仍准确。这让"update vs create"的决策从"我需要想清楚"变成"读摘要→答案自明"。

### 5. 受控标签表——防止标签蔓延

标签必须来自 tag-registry.md 中的受控词汇表。新标签先注册再使用。

**原因**：不受控的标签会退化成噪音——一次审计发现 97 个不同标签分布在 33 个条目中，大多数只用过一次。受控词汇表让标签成为有效的检索工具而非分类幻觉。

### 6. 读写非对称——读不加载，写才加载

读取 journal（浏览 index.md 仪表盘、查阅条目、搜索内容）**不需要**加载 journaling 技能。只有在写入或维护时才需要加载。

**原因**：journal 是 Agent 每轮启动都要读取的——index.md 是启动协议的第一步。如果每次读 journal 都要加载 15 个 reference 文件，上下文成本不可接受。读写分离让"读取"保持零成本（index.md 本身就是自描述的仪表盘），把技能的复杂性全部推到"写入"侧——写入时才需要规范（格式、目录、标签、摘要锚定），维护时才需要协议（四阶段流程）。

**设计自洽**：index.md 的协议声明行明确标注"读此文件不需要加载 skill · 写入或维护时必须加载"。这和渐进式披露形成配合——读取是零成本高频操作，写入是中等成本低频操作，维护是高成本极低频操作，三者的上下文成本逐层递增。

### 7. Journal 服务于 Agent 而非用户

用户不审批 journal 操作。Agent 自主维护（归档、合并、删除、标签修复），可以征求建议但不请求许可。

**设计约束**：这不是协作偏好，是设计要求——用户无法判断 journal 操作的正确性，因此技能必须让 Agent 自给自足。

### 8. 初始化协议——从零生长而非预设填充

初始化协议（`references/initialization.md`）引导 Agent 从零创建完整 journal：目录结构、初始 index.md（四个核心节：协议声明 + 经验摘要 + 专项工作 + 最近变更，附带填充说明）、维护备忘、引导条目。

**关键设计**：初始化时所有内容区（经验摘要、专项工作、最近变更）都是空的，附带说明"何时开始填充"。闸门不在初始模板中——作为可选工作流工具，在实际失败积累时逐条添加。这让 journal 从真实使用中有机生长，不从预设规范中复制。

## 措辞风格约定

- **第一人称**："我记的，我用的"——journal 是 Agent 自己的笔记本，不是外部要求
- **中英混合**：结构性内容（frontmatter、标签）用英文，解释性内容和操作规则用中文
- **"闸门"而非"检查点"**：闸门（gate）强调强制中断，检查点（checkpoint）暗示可选
- **"场景编码"而非"状态标签"**：dashboard 行编码的是"正在做什么类型的工作"，不是"active/completed"
- **指令式而非叙事式**：操作规则用祈使句（"验证..."、"不要..."），不用"曾经犯了X错所以应该Y"的叙事结构。教训吸收为指令，案例留在 journal 中。

---

## 维护注意事项

- **操作规则是活文档**：SKILL.md 的 Operating Rules 在遭遇新的操作失败时追加。新增规则用指令式措辞，不夹带历史溯源——历史案例属于 journal 经验条目，不属于技能规范。
- **闸门内容由 journal 维护**：行动闸门和写入闸门的具体规则是 index.md 中的活文档，不在技能文件中。技能只提供设计框架（index-spec.md Section 5/6）和维护流程（maintenance.md Phase 1 Step 8）。
- **references 之间有交叉引用**：修改一个 reference 中的概念时，检查其他 reference 是否引用了同一概念（如 journal-concept 的设计原则和 index-spec 的 dashboard 设计是同一套思想的不同层面）。
- **SKILL.md 的路由表和 Linked Files 要同步**：新增或移除 reference 文件时，两个位置都要更新。
- **泛化与脱敏**：本技能从特定运行环境迁移而来。示例中使用"项目 A/B"等通用占位符代替真实项目名。journal 路径用 `<journal-root>` 参数化。维护时保持这种通用性——不要回填特定项目名或硬编码路径。
