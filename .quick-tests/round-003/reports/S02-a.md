## 2. 每个停顿/困惑/误解的具体位置

### 🔴 停顿点 1 — `references/web-backends-china.md:1-3`
**位置**: 文件头部段落  
**类型**: 半知用户的上下文缺失  
**描述**: 用户从侧门进入，该文件第一句直接提到 "DDGS (DuckDuckGo) backend timed out" 和 "parallel keyless MCP"。对于一个"记得有实测数据但不记得文件名"的半知用户，这里缺少前缀说明：什么是 "backend"？什么是 "MCP"？这些概念在本文档中没有任何展开解释。用户需要从 SKILL.md 补课，但侧门进入时并没有引导用户去读 SKILL.md。  
**推演**: 实际推演中，我必须读取 SKILL.md 才能理解这些术语的来源。文档没有侧门读者的上下文引导。

### 🔴 停顿点 2 — `references/web-backends-china.md:表格（第 7-16 行）`
**位置**: 后端可访问性表格  
**类型**: 数据不完整导致无法满足用户需求  
**描述**: 用户意图是"确认哪些后端在国内可用"。表格列出了 8 个后端，但只有 2 个（parallel ✅, ddgs ❌）有明确结论；1 个（searxng ✅ 自控）有条件结论；**5 个标记为 "⚠️ untested"**（firecrawl, tavily, exa, brave-free, xai）。这意味着该"实测数据文件"实际上只覆盖了 3/8 的后端。对于回答"哪些可用"这个查询，数据完整度仅有 37.5%。  
**推演**: 如果用户依赖这个文件作为权威来源，会得到大量"待测试"结果，无法满足确认需求。

### 🔴 困惑点 3 — `SKILL.md:84`
**位置**: §2.1.1 末尾  
**类型**: 术语不统一 / 配置键名混淆  
**描述**: 文本说 "`web.search_backend` 和 `web.extract_backend` 可分别覆盖" —— 但前面的整个 §2.1.1 表格（line 61-82）以及用户入口文件提到的都只有 `web.backend` 这一个键。用户会困惑：到底有几个配置键？`web.backend` 和 `web.search_backend` 之间是什么关系？覆盖规则是什么？没有任何解释。  
**推演**: 试图查找 `web-search-protocol` 或 `~/.hermes/config.yaml` 相关文档来澄清此关系，但当前技能内无此内容。

### 🔴 困惑点 4 — `references/design-journal.md:7-8`
**位置**: "Key Design Decisions" 列表  
**类型**: 严重过期 + 直接矛盾  
**描述**: 
- 第 7 行: "Search entry unified to `uvx ddgs`"  
- 第 8 行: "`web_search` tool deprecated (no API-key backends configured)"  
这些陈述与 SKILL.md 完全矛盾——SKILL.md 明确说 `web_search` 是主入口（§2.1），`uvx ddgs` 只是备用 fallback（§2.2）。用户读到此处会产生严重困惑：到底哪个是正确的？  
**推演**: 推演中我交叉检查了 SKILL.md 的历史记录（§8），发现 design-journal.md 记录的是 2026-05-16 的状态，而 SKILL.md 记录 `web_search` 在 2026-05-17 已恢复。即 design-journal.md 过时了 1 年以上（假定现在 2026-06-17，实际设计决策距今约 13 个月）。没有任何过期警告。

### 🔴 困惑点 5 — `references/design-journal.md:32`
**位置**: 末尾  
**类型**: 死链 / 外部引用断裂  
**描述**: 文件末尾指向 `~/.agents/journal/experience/web-search-protocol-design.md`。这个路径：
1. 使用 POSIX 风格路径 (`~/.agents/`)，在 Windows 上环境变量展开可能失败
2. 不在 skill 目录内部，不保证存在
3. 作为"Full Narrative"引用，但读者无法确认其存在性  
**推演**: 实际尝试定位该文件失败（仅允许 read_file，该路径超出 skill 目录且可能不存在）。

### 🟡 困惑点 6 — `SKILL.md:86`（相对链接完整性）
**位置**: 跨引用锚点  
**类型**: 确认性发现（正+负）  
**描述**: `[`references/web-backends-china.md`](references/web-backends-china.md)` 相对链接可以正确定位。但从 SKILL.md 反向指向 references 时，读 references 的用户不会自然获得此引导。这是一个单向链接结构。

### 🟡 困惑点 7 — `SKILL.md:15-29` vs `SKILL.md:88-98`
**位置**: 决策树对比 §2.1a  
**类型**: 模型不一致  
**描述**: 主决策树（§1）只包含 web_search → 选 URL → 提取流程这条路径。但 §2.1a 引入了"浏览器搜索"作为搜索发现的另一个选项。决策树没有将这个选项可视化表现出来。用户阅读决策树时会认为搜索只有 web_search 一条路，到 §2.1a 才发现还有浏览器搜索。  
**推演**: 不会导致理解错误，但会感觉结构不一致。

### 🟡 困惑点 8 — `references/web-backends-china.md:53-54`
**位置**: bing-cn MCP 评估  
**类型**: 路径依赖风险  
**描述**: 该章节说 bing-cn MCP 的 "`fetch_webpage` has 30s timeout, fails on slower sites"。但没有说明这个 30s 超时是 bing-cn MCP 自身的限制还是 Hermes 终端工具的默认超时。用户可能误解为 terminal 工具的限制而放弃使用。

## 4. 意外发现

### 正面发现 ✅
1. **Quick Fix 命令可直接使用** (`references/web-backends-china.md:38-40`)：文件末尾提供了 `hermes config set web.backend parallel` 一行命令，用户可以直接执行来解决问题。这对半知用户很友好。
2. **反向引用存在** (SKILL.md:86): SKILL.md 明确引用 `references/web-backends-china.md`，说明用户如果先读了 SKILL.md 也能自然流向参考文件。只是侧门进入的人看不到这个链接。
3. **MCP 工具参数明确** (`references/web-backends-china.md:33-34`): parallel 的 MCP 参数 `objective` 和 `search_queries` 正确记录了接口协议要求，对于需要手动调用 MCP 的用户有价值。
4. **提取链可用性检查** (`SKILL.md:252-265`): §4.0a 提供了一个决策子图来判断当前 session 的提取能力，包括 search-synthesis 的兜底模式。这在工具链部分缺失时能辅助决策。

### 负面发现 ❌
1. **design-journal.md 严重过期且无警告**：该文件记录的决策状态（web_search deprecated, ddgs 为主入口）与当前 SKILL.md 完全相反。更危险的是，文件没有任何"注意：此文档可能已过时"的提示。一个认真阅读全部文档的用户会被误导。
2. **实测数据覆盖度不足**：虽然文件名强调"实测数据"，但 8 个后端中 5 个是 "⚠️ untested"。对于"确认哪些后端可用"的查询，该文件只能给出 3/8 的结论。用户需要额外工作来补齐剩余 5 个的状态。
3. **无版本历史或变更日志**：skill 目录没有 CHANGELOG 或版本标记。用户无法判断 files 之间的相对时效性。当前只能通过对比设计日志日期（2026-05-16）和 SKILL.md 历史记录（2026-05-17 / 2026-05-29）手动推断。
4. **异常路径 1 — 跳过 SKILL.md 直接进入 references**：如果用户只读 references/web-backends-china.md，会完全错过 SKILL.md 中的决策树、提取链、GitHub API 通道等核心内容。侧门入口没有"前情提要"或引导。
5. **异常路径 2 — 从 design-journal.md 开始读**：如果用户无意中先读了 design-journal.md，会建立一个"web_search 已弃用"的错误预设，影响后续整个协议的理解。
