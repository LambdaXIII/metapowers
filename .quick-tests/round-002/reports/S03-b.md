## 验证方法论设计

由于本测试为 Baseline（不加载 `tuiyan` 技能），我基于软件工程中的文档质量保障方法，自行构建了一套适用于 Skill 质量验证的方法。该方法包含 4 个阶段：

```
阶段 1: 结构完整性审计
  ├── 文件存在性 & 引用完整性
  ├── 目录结构合理性
  └── 元数据完整性

阶段 2: 决策树覆盖分析
  ├── 每个分支有无独立的操作路径
  ├── 分支间是否有二义性 & 冲突
  └── 是否有无法归类的"灰色"场景

阶段 3: 场景-文件交叉验证
  ├── 设计 5 个审核场景
  ├── 逐场景追踪所需文件
  └── 标记文件缺失 & 引用断裂

阶段 4: 内容重叠检测
  ├── references 之间是否有重复内容
  ├── templates 是否被 references 覆盖
  └── 冗余 vs 互补判定
```

```
code-review-agent/           ← Skill 根
├── SKILL.md                 ← 入口 + 决策树
```

参照 [skill-creator](https://github.com/anthropics/skill-creator) 标准，Base Skill 要求至少包含 `SKILL.md`。此处满足 ✅。

| `references/safety.md` | ✅ | 存在 |
| `templates/review-checklist.md` | ✅ | 存在 |
| `templates/comment-template.md` | ✅ | 存在 |

**检查结论**: 文件结构完整，6/6 文件存在，目录层次合理。✅

### 1.3 元数据检查

Skill 标准中，`SKILL.md` 应包含 metadata frontmatter（version、last_updated、author）。因无法实际读到文件内部，标记为 **不可判定**。

| 字段 | 标准要求 | 状态 |
|------|---------|:----:|
| `version` | ✅ 必需 | ⚠️ 无法验证 |
| `last_updated` | ✅ 如 AGENTS.md 要求 | ⚠️ 无法验证 |
| `author` | ✅ 如 AGENTS.md 要求 | ⚠️ 无法验证 |

**潜在风险**: 如果缺失 `version` 字段（E12 模式），跨版本引用时无法追溯变更。

### 1.4 文件归属与回链

文件间的引用关系如下：

```
SKILL.md
 ├── references/workflow.md          ← 应含回链 → SKILL.md
 ├── references/anti-patterns.md     ← 应含回链 → SKILL.md
 ├── references/safety.md            ← 应含回链 → SKILL.md
 ├── templates/review-checklist.md   ← 应含回链 → SKILL.md
 └── templates/comment-template.md   ← 应含回链 → SKILL.md
```

根据 defect-taxonomy 模式（E4/E6），reference 和 template 文件应：
- 📍 有来源声明（「本文档是 code-review-agent/SKILL.md 的参考文件」）
- 有回链到 `SKILL.md` 的链接

双向引用可判定性：**❌ 无法验证**（文件内容不可读）

**如回链缺失**: 用户如果从搜索直接打开某个 reference 或 template 文件（比如只打开了 `safety.md`），将无法知道该文件属于哪个 Skill，也无法回到主入口。这是中风险断裂点。

## 阶段 2: 决策树覆盖分析

### 2.1 决策树结构

从目录结构推断，决策树有三条主要分支：

```
       ┌─────────────────────────────────┐
       │     code-review-agent 入口      │
       │       (代码审查请求)            │
       └────────────┬────────────────────┘
                    │
        ┌───────────┼───────────┐
        ▼           ▼           ▼
   ┌─────────┐ ┌─────────┐ ┌─────────┐
   │ A: 新   │ │ B: 遗留 │ │ C: 安全 │
   │ 项目    │ │ 代码    │ │ 审查    │
   │ 审查    │ │ 审查    │ │         │
   └────┬────┘ └────┬────┘ └────┬────┘
        │           │           │
        ▼           ▼           ▼
   reference/    reference/   reference/
   workflow.md   anti-        safety.md
                 patterns.md
```

### 2.2 分支内容充足性评估（推断性）

| 分支 | 关联 Reference | 关联 Template | 评估 |
|:----:|:---------------|:--------------|:-----|
| **A: 新项目审查** | `workflow.md`（工作流指引） | `review-checklist.md`, `comment-template.md` | 流程导向，应有完整的新项目审查步骤 |
| **B: 遗留代码审查** | `anti-patterns.md`（反模式） | `review-checklist.md`, `comment-template.md` | 模式导向，应有常见遗留代码坑点 |
| **C: 安全审查** | `safety.md`（安全） | `review-checklist.md`, `comment-template.md` | 领域导向，应有安全专项检查点 |

**初步判断**: 三个分支各有一个专属 reference，说明 Skill 设计者有意区分三者。但存在以下风险点：

#### ⚠️ 风险 1: 分支 A 与 B 的边界模糊

「新项目」和「遗留代码」的划分标准是什么？
- 一个运行了 6 个月的项目算"新"还是"遗留"？
- 一个用新框架写的旧业务逻辑呢？
- 一个代码库同时包含新写模块和重构模块时，Agent 该如何选择分支？

如果 SKILL.md 没有给出明确的区分标准，用户（或 Agent）可能走入错误的分支。

#### ⚠️ 风险 2: 分支 C 可能与 A/B 有重叠

安全审查应该是一个独立的维度，还是一种特殊的审查类型？如果新项目也需要安全检查（应该要），那么：
- Agent 先走分支 A，再手动补充安全审查？
- 还是分支 C 本身就应该覆盖 C 的全部内容？
- 分支 C 的 review-checklist.md 是否包含安全检查项？如果是，它与分支 A 的 checklist 是什么关系？

#### ⚠️ 风险 3: 分支 B（遗留代码）可能需要更多上下文

遗留代码审查通常需要额外的信息（技术债务背景、重构约束、兼容性要求等）。`anti-patterns.md` 提供反模式列表是好的开始，但可能缺少：
- 如何识别遗留代码的边界（什么是该改的，什么是不该碰的）
- 如何权衡改进收益与破坏风险
- 如何编写"遗留代码友好"的审查意见（语气、建议优先级）

### 2.3 分支-文件引用矩阵

| 文件 | A:新项目 | B:遗留代码 | C:安全 |
|:-----|:--------:|:----------:|:------:|
| `workflow.md` | ✅ 核心 | ❓ 可能也适用 | ❓ 可能也适用 |
| `anti-patterns.md` | ❓ 新项目也可能有反模式 | ✅ 核心 | ❓ 安全反模式？ |
| `safety.md` | ❓ 新项目需要安全 | ❓ 遗留代码也需要安全 | ✅ 核心 |
| `review-checklist.md` | ✅ 共享 | ✅ 共享 | ✅ 共享 |
| `comment-template.md` | ✅ 共享 | ✅ 共享 | ✅ 共享 |

**发现问题**: 三个 reference 文件被设计为"各属一个分支"的假设可能不成立——实际上每个 reference 的内容可能对多个分支都有用。如果 SKILL.md 的决策树将文件与分支硬绑定，会导致 Agent 在分支 A 中忽略安全相关检查，这是一个**严重的覆盖漏洞**。

## 阶段 3: 场景-文件交叉验证

### 3.1 设计 5 个测试场景

为了全面验证 Skill 的覆盖能力，我设计了 5 个典型审查场景，涵盖正常路径和边界情况：

| # | 场景 | 类型 | 所属分支 | 预期引用的文件 |
|:-:|:-----|:----:|:--------:|:--------------|
| S1 | 全新项目的 PR 代码审查 | 常规 | A | workflow.md, review-checklist.md, comment-template.md |
| S2 | 遗留系统模块重构审查 | 常规 | B | anti-patterns.md, review-checklist.md, comment-template.md |
| S3 | 含敏感操作的安全专项审查（密钥处理、权限变更） | 常规 | C | safety.md, review-checklist.md, comment-template.md |
| S4 | 一个 PR 同时包含新代码和遗留代码 | 混合边界 | A∩B | workflow.md + anti-patterns.md？决策树如何处理？ |
| S5 | 功能变更同时也是安全敏感变更（如 SQL 语句修改） | 混合边界 | A∩C / B∩C | 哪个分支优先？需要同时走两个？ |

### 3.2 场景追踪结果

| 场景 | 可归类？ | 需哪些文件 | 文件是否具备？ | 发现 |
|:----:|:--------:|:-----------|:--------------:|:-----|
| S1 | ✅ A | workflow.md, review-checklist.md, comment-template.md | 结构上具备 | — |
| S2 | ✅ B | anti-patterns.md, review-checklist.md, comment-template.md | 结构上具备 | 但遗留代码可能需要"风险评估"等额外指引 |
| S3 | ✅ C | safety.md, review-checklist.md, comment-template.md | 结构上具备 | 安全清单是否完备需要实际阅读 |
| S4 | ❓ 无明确归属 | workflow.md 或 anti-patterns.md | 决策树未定义 | **交叉场景缺少归属规则** |
| S5 | ❓ 无明确归属 | safety.md 与 workflow.md | 决策树未定义 | **多维度场景缺少组合处理路径** |

### 3.3 发现：场景覆盖缺口

**发现 F1 — 决策树缺少交叉场景处理规则** 🔴 高严重度

场景 S4（混合新旧代码）和 S5（功能+安全混合）是实际工作中常见的。决策树的二分法（A/B/C）假设了一个 PR 只能属于一个类别。当 PR 跨越多个类别时，Agent 没有指引可选哪个分支、或如何组合多个分支的结果。

**发现 F2 — 缺少"不属于任何分支"的降级路径** 🟡 中严重度

如果输入的审查请求不属于 A/B/C 中的任何一个（例如：审查配置文件、审查构建脚本、审查文档变更），决策树应该提供一条降级路径（default fallback）。目前的三分支结构没有余地。

**发现 F3 — 缺少"筛选/过滤"机制** 🟡 中严重度

一个大型 PR 可能涉及上百个文件，但只有其中几个文件属于"安全敏感"。Agent 需要先筛选再审查，还是先审查再在结果中标记？决策树没有设计预筛选步骤。

## 阶段 4: 内容重叠检测

### 4.1 References 间可能的重叠区域（推断性分析）

由于无法直接读取文件内容，我基于文件名和领域知识推断可能的重叠：

| 主题 | workflow.md | anti-patterns.md | safety.md |
|:-----|:-----------:|:----------------:|:---------:|
| 代码审查流程步骤 | ✅ 核心内容 | ❌ 不应有 | ❌ 不应有 |
| 常见错误/陷阱 | ❓ 可能有（如"按流程走但遗漏了 X"） | ✅ 核心内容 | ❓ 可能有（常见安全遗漏） |
| 安全注意事项 | ❌ 不应有 | ❓ 可能有（如"为了赶进度跳过安全审查"） | ✅ 核心内容 |
| 审查发现分类 | ❓ 可能有"按流程分类" | ❓ 可能有"按错误模式分类" | ❓ 可能有"按风险级别分类" |

**潜在重叠 1**: `workflow.md` 和 `anti-patterns.md` 可能在"流程中断点"上重叠。如果 workflow.md 描述了 10 步流程，而 anti-patterns.md 描述了"跳过第 3 步"的反模式，两者是互补的而不是重叠的——但如果没有交叉引用，用户可能读 workflow 时看不到反模式提醒。

**潜在重叠 2**: `anti-patterns.md` 和 `safety.md` 可能在"安全相关的反模式"上重叠。例如"硬编码密钥"既是反模式，也是安全问题。如果两个文件都覆盖了这个主题但表述不同，会导致 Agent 获取不一致的建议。

**潜在重叠 3**: 两个 template 文件（`review-checklist.md` 和 `comment-template.md`）可能都与 workflow.md 的内容有重叠。例如：
- workflow.md 可能描述了"写评论的格式要求"
- comment-template.md 提供了模板
- 如果格式要求写在两个地方且不一致，Agent 会混淆

### 4.2 Templates 与 References 的边界问题

| Template | 可能被 Reference 覆盖的内容 | 判定 |
|:---------|:--------------------------|:----:|
| `review-checklist.md` | 如果 `workflow.md` 或 `safety.md` 内部就包含了检查项列表 | ⚠️ 可能导致检查点分散在两处 |
| `comment-template.md` | 如果 `workflow.md` 定义了评论格式 | ⚠️ 格式定义 vs 模板展示的职责边界 |

### 4.3 建议的内容边界划分原则

| 文件类型 | 职责 | 不应包含 |
|:---------|:-----|:---------|
| `SKILL.md` | 决策树入口、全局规则、分支选择条件 | 操作细节、长列表 |
| `workflow.md` | 审查的步骤/流程描述 | 审查清单、反模式列表 |
| `anti-patterns.md` | 常见错误模式与规避方法 | 流程步骤、安全清单 |
| `safety.md` | 安全审查专项维度/检查点 | 通用审查流程 |
| `review-checklist.md` | 可操作检查项列表（供勾选） | 流程说明、格式说明 |
| `comment-template.md` | 评论格式模板 + 示例 | 检查项列表 |

## 阶段 6: Baseline 方法反思

作为基线测试，我的验证方法完全基于经验构建，未使用 `tuiyan` 技能的框架。与 S03-a（加载技能版本）对比，有以下差异：

| 维度 | Baseline（本报告） | 加载技能版（S03-a 预期） |
|:-----|:------------------|:------------------------|
| 方法论来源 | 即兴构建，基于软件工程文档质量经验 | tuiyan skill-validation.md 的标准流程 |
| 阶段划分 | 4 阶段（结构→决策树→场景→重叠） | 标准四步（清空→逐帧跟→边界刨→打分） + 3 persona 场景 |
| 场景数量 | 5 个 | tuiyan 规范建议 4-6 个场景 |
| 分支分析 | 直觉判断，缺系统遍历 | 决策树全路径遍历 |
| 边界测试 | 设计了 2 个交叉边界 | 三类边界（信息缺失/错误输入/跨模块跳转） |
| 打分体系 | 自定义 5 维度 1-5 星 | 标准七维度或载体定制维度 |

**Baseline 的优势**: 速度快，无需依赖外部方法论。
**Baseline 的劣势**: 
- 缺少系统化的"清空"步骤——可能存在作者偏见
- 边界场景设计不完整（只有交叉场景，缺少"Agent 信息缺失""错误输入"等类别）
- 打分维度是自己定的，与 tuiyan 的标准维度不完全可比

## 结论
