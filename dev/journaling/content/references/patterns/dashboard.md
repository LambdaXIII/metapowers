# Dashboard — 项目/领域级次级 INDEX 设计参考

> 这是一种项目级别的组织设计参考，不是规范或现成模板。旨在提供设计启发——
> 你的 Dashboard 应从实际使用中生长，不预设最终形态。

---

## 设计意图（为什么需要 Dashboard）

INDEX.md 是 journal 的全局入口——每个 session 启动时必经。但对于聚焦于特定项目/领域的工作 session，INDEX.md 的全局视图可能不够用：

- **信息过载**：全局信号中包含当前项目不相关的上下文
- **信号深度不足**：一个项目的详细进展、未决问题和关联条目无法在 INDEX.md 的一行中充分表达
- **加载成本**：每个 session 加载所有项目的内容是不必要的

Dashboard 提供**项目级隔离的详细视图**：比 INDEX.md 更聚焦、更详细、高度上下文相关。一次 session 通常只处理一个项目，Dashboard 作为次级路由，让 agent 不必加载全局上下文中不相关的部分。

Dashboard **不是分类体系的替代品**。所有内容条目仍按规范存放在标准分类目录下（experience/、knowledge/ 等）。Dashboard 只是指引"哪些条目和当前项目相关"的线索，不持有内容副本。

## 概念澄清

### 是什么

**Dashboard 是项目/领域级的次级 INDEX**。和 INDEX.md 是类比关系：

```
INDEX : journal = Dashboard : project/domain
```

功能一致（状态概览 + 快速路由），但：
- INDEX.md 是全局视图——覆盖整个 journal
- Dashboard 是聚焦视图——覆盖一个项目/领域

### 不是什么

1. **不是分类目录的索引**——"该项目的所有文件"是搜索功能的责任，不是 Dashboard 的。Dashboard 不枚举条目，只指引关键路径。
2. **不是 INDEX.md 的子章节或板块**——INDEX.md 中只留一行路由链接（标题 + 一句话），不内嵌 Dashboard 内容。Dashboard 是独立文件，按需加载。
3. **不是独立内容容器**——它不持有内容副本。所有条目仍按规范存放在标准分类目录下。Dashboard 通过 wikilink 提供指引。
4. **不是初始化预设**——Dashboard 从实际需求中生长出来，不是创建新 journal 时就预设的。

### 与 INDEX.md 的路由关系

- INDEX.md → 一行路由（链接 + 一句话描述）→ Dashboard
- Dashboard 不引用回 INDEX.md（INDEX.md 是已知入口，无需反向引用）
- 当前 session 处理该项目 → 顺着路由加载 Dashboard
- 不处理该项目 → Dashboard 不加载

路由示意：

```
INDEX.md
├── 协议声明
├── 经验摘要         ← 全局跨域信号
├── 专项工作           ← 一行一域，含 Project-A [→](active_works/project-a/dashboard.md)
├── 最近变更
└── 个性化规则文件

active_works/project-a/dashboard.md  ← 次级入口
├── 当前状态
├── 关键决策记录
└── ...
```

### 内容范围

Dashboard 关注的是**和当前项目直接相关**的信息：

| 可以放在 Dashboard | 不应放在 Dashboard |
|-------------------|-------------------|
| 当前项目状态（摘要级） | 项目内容的完整副本 |
| 关键决策及理由 | 日常操作记录 |
| 指向 experience/ 中相关条目的链接 | 条目本身的内容 |
| 上下游依赖和注意事项 | 整个项目的文件清单 |
| 指向分类目录中相关内容的快速路由 | 分类目录的替代品 |

## 设计参考

### 创建信号

以下信号提示"可能需要创建一个 Dashboard"。由 agent 自行判断，非硬性规则：

| 信号 | 说明 |
|------|------|
| INDEX.md 专项工作中该项目需要多行说明状态 | 一行已不足以表达，需要隔离视图 |
| 项目产生了分布在 3+ 个分类目录的条目 | 条目分散，加载路径变长 |
| 每次处理该项目都需要从多目录收集信息 | 上下文收集成本超出单行路由的收益 |
| 项目进入活跃开发/维护期 | 长期活跃项目值得独立视图 |

### 生命周期

| 阶段 | 操作 |
|------|------|
| 创建 | 维护阶段创建，放在项目的目录下（如 `active_works/<project>/dashboard.md`） |
| 持续 | 随项目进展更新状态、关键决策、路由链接 |
| 归档 | 项目结束时不再更新。项目条目已按分类存放好，Dashboard 变成历史参考文档，留在原位置或视情况移入 `archive/` |

### 内容组织示例

Dashboard 和 INDEX.md 一样拥有组织灵活性。以下**不是模板**——它只是"可能长什么样"的示例，你的 Dashboard 应从实际需求中演化：

- **当前状态**：项目当前阶段和主要关注点
- **关键决策记录**：链接到记录了关键决策的 entry
- **相关经验条目**：指向 experience/ 中与项目相关条目
- **上下游依赖**：来自 knowledge/ 的技术依赖或外部资源
- **待办项**：短期可执行事项

与 INDEX.md 一样，从简单开始逐渐演化。一个新建 Dashboard 可以只有两行——状态摘要和指向主要 entry 的链接。

### 设计原则

1. **不破坏分类体系**——所有内容条目仍按规范存放在标准分类目录下。Dashboard 只通过链接指引，不持有内容副本。
2. **按需加载**——与当前 session 无关的 Dashboard 不加载。INDEX.md 中的路由行就是在每 session 判断是否需要加载的依据。
3. **可演化结构**——无固定板块模板。Dashboard 和 INDEX.md 一样，从简单开始逐渐演化出适合该项目的结构。
4. **信号价值门**——不产生信号的 Dashboard（所有链接都已指向归档条目、状态长期无变化、Agent 从不加载它）应在维护期间清理。Dashboard 不是永久保留物。
